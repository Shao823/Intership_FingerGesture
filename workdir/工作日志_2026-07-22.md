## 3. Address Editor 中 Unassigned / OCM 的理解

今天检查了更换硬件后 Address Editor 中出现的 `Unassigned` 地址项。

### 3.1 Unassigned 的含义

`Unassigned` 表示某个 AXI master 到某个可寻址 slave/address segment 的映射尚未被 Vivado 分配，并不直接表示 accelerator RTL 出错。

处理时应先判断该地址段是否真的会被当前数据路径访问：

```text
1. PS 是否需要通过 AXI-Lite访问该外设。
2. DMA MM2S 是否需要访问该 DDR 地址空间。
3. 当前 master 是否实际连接到该 slave。
4. 该地址段是否只是 PS 内部可选存储区域。
```

不要因为看到 `Unassigned` 就随意分配一个地址，也不要因此删除整个 accelerator IP。

### 3.2 OCM 是什么

OCM 是 Zynq UltraScale+ MPSoC 的 On-Chip Memory，即 PS 内部片上 SRAM。

其特点：

```text
容量较小
访问延迟低
位于 PS 内部
可用于启动代码、小型数据、异常向量或共享缓冲
不是板载 DDR
不是 PL 中的 Block RAM
```

当前 EMG 测试帧主要放在 PS DDR，由 AXI DMA MM2S 读取，并不要求使用 OCM。与当前 DDR -> DMA -> accelerator 数据路径无关的 OCM segment 可以不作为首轮上板测试重点。

## 7. 当前 PS 程序的具体逻辑

### 7.1 当前验证目标

当前程序验证的是：

```text
一次 DDR 数据准备
一次 AXI DMA MM2S 搬运
accelerator 接收完整 435-beat 帧
一次 accelerator 推理
结果和 cycle count 回读
```

当前还不是十帧连续运行，也不是中断驱动的 ping-pong 流水调度程序。

### 7.2 main() 总体流程

程序主要步骤：

```text
1. 使能 I-Cache 和 D-Cache。
2. 初始化 XAxiDma 实例。
3. 检查 DMA 是否为 Simple Mode，拒绝 Scatter-Gather 模式。
4. 复位 DMA，并等待复位完成。
5. 关闭当前未使用的 DMA 中断。
6. 读取 accelerator VERSION 和 FRAME_INFO。
7. 检查硬件版本和帧长度是否与软件一致。
8. 清除 accelerator 旧的 DONE 和 ERROR sticky 状态。
9. CONFIG 写 0，关闭 AUTO_START 和 accelerator IRQ。
10. 将一组测试帧复制到对齐的 DDR 缓冲区。
11. Flush D-cache，确保 DMA 能看到 CPU 写入的数据。
12. 调用 XAxiDma_SimpleTransfer() 提交 1740-byte MM2S 传输。
13. 轮询 DMA Busy，等待 DMA 搬运完成。
14. 轮询 accelerator FRAME_READY。
15. 检查 ERROR_STATUS。
16. 写 COMMAND.START 启动推理。
17. 轮询 DONE。
18. 读取 RESULT 和 CYCLE_COUNT。
19. 检查预测类别是否为 3，周期数是否接近仿真值 40527。
20. 清除 DONE，打印 PASS/FAIL 后停在空循环。
```

### 7.3 当前单帧数据格式

```text
输入长度       = 435 samples
每个 sample    = 32 bit
AXIS beat 数   = 435
传输字节数     = 435 × 4 = 1740 bytes
预期类别       = 3
预期周期数     ≈ 40527
```

DMA 发送最后一个 beat 时产生 `TLAST`，accelerator 收到完整帧后设置 `FRAME_READY`。

## 8. 为什么当前不需要单独的 emg_dma.c

当前 application 已通过：

```c
#include "xaxidma.h"
```

使用 Xilinx BSP 提供的标准 AXI DMA 驱动。

当前调用：

```c
XAxiDma_LookupConfig()
XAxiDma_CfgInitialize()
XAxiDma_Reset()
XAxiDma_IntrDisable()
XAxiDma_SimpleTransfer()
XAxiDma_Busy()
```

这些函数内部会通过 AXI-Lite/MMIO 实际访问 DMA 寄存器，包括：

```text
MM2S_DMACR
MM2S_DMASR
MM2S Source Address
MM2S Source Address MSB（如需要）
MM2S Length / BTT
```

因此“不存在项目自己的 emg_dma.c”不代表 DMA 没有配置寄存器；只是当前直接复用了 BSP 中成熟的 `XAxiDma` 驱动，没有重复封装一层。

后续进入多帧、中断和 ping-pong 调度阶段时，再增加：

```text
emg_dma.c
emg_dma.h
```

其职责应是应用层策略封装，而不是重新实现 Xilinx 底层驱动。建议封装：

```text
DMA初始化
DMA中断注册
提交帧地址和长度
cache flush与memory barrier
DMA错误解析和复位恢复
多帧DDR地址管理
ping-pong bank调度
DMA与accelerator联合状态机
```

## 9. DMA 与 Accelerator 中断的完整职责

### 9.1 DMA MM2S 中断

DMA 中断主要表示“DDR 中的一帧数据已经被 DMA 发送到 AXI-Stream”。

正常完成中断 ISR 建议执行：

```text
1. 读取 MM2S_DMASR。
2. 判断 IOC IRQ 是否置位。
3. 写 1 清除 IOC 中断状态。
4. 标记 dma_frame_sent = true。
5. 记录当前完成的 DDR buffer/case 编号。
6. 根据 accelerator bank 空闲状态决定是否提交下一帧 DMA。
7. 不在 ISR 中执行长时间轮询或打印大量字符串。
```

错误中断 ISR 建议执行：

```text
1. 保存 MM2S_DMASR 错误位。
2. 清除错误中断。
3. 停止继续提交新帧。
4. 设置 dma_error 标志。
5. 在主循环中执行 DMA reset 和状态恢复。
```

DMA 完成只代表 AXI DMA 已发送完 BTT 指定字节数，不等于 accelerator 已完成计算。

### 9.2 Accelerator 中断

accelerator 中断主要表示“推理完成，RESULT 和 CYCLE_COUNT 可读取”。

ISR 建议执行：

```text
1. 读取 STATUS。
2. 确认 DONE sticky 有效。
3. 读取 RESULT。
4. 读取 CYCLE_COUNT。
5. 记录完成 bank、frame index 和分类结果。
6. 写 COMMAND.CLEAR_DONE 清除完成状态并撤销 IRQ。
7. 标记 accelerator_done = true。
8. 通知调度器启动已准备好的另一个 bank。
```

如果 STATUS/ERROR_STATUS 表明协议错误或启动被拒绝，则记录错误并停止正常调度。

### 9.3 当前为什么使用轮询

首轮上板先使用轮询，便于按步骤隔离问题：

```text
DDR/cache
DMA配置
AXI-Stream帧接收
accelerator寄存器
推理结果
```

等单帧闭环稳定后再引入：

```text
GIC初始化
中断优先级与触发类型
DMA ISR
accelerator ISR
共享状态变量
多帧状态机
```

这样可以避免把数据路径问题和中断配置问题混在一起调试。

```text

3. 点击 Arm Trigger / Run Trigger。
4. 再回到 Vitis Resume。
5. 保证 ILA 已经等待后续 DMA 和 accelerator 信号。
```


### 13.2 Out of Date 与 Force Up-to-Date

仅修改 run strategy 或切换增量模式后，Vivado 可能把已有结果标为 Out of Date。

如果确认：

```text
RTL没有变化
BD没有变化
约束没有变化
Part没有变化
已有结果本身有效
```

可以针对这一次结果使用 Force Up-to-Date，目的是保留本次已经正确的结果，同时让以后修改时使用增量流程。

但以下情况不能靠 Force Up-to-Date 跳过重跑：

```text
更换Part
修改PS配置
新增UART
修改BD连接
修改XDC约束
更新IP RTL
```

今天加入 UART0 属于硬件配置变化，因此最终仍需要重新综合、实现和生成 bitstream。
