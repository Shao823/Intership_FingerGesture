## 3. Vivado 综合和时序分析

### 3.1 综合流程观察

综合变慢后，检查过 Vivado 状态：

```text
1. CPU 和内存占用较高，说明综合进程大概率仍在工作，不是完全卡死。
2. cached IP results 本身通常不是综合变慢主因。
3. incremental checkpoint 可能让综合复用旧结果，但在 RTL 大改后也可能引入额外匹配/恢复开销。
```

建议后续 RTL 大改后：

```text
1. 必要时关闭 incremental synthesis。
2. 清理过旧的 synth run。
3. 重新 launch synthesis。
4. 等稳定后再考虑重新启用 incremental。
```

### 3.2 资源和时序判断

当前资源截图说明小器件资源压力较大，但用户后续会换更好的板子，所以今日重点放在时序结构问题上。

关键判断：

```text
1. IO 端口数量不是核心时序问题，后续 accelerator 顶层会自然减少大量无用端口。
2. 当前 100MHz 跑不动的原因主要是 RTL 中存在过长组合路径。
3. setup 和 hold 是两类不同问题：
   - setup 负 slack：组合逻辑太长或 fanout/route 过大。
   - hold 负 slack：源寄存器到目的寄存器/BRAM 端口路径太短或时钟偏斜导致 min-delay 不够。
```

截图中曾出现的关键路径：

```text
FC setup:
  u_fc/u_compute/avg_gap_group_reg -> avg_div_group_reg
  主要问题是 GAP average 除以 174 路径过长。

DW2 setup:
  u_dw2/resp_group_idx_reg -> u_dw2/fifo_vec_reg
  主要问题是 DW compute 输出阶段组合逻辑过长。

PW1->DW2 / buffer setup:
  u_pw1_to_dw2/u_ctrl/slot_tag_reg -> max_dw2_pending_count_reg
  主要问题是 buffer metadata 查询和顶层统计/调度控制耦合。

DW1 hold:
  u_dw1/fifo_vec_reg -> u_dw1_to_pw1 BRAM DI
  主要问题是 FF 到 BRAM 写数据口路径过短。
```

## 4. FC GAP Average 时序优化

### 4.1 原问题

FC setup 截图显示最差路径落在：

```text
Source:
  gap_latched_mem_reg / avg_gap_group_reg

Destination:
  avg_mem_reg / avg_div_group_reg
```

原设计等价于在很少的拍数内完成：

```text
GAP sum 读取
  -> 除以 174
  -> 饱和到 Q8.8
  -> 写入 avg_mem
```

这会让 48-bit 除法、动态索引 mux、符号处理、饱和逻辑集中在一条 setup path 上。

### 4.2 首轮拆拍

先将 96 通道 average 分组处理：

```text
AVG_LANES = 12
AVG_GROUPS = 8
```

状态拆分为：

```text
STATE_AVG_LOAD
STATE_AVG_ABS
STATE_AVG_MUL
STATE_AVG_SHIFT
STATE_AVG_SIGN
STATE_AVG_STORE
```

### 4.3 常数除法替换

将 `/174` 替换为常数倒数乘法和移位：

```text
AVG_RECIP_SHIFT = 30
AVG_RECIP_MUL   = 6170931

avg_abs * AVG_RECIP_MUL >> AVG_RECIP_SHIFT
```

该常数近似：

```text
round(2^30 / 174) = 6170931
```

处理流程：

```text
1. 取 GAP sum 的绝对值。
2. 乘常数倒数。
3. 右移得到商。
4. 恢复符号。
5. 饱和到 DATA_W。
```

这样避免综合器推断大除法器，并把乘法、移位、符号恢复、饱和拆开。

当前 FC compute 结构：

```text
emg_gesture_vivado/src/fc_classifier_96_5/fc_classifier_96_5_compute.v
  AVG_LANES
  AVG_RECIP_MUL
  AVG_RECIP_SHIFT
  STATE_AVG_LOAD/ABS/MUL/SHIFT/SIGN/STORE
```

影响：

```text
1. FC 延迟增加若干拍。
2. 前五级流水线本来就等待 FC 完成，所以功能协议不变。
3. 关键路径从大除法转为较短的乘法/移位/寄存器间路径。
```

## 5. DW1/DW2 Compute 三阶段 II=1 流水

### 5.1 原问题

DW2 setup 截图曾显示：

```text
u_dw2/resp_group_idx_reg[1]
  -> u_dw2/fifo_vec_reg[0][x]
```

这表示 DW compute 在一个周期内完成了较多组合工作：

```text
1. 根据 resp_group_idx 选择当前 group。
2. 从 win_vec / weight_mem 取多个 tap/lane。
3. 做 K=5 的乘加。
4. 做 shift / saturate。
5. 写入 fifo_vec。
```

虽然 MAC 数量不算特别大，但 LUT 级数、动态索引、符号扩展、饱和逻辑和路由叠加后，100MHz setup 会失败。

### 5.2 修改方案

将 DW1/DW2 compute datapath 改为内部三阶段 valid pipeline：

```text
S0:
  latch window data / weight data

S1:
  multiply

S2:
  accumulate + shift + saturate

FIFO:
  接收 S2 输出
```

关键要求：

```text
内部 datapath II = 1
输出 FIFO 满时 pipeline hold
窗口请求接口仍保持原 one-outstanding 协议
```

说明：

```text
II=1 指 compute datapath 内部每拍可推进一组已进入流水的数据。
但上游窗口请求/响应接口仍不是无限 outstanding。
如果上游 buffer 仍是同步 1-cycle 响应且 one-outstanding，接口层吞吐可能不是每拍一个响应。
```

### 5.3 背压风险判断

DW 模块加流水后增加的是固定延迟，不降低内部持续吞吐。原先顶层统计里：

```text
DW1 局部 stalls 可被前后级 buffer 吸收。
DW2 max pending 曾达到 4，但后续空档能消化。
```

因此三阶段 II=1 不应成为新的系统级瓶颈。真正需要关注的是：

```text
1. 输出 FIFO 是否足够吸收下游短暂停顿。
2. one-outstanding 窗口接口是否限制顶层瞬时吞吐。
3. DW2 pending FIFO 是否仍有足够余量。
```

## 6. Buffer Metadata Setup 优化

### 6.1 原结构问题

DW1->PW1 / DW2->PW2 buffer 原来判断 tile 是否可读时，需要按 tile 的 4 个 row 扫描所有 bank 的：

```text
slot_valid[bank][row]
slot_tag[bank][row]
```

这会形成较宽的组合判断：

```text
tile_available = all banks of all required rows are present
```

综合后可能表现为：

```text
slot_tag_reg -> 顶层统计/调度寄存器 CE
slot_tag_reg -> input_tile_req_ready
```

### 6.2 当前优化

在 DW1->PW1 和 DW2->PW2 buffer 中维护 row-level metadata：

```text
row_full_valid[row]
row_full_tag[row]
```

写入某个 bank/row 时：

```text
1. 写 slot_valid/slot_tag。
2. 先假设 row_full_valid[row] = 1。
3. 扫描同一 row 的其他 bank。
4. 如果其他 bank 未包含同一时间点，则 row_full_valid[row] = 0。
```

tile request 判断时改为只看：

```text
row_full_valid[row_idx]
row_full_tag[row_idx]
```

而不是每次 request 都扫描全部 bank。

影响：

```text
1. tile_available 查询逻辑显著变短。
2. 写入侧多一点 row_full 更新逻辑，但写入侧本来不是当前最关键 setup 路径。
3. 功能语义不变：只有所有 bank 都写齐同一时间点时 row_full 才为真。
```

涉及文件：

```text
emg_gesture_vivado/src/dw1_to_pw1_buffer/dw1_to_pw1_buffer.v
emg_gesture_vivado/src/dw2_to_pw2_buffer/dw2_to_pw2_buffer.v
```

## 8. Hold Timing 修复

### 8.1 现象

Vivado hold 截图显示：

```text
Slack (Hold) = -0.057 ns

Source:
  u_dw1/fifo_vec_reg[0][x]

Destination:
  u_dw1_to_pw1/.../BRAM/DIADI[x]

Logic Levels = 0
Data Path Delay ~= 0.346 ns
```

该路径不是组合逻辑太长，而是 FF 到 BRAM 写数据口路径太短，属于 min-delay 不足。

### 8.2 修改方案

在 DW1->PW1 和 DW2->PW2 buffer 的 BRAM 写数据口前插入综合专用 identity LUT delay：

```text
dw1_to_pw1_hold_delay
dw2_to_pw2_hold_delay
```

仿真：

```verilog
assign dout = din;
```

综合：

```verilog
(* DONT_TOUCH = "true" *) LUT1 #(
    .INIT(2'b10)
) u_lut (
    .I0(din[bit_i]),
    .O(dout[bit_i])
);
```

目的：

```text
1. 不改变 ready/valid 协议。
2. 不增加计算周期。
3. 只给 FF -> BRAM DI 的 hold path 增加最小数据延迟。
```

涉及文件：

```text
emg_gesture_vivado/src/dw1_to_pw1_buffer/dw1_to_pw1_buffer.v
emg_gesture_vivado/src/dw2_to_pw2_buffer/dw2_to_pw2_buffer.v
```

验证结果：

```text
PASS tb_dw1_to_pw1_buffer latency=1
PASS tb_dw1_to_pw1_buffer latency=2
PASS tb_dw2_to_pw2_buffer latency=1
PASS tb_dw2_to_pw2_buffer latency=2

CASE 0 cycles=39866 class=2 PASS
PASS tb_accelerator
```

后续综合检查重点：

```text
1. 原 fifo_vec_reg -> BRAM DI hold path 是否消失或 slack 转正。
2. 如果新的 hold 最差路径变成 BRAM address/en 等其他短路径，再按具体端口定点处理。
3. 不建议全局乱加 delay，只处理 Vivado 报告中的真实 min-delay 失败路径。
```
