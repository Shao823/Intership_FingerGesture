# 参数与激活读取策略 2026-07-06

本文集中记录当前 `emg_gesture_vivado/src` 里所有已完成计算模块的参数、激活、BN folded 参数读取策略，作为后续设计 weight ROM、activation buffer、row packer、window loader、top scheduler 的接口参考。

核心原则：

```text
1. 计算模块只发逻辑请求，不绑定 ROM/BRAM 的物理地址布局。
2. ROM controller / activation loader / window loader 负责地址映射和 vector 打包。
3. 所有 vector lane 统一按 Verilog 低位到高位解释：
       vec[lane*DATA_W +: DATA_W] = logical_value[base + lane]
4. 对每个 accepted request 必须恰好返回一次 response。
5. 同一个计算核内部的 response 必须按 request 顺序返回。
6. 当前计算模块已经完成；后续重点是存储/controller/loader 与 top 集成。
```

## 1. stem_conv_array_4x4

文件：

```text
emg_gesture_vivado/src/stem_conv_array_4x4.v
```

### 1.1 模块职责

```text
输入:  raw/normalized input, 5 channels, 348 high-res samples
计算:  Conv1d 5 -> 32, k=7, pad=3
后处理: folded BN scale/bias + ReLU
输出:  stem feature, 32 channels, 348 high-res samples
```

一次 `start`：

```text
4 high-res rows x 4 output channels
```

外层 scheduler 仍需扫：

```text
oc_base = 0,4,8,...,28
```

### 1.2 weight 读取

请求：

```text
weight_req_valid / weight_req_ready
weight_req_oc_base
weight_req_k = 0..34
```

逻辑含义：

```text
k_flat = weight_req_k
input_channel = k_flat / 7
tap           = k_flat % 7

weight_vec[lane] =
    W_stem[weight_req_oc_base + lane][input_channel][tap]
```

lane：

```text
lane = 0..3
weight_vec[lane*DATA_W +: DATA_W]
```

推荐 ROM packed layout：

```text
stem_weight_word[oc_group][k_flat]
  lanes 0..3 = W_stem[oc_group*4 + lane][k_flat]

oc_group = weight_req_oc_base / 4
address  = oc_group * 35 + k_flat
word width = 4 * DATA_W
```

### 1.3 activation 读取

请求：

```text
act_req_valid / act_req_ready
act_req_t_base
act_req_k = 0..34
```

逻辑含义：

```text
k_flat = act_req_k
input_channel = k_flat / 7
tap           = k_flat % 7

in_t = act_req_t_base + row + tap - 3
act_vec[row] = input[in_t][input_channel]
```

边界：

```text
if in_t < 0 or in_t >= 348:
    act_vec[row] = 0
```

关键点：

```text
stem 的 act_req_k 是 flattened conv index，不是 input channel。
input loader 必须做 im2col/window 地址生成。
```

推荐 activation loader：

```text
input_window_buffer[time][5]
or full input buffer[348][5]

收到 act_req_t_base, act_req_k 后：
  解析 input_channel/tap
  读取 4 个 row
  越界补 0
  返回 4-lane act_vec
```

### 1.4 BN folded 参数

接口：

```text
w_fold_we
w_fold_oc
w_fold_wdata

bias_fold_we
bias_fold_oc
bias_fold_wdata
```

存储：

```text
w_fold_mem[32]
bias_fold_mem[32]
```

读取方式：

```text
PE 内部按 output channel 直接读取:
  bn_scale = w_fold_mem[oc_base_reg + col]
  bn_bias  = bias_fold_mem[oc_base_reg + col]
```

策略：

```text
stem BN folded 参数驻留在计算模块本地寄存器/小 RAM。
推理开始前写入。
当前 stem 与 PW1/PW2 一样，只在 state == IDLE && !start 时接受写入。
busy 期间误写会被忽略。
```

## 2. dw_conv1_4ch_5tap

文件：

```text
emg_gesture_vivado/src/dw_conv1_4ch_5tap.v
```

### 2.1 模块职责

```text
输入:  stem output, 32 channels, 348 high-res samples
计算:  depthwise Conv1d, k=5, pad=2
输出:  DW1 output, 32 channels, 348 high-res samples
```

一次 `start`：

```text
一个 4-channel group 的所有新可计算 out_t
```

外层 scheduler 需扫：

```text
ch_base = 0,4,8,...,28
```

### 2.2 weight 读取

DW1 权重本地驻留。

接口：

```text
weight_we
weight_ch
weight_tap
weight_wdata
```

存储：

```text
weight_mem[32][5]
```

写入保护：

```text
weight_we 只在 state=IDLE 且 start=0 时生效。
busy/RUN 期间误写会被忽略。
```

计算时：

```text
lane = 0..3
ch = ch_base_reg + lane

y[ch] =
  win[0][lane] * weight_mem[ch][0] +
  win[1][lane] * weight_mem[ch][1] +
  win[2][lane] * weight_mem[ch][2] +
  win[3][lane] * weight_mem[ch][3] +
  win[4][lane] * weight_mem[ch][4]
```

### 2.3 activation/window 读取

请求：

```text
win_req_valid / win_req_ready
win_req_out_t
win_req_ch_base
```

响应：

```text
win_vec_valid / win_vec_ready
win_vec[(tap*P_CH + lane)*DATA_W +: DATA_W]
```

逻辑含义：

```text
in_t = win_req_out_t + tap - 2
ch   = win_req_ch_base + lane

win_vec[tap][lane] = stem_out[in_t][ch]
```

边界：

```text
if in_t < 0 or in_t >= 348:
    win_vec[tap][lane] = 0
```

推荐 `stem_to_dw1` window loader：

```text
stem_to_dw1_buffer[time][32]
valid_group[time][8]   // 32 channels / 4

收到 win_req_out_t, win_req_ch_base:
  对 tap=0..4:
    in_t = out_t + tap - 2
    if padding: return 0
    else check valid_group[in_t][ch_base/4]
  只有非 padding 数据都 valid 后返回 win_vec_valid
```

### 2.4 BN 参数

DW1 当前无 BN/ReLU。

原因：

```text
PyTorch block1 = depthwise -> pointwise -> BN -> ReLU -> MaxPool
所以 BN/ReLU 已融合到 PW1 模块，不在 DW1 中做。
```

## 3. pw_conv1_array_4x8

文件：

```text
emg_gesture_vivado/src/pw_conv1_array_4x8.v
```

### 3.1 模块职责

```text
输入:  DW1 output, 32 channels, 348 high-res samples
计算:  pointwise Conv1d 32 -> 64
后处理: folded BN scale/bias + ReLU + MaxPool1d(k=2,stride=2)
输出:  pool output, 64 channels, 174 low-res samples
```

一次 `start`：

```text
等待 4 high-res rows x 32 channels ready。
模块内部扫 oc_base = 0,8,16,24,32,40,48,56。
共输出 8 个 tile，每个 tile 是 2 pooled rows x 8 channels。
```

`oc_base` 输入端口当前只为接口兼容保留，full-channel job 内部忽略该输入。

### 3.2 input tile ready

请求：

```text
input_tile_req_valid
input_tile_req_t_base
input_tile_req_ready
```

语义：

```text
input_tile_req_ready=1 表示:
  DW1 output 的 input_tile_req_t_base..+3
  这 4 个 high-res rows 的全部 32 channels 均已 valid，
  且在整个 PW1 job 期间不会被覆盖。
```

推荐判断：

```text
row_full[t] = &valid_group[t][7:0]

input_tile_req_ready =
    row_full[t_base]
 && row_full[t_base+1]
 && row_full[t_base+2]
 && row_full[t_base+3]
```

### 3.3 weight 读取

请求顺序：

```text
for oc_base in 0,8,16,24,32,40,48,56:
    for k in 0..31:
        weight_req_oc_base = oc_base
        weight_req_k       = k
```

逻辑含义：

```text
weight_vec[lane] =
    W_pw1[weight_req_oc_base + lane][weight_req_k]
```

lane：

```text
lane = 0..7
weight_vec[lane*DATA_W +: DATA_W]
```

推荐 ROM packed layout：

```text
pw1_weight_word[oc_group][k]
  lanes 0..7 = W_pw1[oc_group*8 + lane][k]

oc_group = weight_req_oc_base / 8
address  = oc_group * 32 + k
word width = 8 * DATA_W
```

### 3.4 activation 读取

请求顺序：

```text
for oc_base in 0,8,...,56:
    for k in 0..31:
        act_req_t_base = tile_t_base
        act_req_k      = k
```

逻辑含义：

```text
act_vec[row] = DW1_out[act_req_t_base + row][act_req_k]
```

lane：

```text
row = 0..3
act_vec[row*DATA_W +: DATA_W]
```

重要策略：

```text
同一个 4x32 input tile 会被 8 个 output-channel groups 重复使用。
因此 input_tile_req_ready 后，建议把 4x32 锁存到 pw1_input_tile_regs。
```

推荐 activation provider：

```text
pw1_input_tile_regs[4][32]

收到 act_req_k:
  act_vec[0] = tile_regs[0][act_req_k]
  act_vec[1] = tile_regs[1][act_req_k]
  act_vec[2] = tile_regs[2][act_req_k]
  act_vec[3] = tile_regs[3][act_req_k]
```

文件：

```text
emg_gesture_vivado/src/pw_conv2_array_4x12.v
```

### 5.1 模块职责

```text
输入:  DW2 output, 64 channels, 174 low-res samples
计算:  pointwise Conv1d 64 -> 96
后处理: folded BN scale/bias + ReLU
输出:  96-channel GAP sum/count accumulator
```

一次 `start`：

```text
等待 4 low-res rows x 64 channels ready。
模块内部扫 oc_base = 0,12,24,36,48,60,72,84。
共计算 8 个 output-channel group。
每个 group 的 4 rows x 12 channels 结果不再作为 active tile stream 输出，
而是在模块内直接累加进 `gap_sum_vec`。
```

```text
tile_t_base = 172
row_valid_mask = 4'b0011
```

### 5.2 input tile ready

请求：

```text

input_tile_req_t_base
input_tile_req_row_valid_mask
input_tile_req_ready
```

语义：

```text
input_tile_req_ready=1 表示:
  DW2 output 的 requested rows
  在 row_valid_mask 为 1 的行上，全部 64 channels 均已 valid，
  且在整个 PW2 job 期间不会被覆盖。
```

推荐判断：

```text
row_full[t] = &valid_group[t][15:0]

input_tile_req_ready =
  对每个 row=0..3:
    if row_valid_mask[row] == 1:
       row_full[t_base + row] 必须为 1
    else:
       不要求该行 valid
```

### 5.3 weight 读取

请求顺序：

```text
for oc_base in 0,12,24,36,48,60,72,84:
    for k in 0..63:
        weight_req_oc_base = oc_base
        weight_req_k       = k
```

逻辑含义：

```text
weight_vec[lane] =
    W_pw2[weight_req_oc_base + lane][weight_req_k]
```

lane：

```text
lane = 0..11
weight_vec[lane*DATA_W +: DATA_W]
```

推荐 ROM packed layout：

```text
pw2_weight_word[oc_group][k]
  lanes 0..11 = W_pw2[oc_group*12 + lane][k]

oc_group = weight_req_oc_base / 12
address  = oc_group * 64 + k
```

### 5.4 activation 读取

请求顺序：

```text

        act_req_t_base = tile_t_base
        act_req_k      = k
```

### 5.6 输出/GAP 策略

当前 RTL 已采用直接 GAP 累加策略。

active 输出/控制接口：

```text
gap_clear
gap_sum_vec[96]
gap_count
gap_frame_done
done
```

语义：

```text
gap_clear:
  清零 96 个 gap_sum_mem 和 gap_count，并让模块回到 IDLE。

gap_sum_vec[oc]:
  保存当前 frame 内 PW2 post-BN/ReLU 后的逐通道累加和。

gap_count:
  保存当前已经累加的 low-res row 数。
  只在最后一个 oc group 完成时增加 popcount(row_valid_mask)，
  避免 8 个 output-channel group 重复计数。

gap_frame_done:
  当 gap_count + popcount(row_valid_mask) == GAP_LEN(默认 174) 时脉冲 1 拍。

done:
  每个 4-row PW2 job 完成时脉冲 1 拍。
```

内部累加时机：

```text
for oc_base in 0,12,...,84:
    计算 4 rows x 12 channels
    for col in 0..11:
        for row in 0..3:
            if row_valid_mask[row]:
                gap_sum_mem[oc_base + col] += post_relu[row][col]

    if oc_base == 84:
        gap_count += popcount(row_valid_mask)
```

final partial tile：

```text
tile_t_base = 172
row_valid_mask = 4'b0011
```

此时只累加 row 0/1，对应 low-res time 172/173。row 2/3 不累加，
所以最后一组只贡献 2 行，完整 frame 的 `gap_count` 正好到 174。

PW2 不执行除法 averagepool。当前最终设计由 `fc_classifier_96_5` 在启动后对
`gap_sum_vec` 做 `/174`：

```text
gap_avg[ch] = sat_q8_8(gap_sum_vec[ch] / 174)
```

因此 PW2 到 FC 的接口应传递 sum/count，而不是已经平均后的向量：

```text
gap_sum_vec[96]
gap_count
```

推荐在 `gap_frame_done` 后锁存一份 `gap_to_fc_sum_regs[96]` 和
`gap_to_fc_count_reg`，避免后续 `gap_clear` 影响 FC 输入。

旧 tile stream 接口没有作为 active port 保留，但已在
`pw_conv2_array_4x12.v` 文件头注释中保留为 debug/interface reference：

```text
out_ready
out_valid
out_tile_t_base
out_tile_oc_base
out_row_valid_mask
out_last_group
out_tile[4 rows x 12 channels]
```

## 6. fc_classifier_96_5

文件：

```text
emg_gesture_vivado/src/fc_classifier_96_5.v
```

### 6.1 模块职责

```text
输入:  PW2 GAP accumulator, 96 channels
GAP:   gap_avg[ch] = gap_sum_vec[ch] / 174
FC0: 96 -> 96
ReLU
FC1: 96 -> 5
输出: logits_vec[5] + argmax class_idx
```

一次 `start`：

```text
1. 锁存 96 个 gap_sum_vec。
2. 逐通道 /174 得到 96 个 Q8.8 GAP average。
3. FC0 使用 16 个 output lanes，分 6 个 out_base group。
4. FC0 输出过 ReLU，写入 hidden_mem[96]。
5. FC1 5 个 class lanes 并行算 5 个 logit。
6. 对 5 个 logit 做 argmax。
```

### 6.2 PW2 到 FC 输入接口

接口：

```text
start
gap_count
gap_sum_vec[96]
```

语义：

```text
gap_count 正常应为 174。
模块会输出 input_count_error = (gap_count != 174)，但计算仍固定使用 /174。

gap_sum_vec 元素是 PW2 post-BN/ReLU 的逐通道累加和。
每个元素 48-bit signed，逻辑上仍是 Q8.8 的整数累加。
```

当前 averagepool 实现：

```text
gap_avg[ch] = sat_q8_8(gap_sum_vec[ch] / GAP_LEN)
GAP_LEN 默认 174。
```

注意：

```text
当前 RTL 使用常数除法 `/ GAP_LEN`，仿真和语义最直接。
若综合资源不理想，可以替换成 reciprocal multiply/shift，
但需要同步 bit-exact reference。
```

### 6.3 FC0 权重和 bias

接口：

```text
fc0_weight_we
fc0_weight_out = 0..95
fc0_weight_k   = 0..96
fc0_weight_wdata
```

存储：

```text
fc0_weight_mem[96][97]
```

bias 约定：

```text
k = 0..95: FC0 data weight
k = 96:    FC0 bias
```

计算：

```text
for out_base in 0,16,32,48,64,80:
    acc[lane] = bias[out_base + lane] << FRAC_BITS
    for k in 0..95:
        acc[lane] += gap_avg[k] * W_fc0[out_base + lane][k]
    hidden[out_base + lane] = ReLU(sat_q8_8(acc >> FRAC_BITS))
```

### 6.4 FC1 权重和 bias

接口：

```text
fc1_weight_we
fc1_weight_out = 0..4
fc1_weight_k   = 0..96
fc1_weight_wdata
```

存储：

```text
fc1_weight_mem[5][97]
```

bias 约定：

```text
k = 0..95: FC1 data weight
k = 96:    FC1 bias
```

计算：

```text
for class in 0..4:
    acc[class] = bias[class] << FRAC_BITS

for k in 0..95:
    for class in 0..4:
        acc[class] += hidden[k] * W_fc1[class][k]

logits[class] = sat_q8_8(acc[class] >> FRAC_BITS)
class_idx = argmax(logits)
```

argmax 平局策略：

```text
只在 logits[class] > current_best 时更新。
所以完全相等时选择最低 class index。
```

### 6.5 调度和延迟

默认参数：

```text
FC0_OUT_LANES = 16
FC1_OUT = 5
```

周期估算：

```text
AVG: 96 cycles
FC0: 6 groups * (1 init + 96 MAC + 1 store) = 588 cycles
FC1: 1 init + 96 MAC + 1 store = 98 cycles
ARGMAX/OUT: 约 2 cycles + result_ready 等待

合计约 784 cycles，不含 result_ready 反压。
100 MHz 下约 7.8 us。
```

### 6.6 输出接口

```text
result_valid / result_ready
logits_vec[5]
class_idx
done
input_count_error
```

语义：

```text
result_valid=1 时 logits_vec/class_idx 保持稳定。
result_ready=1 后 done 脉冲 1 拍并回到 IDLE。

done 表示分类结果已被后级接受，不只是内部计算完成。
```

## 7. 参数导出时的统一约定

从 PyTorch checkpoint 导出参数时，建议生成硬件专用初始化文件，而不是直接使用 PyTorch 原始 layout。

### 7.1 量化格式

当前 RTL 假设：

```text
activation: Q8.8 signed 16-bit
weight:     Q8.8 signed 16-bit
folded BN:  Q8.8 signed 16-bit
output:     Q8.8 signed 16-bit
acc:        48-bit
```

软件导出时需要：

```text
q = round(float_value * 2^8)
sat to [-32768, 32767]
```

### 7.2 folded BN

Conv + BN 推理折叠：

```text
y = conv * gamma / sqrt(var + eps) + (beta - mean * gamma / sqrt(var + eps))
```

导出：

```text
w_fold[oc]    = gamma[oc] / sqrt(var[oc] + eps)
bias_fold[oc] = beta[oc] - mean[oc] * w_fold[oc]
```

当前 RTL 执行：

```text
conv_q = conv_acc >> FRAC_BITS
bn_mul = conv_q * w_fold
bn_q   = (bn_mul >> FRAC_BITS) + bias_fold
relu
```

注意：

```text
这是 conv 先量化到 Q8.8 再乘 folded scale。
如果软件 golden 使用 full precision folded conv，可能会有逐层量化误差。
如果软件 golden 使用 full precision folded conv，建议另做 bit-exact Python reference。
```

### 7.3 当前 weight_data 文件

当前 `weight_data/` 已按各计算模块的实际加载接口分别导出：

```text
stem/weight_packed.mem      // 8 groups x 35 k, 4 lanes/word
stem/bn_scale.mem
stem/bn_bias.mem

dw1/weight.mem              // 32 ch x 5 tap, ch outer/tap inner

pw1/weight_packed.mem       // 8 groups x 32 k, 8 lanes/word
pw1/bn_scale.mem
pw1/bn_bias.mem

dw2/weight.mem              // 64 ch x 5 tap, ch outer/tap inner

pw2/weight_packed.mem       // 8 groups x 64 k, 12 lanes/word
pw2/bn_scale.mem
pw2/bn_bias.mem

fc0/weight.mem              // 96 out x 97 k, row-major, k=96 is bias
fc1/weight.mem              // 5 out x 97 k, row-major, k=96 is bias
```

其中 stem/PW 的 packed word bit layout 必须统一：

```text
word[lane*DATA_W +: DATA_W] = lane_value
```

FC0/FC1 当前文件不是 packed word，而是直接匹配 `fc_classifier_96_5.v`
的写端口：

```text
fc0 line = out_idx * 97 + k, out_idx=0..95, k=0..96
fc1 line = out_idx * 97 + k, out_idx=0..4,  k=0..96
k=96 is bias
```

如果后续 FC 权重改成物理 packed/banked ROM，可以从这两个 row-major
文件离线重排；计算语义仍保持 `W_aug[out][k]`，`k=96` 为 bias。

## 9. 当前判断

当前计算模块已经完成，读取方式合理且可实现。

关键不是让物理 ROM/BRAM 直接长成：

```text
W[oc_base+lane][k]
A[tile_t_base+row][k]
```

而是设计中间的 controller/loader，让它们把任意实际存储布局转换成计算核期望的 vector。

最重要的工程约束：

```text
1. lane 顺序必须统一。
2. response 顺序不能乱。
3. PW1/PW2 full-channel job 期间 input tile 必须稳定。
4. stem activation loader 必须处理 flattened k 的 im2col。
5. DW window loader 必须用 valid bitmap 判断非 padding 数据是否可读。
6. FC0/FC1 bias 必须按 augmented k=96 写入。
```
