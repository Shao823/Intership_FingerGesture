# 无线通信架构与协议设计

## 1. 设计目标

FPGA 完成手势识别后，需要将识别结果转换成控制命令并无线发送到 PC/Game/上位机。

目标：

- 低延迟：手势控制链路应尽量稳定在毫秒级。
- 高可靠：支持 CRC、序号、ACK、重传。
- 易调试：保留 UART 日志和 PC 端解析工具。
- 低功耗：无线模块空闲时可降功耗，减少无意义重传。
- 可扩展：同一包格式支持 UART、SPI、BLE、WiFi、NRF24L01、ESP32。

推荐分层：

```text
CNN Result
-> Command Mapper
-> Packet Formatter
-> TX FIFO
-> Wireless Driver
-> Wireless Module
-> PC/Game Receiver
```

## 2. 通信方案选择

### 2.1 UART

适用场景：

- 调试日志。
- 低速命令发送。
- FPGA/PS 与 ESP32 简单连接。

优点：

- 实现简单。
- 资源占用低。
- 易用串口工具验证。

缺点：

- 带宽低。
- 抗干扰和重传需软件协议处理。
- 高帧率结果输出时容易阻塞。

建议：

- UART 作为 debug 和 fallback 通道。
- 典型波特率：`115200`、`921600`、`1 Mbps`。

### 2.2 SPI

适用场景：

- FPGA/PS 与 ESP32 或 NRF24L01 通信。
- 低延迟、高可靠控制链路。

优点：

- 延迟低。
- 带宽明显高于 UART。
- FPGA 端易做硬件 FIFO + DMA。

缺点：

- 需要明确 master/slave。
- 线数比 UART 多。
- 协议层仍需包格式和 CRC。

建议：

- FPGA/PS 作为 SPI master，ESP32/NRF24L01 作为 slave。
- 对实时手势控制，SPI 是 FPGA 到无线模块的首选接口。

### 2.3 BLE

适用场景：

- 手机、平板、低功耗外设。
- 控制频率中等、功耗敏感。

优点：

- 功耗低。
- 移动设备支持好。

缺点：

- 延迟和连接间隔受 BLE 参数影响。
- 吞吐较低。
- FPGA 直接实现 BLE 不现实，通常依赖 ESP32/nRF52 模块。

建议：

- FPGA -> SPI/UART -> ESP32/nRF52 -> BLE。
- BLE 适合移动端控制，不适合极低延迟游戏控制。

### 2.4 WiFi

适用场景：

- PC/Game 接收。
- 需要较高吞吐、网络化传输。

优点：

- 带宽高。
- PC 端开发方便，UDP/TCP 均可。

缺点：

- 功耗较高。
- 延迟抖动比 SPI/NRF24 更明显。
- 网络配置复杂。

建议：

- 使用 ESP32 作为 WiFi 网关。
- 实时控制优先 UDP，可靠传输或配置命令使用 TCP。

### 2.5 NRF24L01

适用场景：

- 点对点低延迟无线控制。
- 简单 PC/Game 接收器或 MCU 接收器。

优点：

- 低功耗。
- 低延迟。
- 支持硬件 CRC、ACK、自动重传。
- SPI 接口简单。

缺点：

- 需要接收端也有 NRF24L01。
- 不直接连接普通 PC/手机。
- 2.4 GHz 干扰环境下需要跳频或频道选择。

建议：

- 若最终 Demo 是专用接收器，NRF24L01 是低延迟优先方案。
- 若最终接 PC/手机，ESP32 更方便。

### 2.6 ESP32

适用场景：

- FPGA 到 PC/手机/Game 的无线网关。
- 支持 BLE 和 WiFi。

优点：

- 成熟、便宜、开发工具完整。
- 可用 SPI/UART 接 FPGA。
- 可通过 WiFi UDP 直接发 PC。

缺点：

- 功耗高于 NRF24L01。
- WiFi 延迟抖动需要协议层容错。

建议：

- 推荐主方案：FPGA/PS -> SPI -> ESP32 -> WiFi UDP -> PC/Game。
- 备用方案：FPGA/PS -> UART -> ESP32 -> BLE/WiFi。

## 3. 推荐总体架构

### 3.1 低延迟 PC/Game 方案

```text
CNN Accelerator
-> Result FIFO
-> Command Mapper
-> Packet Formatter
-> SPI TX FIFO
-> ESP32
-> WiFi UDP
-> PC/Game
```

特点：

- FPGA 与 ESP32 间用 SPI，减少本地接口延迟。
- ESP32 到 PC 用 UDP，减少协议栈等待。
- 应用层使用 seq、CRC、ACK/NAK 可选机制。

### 3.2 低功耗点对点方案

```text
CNN Accelerator
-> Result FIFO
-> Packet Formatter
-> SPI
-> NRF24L01
-> NRF24L01 Receiver
-> PC/Game MCU Bridge
```

特点：

- NRF24L01 硬件支持 ACK 和重传。
- 功耗低，延迟稳定。
- PC 端需要额外接收器。

### 3.3 调试方案

```text
CNN Accelerator
-> Result FIFO
-> Packet Formatter
-> UART
-> USB Serial
-> PC Debug Tool
```

特点：

- 最容易验证协议。
- 不适合作为最终高帧率主通道。

## 4. 内部模块设计

### 4.1 Command Mapper

输入：

- `class_id`
- `confidence`
- `timestamp`
- `frame_id`

输出：

- `command_id`
- `command_value`

职责：

- 将手势类别映射为游戏命令。
- 根据 confidence 过滤低置信度结果。
- 支持 hold、repeat、debounce。

建议：

- 设置 `confidence_threshold`。
- 连续 N 帧一致再触发稳定命令。
- 对游戏控制可输出 press/release/hold。

### 4.2 Packet Formatter

职责：

- 组包。
- 添加 header、length、seq、timestamp、payload、CRC。
- 写入 TX FIFO。

建议：

- FPGA 端实现固定长度或短变长包。
- CRC 可用 CRC-16-CCITT，硬件成本低。

### 4.3 TX FIFO / RX FIFO

职责：

- 解耦 CNN result 与无线发送速率。
- 支持 backpressure。

建议：

- TX FIFO 深度至少覆盖最大无线阻塞时间。
- 若 TX FIFO 接近满，丢弃旧包或低优先级包，不阻塞 CNN 主链路。

### 4.4 Wireless Driver

职责：

- SPI/UART 驱动。
- ACK/NAK 处理。
- 重传计数。
- 超时检测。

建议：

- 协议状态机放 PS 软件或 PL Instruction Controller。
- 对 NRF24L01，可利用硬件 ACK 和 auto retransmit。
- 对 ESP32 UDP，需要应用层 ACK 可选。

## 5. Packet Format

### 5.1 通用短包格式

适用于 UART、SPI、BLE、WiFi UDP、NRF24L01 payload。

```text
+--------+--------+--------+--------+--------+--------+---------+---------+-------+
| SOF0   | SOF1   | VER    | TYPE   | LEN    | SEQ    | FLAGS   | PAYLOAD | CRC16 |
| 0xA5   | 0x5A   | 1 byte | 1 byte | 1 byte | 1 byte | 1 byte  | N bytes | 2 B   |
+--------+--------+--------+--------+--------+--------+---------+---------+-------+
```

字段说明：

| 字段 | 长度 | 说明 |
|---|---:|---|
| `SOF0` | 1 | 固定 `0xA5` |
| `SOF1` | 1 | 固定 `0x5A` |
| `VER` | 1 | 协议版本，初始为 `0x01` |
| `TYPE` | 1 | 包类型 |
| `LEN` | 1 | payload 长度 |
| `SEQ` | 1 | 包序号，循环递增 |
| `FLAGS` | 1 | ACK 请求、重传标记、错误标记 |
| `PAYLOAD` | N | 业务数据 |
| `CRC16` | 2 | 对 `VER` 到 `PAYLOAD` 计算 CRC |

CRC 字节序：

```text
CRC_L
CRC_H
```

### 5.2 TYPE 定义

```text
0x01 GESTURE_RESULT
0x02 COMMAND
0x03 HEARTBEAT
0x04 ACK
0x05 NAK
0x06 CONFIG
0x07 STATUS
0x08 ERROR
```

### 5.3 FLAGS 定义

```text
bit0 ACK_REQUIRED
bit1 IS_RETRANSMIT
bit2 LOW_CONFIDENCE
bit3 DROPPED_FRAME
bit4 RESERVED
bit5 RESERVED
bit6 RESERVED
bit7 ERROR
```

## 6. Payload Format

### 6.1 GESTURE_RESULT Payload

```text
+----------+------------+------------+------------+-------------+----------+
| frame_id | timestamp  | class_id   | confidence | raw_score   | status   |
| 2 bytes  | 4 bytes    | 1 byte     | 1 byte     | 2 bytes     | 1 byte   |
+----------+------------+------------+------------+-------------+----------+
```

长度：`11 bytes`

说明：

| 字段 | 说明 |
|---|---|
| `frame_id` | CNN 输入帧号 |
| `timestamp` | FPGA/PS 时间戳，单位可设为 us 或 tick |
| `class_id` | 手势类别 |
| `confidence` | 0-255 量化置信度 |
| `raw_score` | 可选 logits 或最大分数，定点格式 |
| `status` | bit 标记，如 valid、low confidence |

### 6.2 COMMAND Payload

```text
+------------+---------------+-------------+------------+----------+
| command_id | command_value | duration_ms | frame_id   | status   |
| 1 byte     | 1 byte        | 2 bytes     | 2 bytes    | 1 byte   |
+------------+---------------+-------------+------------+----------+
```

长度：`7 bytes`

示例：

```text
command_id:
0x01 LEFT
0x02 RIGHT
0x03 UP
0x04 DOWN
0x05 FIRE
0x06 PAUSE

command_value:
0x00 RELEASE
0x01 PRESS
0x02 HOLD
```

### 6.3 ACK Payload

```text
+---------+--------+
| ack_seq | status |
| 1 byte  | 1 byte |
+---------+--------+
```

### 6.4 STATUS Payload

```text
+----------+----------+------------+------------+----------+-------------+
| frame_id | tx_count | retry_count| drop_count | fifo_lvl | link_status |
| 2 bytes  | 2 bytes  | 2 bytes    | 2 bytes    | 1 byte   | 1 byte      |
+----------+----------+------------+------------+----------+-------------+
```

## 7. CRC 设计

推荐：

```text
CRC-16-CCITT
poly = 0x1021
init = 0xFFFF
xorout = 0x0000
```

原因：

- 硬件实现简单。
- 对短包足够可靠。
- PC、ESP32、MCU 端都有成熟实现。

CRC 范围：

```text
VER, TYPE, LEN, SEQ, FLAGS, PAYLOAD
```

不包含：

```text
SOF0, SOF1, CRC16 自身
```

## 8. 重传策略

### 8.1 强可靠模式

适用：

- 配置包。
- 状态控制包。
- 关键命令。

流程：

```text
TX sends packet with ACK_REQUIRED
TX starts timeout timer
RX checks CRC
RX sends ACK(seq) or NAK(seq)
TX retries if timeout or NAK
TX drops packet after max_retry
```

建议参数：

```text
ack_timeout = 2 ms 到 10 ms
max_retry = 2 到 3
```

### 8.2 实时低延迟模式

适用：

- 高频手势结果。
- 游戏控制方向更新。

策略：

- 不对每个结果强制 ACK。
- 使用 `SEQ` 检测丢包。
- 新包优先于旧包。
- TX FIFO 满时丢弃旧结果。

原因：

- 手势结果具有时效性。
- 过期包重传会增加控制延迟。

### 8.3 NRF24L01 模式

建议：

- 使用 NRF24L01 硬件 CRC。
- 使用 auto ACK。
- 使用 auto retransmit。
- 应用层仍保留 `SEQ` 和 `frame_id`，用于检测重复包和乱序。

## 9. 同步机制

### 9.1 帧同步

使用 `SOF0=0xA5`、`SOF1=0x5A` 搜索包头。

接收状态机：

```text
WAIT_SOF0 -> WAIT_SOF1 -> READ_HEADER -> READ_PAYLOAD -> READ_CRC -> CHECK
```

CRC 错误：

- 丢弃当前包。
- 重新搜索 SOF。
- 可发送 NAK。

### 9.2 时间同步

可选方案：

- FPGA/PS 维护本地 timestamp。
- PC 接收端记录 host timestamp。
- 周期性发送 HEARTBEAT 包估计链路延迟。

HEARTBEAT 周期：

```text
100 ms 到 1000 ms
```

### 9.3 命令去抖

建议：

- 连续 N 帧相同手势才输出 `PRESS`。
- 连续 M 帧消失才输出 `RELEASE`。
- 低 confidence 包只做状态上报，不触发命令。

## 10. Interrupt 与 DMA

### 10.1 中断源

推荐中断：

```text
CNN_DONE_IRQ
TX_FIFO_ALMOST_EMPTY_IRQ
TX_FIFO_ALMOST_FULL_IRQ
WIRELESS_TX_DONE_IRQ
WIRELESS_RX_ACK_IRQ
WIRELESS_ERROR_IRQ
DMA_DONE_IRQ
```

### 10.2 中断使用建议

- CNN 每帧完成后产生 `CNN_DONE_IRQ`。
- Command Mapper 或 PS 读取 Result FIFO。
- Packet Formatter 写 TX FIFO。
- Wireless Driver 在 FIFO 非空时发送。
- ACK 或错误由无线驱动中断处理。

### 10.3 DMA 使用建议

适合 DMA 的场景：

- 批量发送日志。
- 大量结果记录回 DDR。
- ESP32 SPI 高速批量传输。

不适合 DMA 的场景：

- 每帧只有 8-32 字节的小命令包，DMA 启动开销可能大于收益。

建议：

- 小包使用 FIFO + interrupt。
- 连续批量状态或日志使用 DMA。

## 11. Timing Diagrams

### 11.1 单帧低延迟发送

```text
Time ->

CNN          | compute frame N | done |
Result FIFO |                 | write result |
Mapper      |                       | map command |
Packet      |                            | format + CRC |
SPI/UART    |                                  | transmit packet |
Wireless    |                                           | air tx |
Receiver    |                                                 | parse + apply |
```

目标：

```text
CNN done -> packet ready < 100 us
local wired transmit < 100 us 到 1 ms
wireless air + receiver < 1 ms 到 20 ms，取决于模块
```

### 11.2 ACK/重传时序

```text
TX                                      RX
|                                       |
| ---- DATA(seq=10, ACK_REQUIRED) ----> |
|                                       | CRC OK
| <------------- ACK(10) -------------- |
|                                       |
| mark seq=10 done                      |
```

超时重传：

```text
TX                                      RX
| ---- DATA(seq=11, ACK_REQUIRED) ----> |  lost
| wait ack_timeout                      |
| ---- DATA(seq=11, RETRANSMIT) ------> |
| <------------- ACK(11) -------------- |
```

### 11.3 Ping-Pong + Wireless 并行时序

```text
Time ->

Buffer A   | DMA/Preprocess write N+1 |
Buffer B   | CNN compute N            |
Result     |              result N -> packet |
Wireless   |                    send command N |

Next:
Buffer B   | DMA/Preprocess write N+2 |
Buffer A   | CNN compute N+1          |
Wireless   |                    send command N+1 |
```

目标：

```text
输入准备、CNN 计算、无线发送三者重叠。
系统周期接近三者中最慢的那个阶段。
```

### 11.4 接收端同步状态机

```text
WAIT_SOF0
  |
  v
WAIT_SOF1
  |
  v
READ_HEADER
  |
  v
READ_PAYLOAD
  |
  v
READ_CRC
  |
  v
CHECK_CRC -- fail --> WAIT_SOF0
  |
 pass
  v
DISPATCH -> ACK/COMMAND/APPLY
```

## 12. 功耗建议

UART：

- 最低资源开销。
- 低速下功耗低，但发送时间长。

SPI：

- 本地链路功耗中等。
- 发送时间短，整体能耗通常可控。

BLE：

- 低功耗优先。
- 延迟受连接参数影响。

WiFi：

- 功耗最高。
- 适合 PC/Game 快速接入。

NRF24L01：

- 低功耗和低延迟平衡较好。
- 适合专用接收器。

ESP32：

- 功耗高于 NRF24L01。
- 工程便利性高。

建议：

- Demo 快速落地：ESP32 WiFi UDP。
- 低延迟专用控制：NRF24L01。
- 手机低功耗控制：ESP32/nRF52 BLE。

## 13. 验证方法

### 13.1 单元验证

- Packet Formatter testbench。
- CRC 软硬件 golden model 比对。
- Parser 状态机随机字节流测试。
- FIFO overflow/underflow 测试。
- SEQ wraparound 测试。

### 13.2 链路验证

- UART loopback。
- SPI loopback。
- ESP32 收包打印。
- NRF24L01 点对点 ACK 测试。
- BLE/WiFi PC 端脚本收包。

### 13.3 故障注入

- 随机 bit flip，验证 CRC。
- 丢包，验证 seq 检测。
- ACK 丢失，验证重传。
- 包头错位，验证同步恢复。
- FIFO 满，验证丢弃策略。

### 13.4 性能验证

记录指标：

```text
cnn_done_time
packet_ready_time
tx_start_time
tx_done_time
rx_time
apply_command_time
retry_count
drop_count
crc_error_count
fifo_level_max
```

计算：

```text
local_latency = tx_start_time - cnn_done_time
air_latency = rx_time - tx_start_time
end_to_end_latency = apply_command_time - cnn_done_time
packet_loss_rate = lost_packets / total_packets
retry_rate = retry_count / total_packets
```

### 13.5 长时间稳定性测试

- 连续运行 1 小时。
- 高频手势切换。
- 无线干扰环境测试。
- 低电压或电池供电测试。
- PC/Game 端命令一致性检查。

## 14. 推荐协议参数

第一版推荐：

```text
local link: SPI
wireless module: ESP32 WiFi UDP
debug link: UART 921600
packet CRC: CRC-16-CCITT
seq width: 8 bit
frame_id width: 16 bit
timestamp width: 32 bit
TX FIFO depth: >= 16 packets
ACK mode: config/status 使用 ACK，gesture/command 默认不逐包 ACK
heartbeat: 500 ms
max_retry: 2
ack_timeout: 5 ms
```

低功耗专用版推荐：

```text
local link: SPI
wireless module: NRF24L01
NRF hardware CRC: enabled
auto ACK: enabled
auto retransmit: enabled
application seq: enabled
payload size: <= 32 bytes
```

## 15. 最终建议

如果目标是尽快完成 PC/Game Demo：

```text
FPGA/PS -> SPI -> ESP32 -> WiFi UDP -> PC/Game
```

如果目标是低功耗、低延迟、专用手势控制器：

```text
FPGA/PS -> SPI -> NRF24L01 -> NRF24L01 Receiver -> PC/Game Bridge
```

无论选择哪种无线模块，建议统一使用同一套应用层包格式。这样可以先用 UART 调试，再切换到 SPI+ESP32 或 SPI+NRF24L01，而不需要重写上层协议。
