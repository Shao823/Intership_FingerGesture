## 6. XSim 仿真失败根因

AXI VIP package 和 testbench 类型本身没有编译错误。真实 `xvlog.log` 已经成功分析：

```text
emg_finger_sim_axi_vip_0_0_pkg.sv
tb_emg_finger_sim.sv
```

Vivado 编辑器对以下内容标红：

```systemverilog
import axi_vip_pkg::*;
import emg_finger_sim_axi_vip_0_0_pkg::*;
```

主要是编辑器静态索引无法解析预编译 VIP library，不是当前 XSim compile 的真实失败点。

真实失败为：

```text
lib_fifo_v1_0_16.vdbl needs to be re-saved
since fifo_generator_v13_2_7 changed

lib_fifo_v1_0_16.sync_fifo_fg failed to restore
axi_sg_v4_1_15.axi_sg failed to restore
axi_dma_v7_1_28.axi_dma failed to restore
```

进一步检查发现：

```text
E:/Xilinx_2022/Vivado/2022.2/data/xsim/ip/fifo_generator_v13_2_7
```

中有一批 `.vdb` 文件在今天被重新生成，而依赖它的 `lib_fifo/axi_sg/axi_dma` 仍是安装时的预编译版本，导致 XSim library dependency identity 不一致。

该问题会让当前 `sim_1` 中任何顶层在编译到 DMA VHDL wrapper 时失败，包括原本可以通过的 `tb_accelerator`。这不代表 accelerator RTL 被修改坏，而是当前 simulation fileset 同时编译了 BD DMA 生成文件。

当前处理决定：

```text
1. 暂时不阻塞综合和上板工作。
2. 后续通过 Vivado 2022.2 installer repair 或同版本干净重装恢复 XSim IP library。
3. 修复后先运行 tb_accelerator，再恢复 emg_finger_sim 四组系统仿真。
4. 不再通过修改 AXI VIP import 或 testbench 类型绕过该问题。
```

## 9. 设计风险复核和合理修改方向

### 9.1 SmartConnect `MAX_BURST_LENGTH=4`

工程中确实存在：

```text
axi_smc/M00_AXI/MAX_BURST_LENGTH = 4
S_AXI_HP0_FPD data width          = 128 bit
```

但“直接把 4 改成 64/256 就会减少 1740-byte 帧的 burst 数量”这个结论需要修正。

当前 DMA 配置为：

```text
C_M_AXI_MM2S_DATA_WIDTH = 32 bit
C_MM2S_BURST_SIZE       = 16 beat
```

因此 DMA 每个 memory burst 搬运：

```text
16 beat × 4 byte = 64 byte
```

经过 SmartConnect 32->128 bit width conversion 后，64 byte 正好是：

```text
4 beat × 16 byte = 64 byte
```

所以当前 `M00_AXI MAX_BURST_LENGTH=4` 与 DMA当前 64-byte burst 实际上是匹配的。1740 byte 约 28 个 burst，主要由 DMA自己的 32-bit memory width和 16-beat burst配置决定，不是 SmartConnect单独造成。SmartConnect通常也不会把多个独立 upstream burst自动合并成一个更长 burst。

明日合理方向：

```text
1. 不单独盲改 SmartConnect MAX_BURST_LENGTH。
2. 先以当前配置上板测量 DMA传输延迟。
3. 如果 DMA确实成为瓶颈，优先评估将 DMA M_AXI_MM2S memory-map width改为128 bit。
4. 同时检查 DMA burst size和SmartConnect downstream max burst，保持端到端一致。
5. 每次参数修改后重新Validate BD、生成output products、综合，并用ILA观察传输周期。
```

结论：

```text
参数值4真实存在，但当前不能直接判定为错误；它更像与现有32-bit DMA配置匹配的结果。
```

### 9.2 HP0 非一致性和 cache coherency

当前 DMA通过：

```text
S_AXI_HP0_FPD / SAXIGP2
```

访问 DDR。这不是 APU cache coherent 接口。因此 CPU通过 cache写入 input buffer 后，DMA不保证立即看到 cache中的最新内容。

这是当前拓扑必须明确的软件契约。

Bare-metal 第一版必须在启动 DMA 前执行：

```c
Xil_DCacheFlushRange((INTPTR)input_buffer, 1740);
```

同时应保证：

```text
1. input buffer地址满足DMA对齐要求。
2. flush长度覆盖完整1740 bytes，最好按cache line向上对齐。
3. 如果以后加入S2MM输出，CPU读取DMA写回数据前还需要invalidate cache。
```

Linux/PetaLinux 方向应使用 DMA API/coherent buffer，而不是普通 cached userspace buffer直接交给硬件。

明日应将该约束补入：

```text
ip_repo/emg_accelerator_1_0/PACKAGING_AND_INTEGRATION.md
```

并在第一版 Vitis 软件中封装统一的：

```text
prepare_input_buffer()
flush_input_cache()
start_dma()
```

### 9.3 AXI-Lite AW/W 通道耦合

当前实现确实要求：

```verilog
S_AXI_AWVALID && S_AXI_WVALID && !axi_bvalid
```

同时成立后才拉高 AWREADY/WREADY。

AXI4-Lite 允许 AW 和 W 独立到达，所以该实现虽然通常能与 Xilinx PS master配合，但不是严格健壮的 slave实现，属于 latent compatibility bug。

明日修改方向：

```text
1. 分别增加 aw_hold_valid/aw_hold_addr。
2. 分别增加 w_hold_valid/w_hold_data/w_hold_strb。
3. AW 和 W 各自独立握手并锁存。
4. 当两边hold均有效且当前无BVALID时执行一次寄存器写。
5. 写完成后同时清除两个hold valid并产生BVALID。
6. 保持当前单 outstanding transaction模型，不必增加复杂队列。
```

必须补充定向测试：

```text
AW先到、W后到
W先到、AW后到
AW/W同拍到达
BREADY延迟
连续两次写事务
```

完成后重新运行 AXI-Lite testbench、AXI shell testbench和10组DMA系统仿真，再重新package IP。

### 9.4 CONTROL 写覆盖 `AUTO_START/IRQ_ENABLE`

当前实现确实存在：

```verilog
auto_start <= S_AXI_WDATA[1];
irq_enable <= S_AXI_WDATA[4];
```

因此每次写 `CONTROL` 都会覆盖两个持久位。例如只写：

```text
0x01 = START
```

会同时把：

```text
AUTO_START = 0
IRQ_ENABLE = 0
```

短期软件必须执行 read-modify-write，或者每次写完整控制字。

控制位常量需要准确区分：

```text
0x13 = START | AUTO_START | IRQ_ENABLE
0x19 = START | CLEAR_ERRORS | IRQ_ENABLE
0x16 = AUTO_START | CLEAR_DONE | IRQ_ENABLE
0x1A = AUTO_START | CLEAR_ERRORS | IRQ_ENABLE
```

因此“`0x13 = START | CLEAR_ERRORS | IRQ_ENABLE`”是不正确的，`CLEAR_ERRORS` 是 bit3，对应正确值应为 `0x19`。

明日建议优先选择更清晰的寄存器语义：

```text
方案A：保留一个CONTROL寄存器，但软件统一read-modify-write。
方案B：START/CLEAR_DONE/CLEAR_ERRORS采用write-one pulse，AUTO_START/IRQ_ENABLE仅在对应WSTRB/bit更新。
方案C：拆分COMMAND和CONFIG寄存器：
       COMMAND = START/CLEAR_DONE/CLEAR_ERRORS
       CONFIG  = AUTO_START/IRQ_ENABLE
```

推荐方案C，软件接口最清晰，也避免命令位和持久配置位互相覆盖。修改寄存器映射时需要同步：

```text
RTL
testbench
AXI IP详细设计文档
PACKAGING_AND_INTEGRATION.md
Vitis driver/header
```
