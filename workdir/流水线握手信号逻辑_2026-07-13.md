# Stem-DW1-PW1-DW2-PW2-FC 流水线握手信号逻辑

本文基于当前 `emg_gesture_vivado/src/top/stem_dw1_pw1_dw2_pw2_pipeline/stem_dw1_pw1_dw2_pw2_pipeline_top.v` 的连接关系整理。重点说明每根 `valid/ready` 是哪个模块输出、哪个模块输入，以及握手成功时发生什么。

## 1. 基本规则

所有 ready/valid 数据通路都按这个规则理解：

```text
发送方：输出 valid + data，输入 ready
接收方：输入 valid + data，输出 ready

fire = valid && ready
```

`fire` 为 1 的那个时钟上升沿，发送方认为数据被接收，接收方锁存或消费数据。

注意：

- `start/done/busy` 不是 ready/valid 握手，是顶层调度用的一拍启动脉冲和状态信号。
- BRAM IP 本身没有 ready/valid。BRAM 的读写由外层 buffer 控制器用 `bram_*_en/addr/data` 管理。
- 原有的 pending FIFO 不搬运 activation 数据，只搬运“这个 tile 已经写入对应级间 buffer，可以启动下一层”的任务元数据。
- 新加的 `emg_stream_fifo_slice` 搬运真实数据 payload，用来隔离 `ready` 反压路径。

## 2. 通用 stream FIFO slice

模块：`emg_stream_fifo_slice`

端口方向：

| 信号 | 方向 | 含义 |
|---|---|---|
| `in_valid` | producer 输出，FIFO 输入 | 上游有数据要写入 FIFO |
| `in_ready` | FIFO 输出，producer 输入 | FIFO 入口有空间 |
| `in_data` | producer 输出，FIFO 输入 | 写入 FIFO 的 payload |
| `out_valid` | FIFO 输出，consumer 输入 | FIFO 出口有数据 |
| `out_ready` | consumer 输出，FIFO 输入 | 下游愿意接收 FIFO 出口数据 |
| `out_data` | FIFO 输出，consumer 输入 | FIFO 出口 payload |
| `busy` | FIFO 输出 | FIFO 非空 |

内部逻辑：

```verilog
in_ready  = (count < DEPTH);
out_valid = (count != 0);
in_fire   = in_valid && in_ready;
out_fire  = out_valid && out_ready;
```

所以经过 FIFO 后，一条原本直接的握手会被拆成两段：

```text
producer -> FIFO 入口
FIFO 出口 -> real consumer
```

第一段 fire 只说明数据进入 FIFO；第二段 fire 才说明数据进入真正下游模块。

## 3. 顶层调度启动信号

这些信号由顶层状态机输出给各计算模块：

| 信号 | 输出方 | 输入方 | 含义 |
|---|---|---|---|
| `stem_start` | top | `u_stem` | 启动一个 stem 4-row x 4-oc tile |
| `dw1_start` | top | `u_dw1` | 启动一个 DW1 job |
| `pw1_start` | top | `u_pw1` | 启动一个 PW1 tile |
| `dw2_start` | top | `u_dw2` | 启动一个 DW2 job |
| `pw2_start` | top | `u_pw2` | 启动一个 PW2 tile |
| `fc_start` | top | `u_fc` | 启动 FC |

这些都是一拍脉冲，不带 `ready`。顶层只在对应模块不 busy、不 start，并且调度条件满足时拉高。

关键调度关系：

```text
stem_issue_now 需要 dw1_pending_launch_room
dw1_start_now 需要 !dw1_pending_empty
dw2_start_now 需要 !dw2_pending_empty
fc_start 需要 front_complete_now && !fc_started && !fc_busy
```

`dw1_pending` 和 `dw2_pending` 是任务 FIFO，不是数据 FIFO。

## 4. Stem 输出到 stem_to_dw1_buffer

### 4.1 第一段：u_stem -> u_stem_to_dw1_ready_slice

`u_stem` 是发送方，`u_stem_to_dw1_ready_slice` 的入口是接收方。

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `stem_out_valid` | `u_stem` | FIFO `in_valid` | stem 有一个输出 tile |
| `stem_to_buffer_ready` | FIFO `in_ready` | `u_stem.out_ready` | FIFO 入口有空间 |
| `stem_out_tile_t_base` | `u_stem` | top 打包到 `stem_to_dw1_slice_in` | tile 时间基址 |
| `stem_out_tile_oc_base` | `u_stem` | top 打包到 `stem_to_dw1_slice_in` | tile 输出通道基址 |
| `stem_out_tile` | `u_stem` | top 打包到 `stem_to_dw1_slice_in` | tile 数据 |

payload 打包：

```verilog
stem_to_dw1_slice_in = {
    stem_out_tile_t_base,
    stem_out_tile_oc_base,
    stem_out_tile
};
```

握手：

```verilog
stem_to_dw1_slice_input_fire = stem_out_valid && stem_to_buffer_ready;
```

这个 fire 成立时，stem tile 只进入 `u_stem_to_dw1_ready_slice`，还不一定已经进入真正的 `stem_to_dw1_buffer`。

`stem_to_buffer_ready` 现在只表示：

```text
stem_to_dw1 stream FIFO 还有空间
```

它不再直接表示：

```text
stem_to_dw1_buffer 的 ring slot 当前可写
```

### 4.2 第二段：u_stem_to_dw1_ready_slice -> u_stem_to_dw1

FIFO 是发送方，`u_stem_to_dw1` 是真正的级间 buffer 接收方。

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `stem_to_dw1_in_valid` | FIFO `out_valid` | `u_stem_to_dw1.stem_out_valid` | FIFO 出口有 stem tile |
| `stem_to_dw1_in_ready` | `u_stem_to_dw1.stem_out_ready` | FIFO `out_ready` | 真实级间 buffer 当前可接收 |
| `stem_to_dw1_slice_out` | FIFO `out_data` | top 解包后送 buffer | FIFO 出口 payload |
| `stem_to_dw1_out_tile_t_base` | top 解包 | `u_stem_to_dw1` | tile 时间基址 |
| `stem_to_dw1_out_tile_oc_base` | top 解包 | `u_stem_to_dw1` | tile 通道基址 |
| `stem_to_dw1_out_tile` | top 解包 | `u_stem_to_dw1` | tile 数据 |

payload 解包：

```verilog
{
    stem_to_dw1_out_tile_t_base,
    stem_to_dw1_out_tile_oc_base,
    stem_to_dw1_out_tile
} = stem_to_dw1_slice_out;
```

真实 buffer 的 ready 来源：

```verilog
u_stem_to_dw1.stem_out_ready = stem_to_dw1_in_ready;
```

`stem_to_dw1_buffer` 内部 ready 条件是：

```text
wr_state == WR_IDLE
tile 参数合法
目标 ring slot 可安全覆盖
没有 clear
```

握手：

```verilog
stem_to_dw1_buffer_fire = stem_to_dw1_in_valid && stem_to_dw1_in_ready;
```

这个 fire 成立时：

1. FIFO 弹出一个 stem tile。
2. `stem_to_dw1_buffer` 锁存 `t_base/oc_base/tile`。
3. `stem_to_dw1_buffer` 后续进入写状态，把 4 个 row word 写进 BRAM。
4. 顶层同时执行：

```verilog
dw1_pending_push = busy && stem_to_dw1_in_valid && stem_to_dw1_in_ready;
```

也就是只有数据真正进入 `stem_to_dw1_buffer` 后，才往 DW1 pending FIFO 里写任务票。

## 5. stem_to_dw1_buffer 到 DW1 的窗口读取

这条链路不是数据 FIFO，而是 DW1 向 `stem_to_dw1_buffer` 请求 5-tap 输入窗口。

### 5.1 请求握手

`u_dw1` 是请求发送方，`u_stem_to_dw1` 是请求接收方。

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `dw1_win_req_valid` | `u_dw1` | `u_stem_to_dw1.dw1_win_req_valid` | DW1 请求一个输出点窗口 |
| `dw1_win_req_out_t` | `u_dw1` | `u_stem_to_dw1` | 请求的输出时间点 |
| `dw1_win_req_ch_base` | `u_dw1` | `u_stem_to_dw1` | 请求通道组 |
| `dw1_win_req_ready` | `u_stem_to_dw1` | `u_dw1.win_req_ready` | buffer 确认窗口可读 |

握手：

```verilog
dw1_win_req_fire = dw1_win_req_valid && dw1_win_req_ready;
```

`dw1_win_req_ready` 表示：

```text
stem_to_dw1_buffer 当前空闲，并且该 5-tap window 所需的非 padding sample 已经在 ring BRAM/tag 中可用
```

### 5.2 数据返回握手

`u_stem_to_dw1` 是窗口数据发送方，`u_dw1` 是接收方。

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `dw1_win_vec_valid` | `u_stem_to_dw1` | `u_dw1.win_vec_valid` | 5-tap window 数据已准备好 |
| `dw1_win_vec` | `u_stem_to_dw1` | `u_dw1.win_vec` | 5-tap x 4ch 数据 |
| `dw1_win_vec_ready` | `u_dw1` | `u_stem_to_dw1.dw1_win_vec_ready` | DW1 可以接收窗口数据 |

握手：

```verilog
dw1_win_vec_fire = dw1_win_vec_valid && dw1_win_vec_ready;
```

这个 fire 成立时，DW1 接收窗口数据并进入计算流水。

## 6. DW1 输出到 dw1_to_pw1_buffer

### 6.1 第一段：u_dw1 -> u_dw1_to_pw1_ready_slice

`u_dw1` 是发送方，`u_dw1_to_pw1_ready_slice` 的入口是接收方。

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `dw1_mid_out_valid` | `u_dw1` | FIFO `in_valid` | DW1 有一个 4ch row vector |
| `dw1_mid_out_ready` | FIFO `in_ready` | `u_dw1.out_ready` | FIFO 入口有空间 |
| `dw1_mid_out_t` | `u_dw1` | top 打包 | row 时间点 |
| `dw1_mid_out_ch_base` | `u_dw1` | top 打包 | 4ch 通道基址 |
| `dw1_mid_out_vec` | `u_dw1` | top 打包 | 4ch 数据 |

payload：

```verilog
dw1_to_pw1_slice_in = {
    dw1_mid_out_t,
    dw1_mid_out_ch_base,
    dw1_mid_out_vec
};
```

握手：

```verilog
dw1_to_pw1_slice_input_fire = dw1_mid_out_valid && dw1_mid_out_ready;
```

这里 `dw1_mid_out_ready` 只表示 FIFO 有空间。

### 6.2 第二段：u_dw1_to_pw1_ready_slice -> u_dw1_to_pw1

FIFO 是发送方，`u_dw1_to_pw1` 是真实级间 buffer 接收方。

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `dw1_to_pw1_in_valid` | FIFO `out_valid` | `u_dw1_to_pw1.dw1_out_valid` | FIFO 出口有 DW1 row |
| `dw1_to_pw1_in_ready` | `u_dw1_to_pw1.dw1_out_ready` | FIFO `out_ready` | 真实 buffer 可写 |
| `dw1_to_pw1_out_t` | top 解包 | `u_dw1_to_pw1` | row 时间点 |
| `dw1_to_pw1_out_ch_base` | top 解包 | `u_dw1_to_pw1` | 通道基址 |
| `dw1_to_pw1_out_vec` | top 解包 | `u_dw1_to_pw1` | 4ch 数据 |

握手：

```verilog
dw1_to_pw1_buffer_fire = dw1_to_pw1_in_valid && dw1_to_pw1_in_ready;
```

这个 fire 成立时，`dw1_to_pw1_buffer` 写入一个 DW1 输出 row/chunk，并更新 row/channel 完整性元数据。PW1 后续通过 tile request 判断 4 行全通道是否已经齐备。

## 7. PW1 从 dw1_to_pw1_buffer 取输入 tile

PW1 的输入读取分两层：

1. 先请求一个 4-row tile 是否已经齐备。
2. tile cache 加载完成后，PW1 按 `k` 请求每个输入通道的 4-row activation vector。

### 7.1 PW1 input tile request

`u_pw1` 是请求发送方，`u_dw1_to_pw1` 是请求接收方。

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `pw1_input_tile_req_valid` | `u_pw1` | `u_dw1_to_pw1.input_tile_req_valid` | PW1 请求一个 4-row 输入 tile |
| `pw1_input_tile_req_t_base` | `u_pw1` | `u_dw1_to_pw1.input_tile_req_t_base` | tile 时间基址 |
| `pw1_input_tile_req_ready` | `u_dw1_to_pw1` | `u_pw1.input_tile_req_ready` | buffer 确认该 tile 可加载 |

握手：

```verilog
pw1_tile_req_fire = pw1_input_tile_req_valid && pw1_input_tile_req_ready;
```

当前版本这里做了一拍 staging：

```text
第 N 拍：
  u_pw1 拉高 pw1_input_tile_req_valid，并给出 t_base。
  u_dw1_to_pw1 锁存请求参数，检查 row_full_valid/tag。

第 N+1 拍：
  u_dw1_to_pw1 用 tile_req_ready_q 输出 pw1_input_tile_req_ready。
  如果 valid 仍然为 1 且 ready 为 1，tile request fire。
```

因此 `pw1_input_tile_req_ready` 不再是同周期从 `row_full_valid/tag` 组合回 PW1 FSM 的 ready。

fire 成立后，`u_dw1_to_pw1` 进入 `RD_LOAD`，从 BRAM 加载 4 rows x all input channels 到内部 tile cache。

### 7.2 PW1 activation vector request/response

tile cache ready 后，PW1 按输入通道 `k` 请求 activation vector。

请求握手：

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `pw1_act_req_valid` | `u_pw1` | `u_dw1_to_pw1.act_req_valid` | PW1 请求某个 input channel 的 4-row vector |
| `pw1_act_req_t_base` | `u_pw1` | `u_dw1_to_pw1.act_req_t_base` | 应匹配已加载 tile |
| `pw1_act_req_k` | `u_pw1` | `u_dw1_to_pw1.act_req_k` | 输入通道索引 |
| `pw1_act_req_ready` | `u_dw1_to_pw1` | `u_pw1.act_req_ready` | buffer 当前可接受 act 请求 |

数据返回握手：

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `pw1_act_vec_valid` | `u_dw1_to_pw1` | `u_pw1.act_vec_valid` | activation vector 有效 |
| `pw1_act_vec` | `u_dw1_to_pw1` | `u_pw1.act_vec` | 4-row x 1-channel 数据 |
| `pw1_act_vec_ready` | `u_pw1` | `u_dw1_to_pw1.act_vec_ready` | PW1 可接收 |

fire：

```verilog
pw1_act_req_fire = pw1_act_req_valid && pw1_act_req_ready;
pw1_act_vec_fire = pw1_act_vec_valid && pw1_act_vec_ready;
```

## 8. PW1 输出到 pw1_to_dw2_buffer

### 8.1 第一段：u_pw1 -> u_pw1_to_dw2_ready_slice

`u_pw1` 是发送方，`u_pw1_to_dw2_ready_slice` 的入口是接收方。

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `pw1_out_valid` | `u_pw1` | FIFO `in_valid` | PW1 有一个 pooled output tile |
| `pw1_out_ready` | FIFO `in_ready` | `u_pw1.out_ready` | FIFO 入口有空间 |
| `pw1_out_pool_t_base` | `u_pw1` | top 打包 | low-resolution 时间基址 |
| `pw1_out_oc_base` | `u_pw1` | top 打包 | PW1 输出通道基址 |
| `pw1_out_tile` | `u_pw1` | top 打包 | pooled tile 数据 |

payload：

```verilog
pw1_to_dw2_slice_in = {
    pw1_out_pool_t_base[LOW_TIME_W-1:0],
    pw1_out_oc_base,
    pw1_out_tile
};
```

握手：

```verilog
pw1_to_dw2_slice_input_fire = pw1_out_valid && pw1_out_ready;
```

这里 `pw1_out_ready` 只表示 `u_pw1_to_dw2_ready_slice` 有空间，不再直接表示 `pw1_to_dw2_buffer` 的 ring slot 可写。

### 8.2 第二段：u_pw1_to_dw2_ready_slice -> u_pw1_to_dw2

FIFO 是发送方，`u_pw1_to_dw2` 是真实级间 buffer 接收方。这里还额外受到 `dw2_pending_room` 约束。

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `pw1_to_dw2_slice_valid` | FIFO `out_valid` | top 组合逻辑 | FIFO 出口有 PW1 tile |
| `pw1_to_dw2_out_ready` | top 组合逻辑 | FIFO `out_ready` | FIFO 出口是否可弹出 |
| `pw1_to_dw2_in_valid` | top 组合逻辑 | `u_pw1_to_dw2.pw1_out_valid` | 送给真实 buffer 的 valid |
| `pw1_to_dw2_in_ready` | `u_pw1_to_dw2.pw1_out_ready` | top 组合逻辑 | 真实 buffer 可写 |
| `pw1_to_dw2_pool_t_base` | top 解包 | `u_pw1_to_dw2` | low-resolution 时间基址 |
| `pw1_to_dw2_oc_base` | top 解包 | `u_pw1_to_dw2` | 通道基址 |
| `pw1_to_dw2_tile` | top 解包 | `u_pw1_to_dw2` | tile 数据 |

top 中的组合关系：

```verilog
pw1_to_dw2_in_valid  = pw1_to_dw2_slice_valid && dw2_pending_room;
pw1_to_dw2_out_ready = pw1_to_dw2_in_ready && dw2_pending_room;
```

真实握手条件：

```verilog
pw1_to_dw2_buffer_fire =
    pw1_to_dw2_slice_valid
    && pw1_to_dw2_in_ready
    && dw2_pending_room;
```

这个 fire 成立时：

1. FIFO 弹出一个 PW1 pooled tile。
2. `pw1_to_dw2_buffer` 接收并写入 BRAM。
3. 顶层同时执行：

```verilog
dw2_pending_push = pw1_to_dw2_in_valid && pw1_to_dw2_in_ready;
```

因此 DW2 pending 只在数据真正进入 `pw1_to_dw2_buffer` 后 push。

## 9. pw1_to_dw2_buffer 到 DW2 的窗口读取

与 DW1 读取 stem buffer 类似，DW2 请求 5-tap low-resolution 输入窗口。

## 10. DW2 输出到 dw2_to_pw2_buffer

### 10.1 第一段：u_dw2 -> u_dw2_to_pw2_ready_slice

### 10.2 第二段：u_dw2_to_pw2_ready_slice -> u_dw2_to_pw2

FIFO 是发送方，`u_dw2_to_pw2` 是真实级间 buffer 接收方。

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `dw2_to_pw2_in_valid` | FIFO `out_valid` | `u_dw2_to_pw2.dw2_out_valid` | FIFO 出口有 DW2 row |
| `dw2_to_pw2_in_ready` | `u_dw2_to_pw2.dw2_out_ready` | FIFO `out_ready` | 真实 buffer 可写 |
| `dw2_to_pw2_out_t` | top 解包 | `u_dw2_to_pw2` | row 时间点 |
| `dw2_to_pw2_out_ch_base` | top 解包 | `u_dw2_to_pw2` | 通道基址 |
| `dw2_to_pw2_out_vec` | top 解包 | `u_dw2_to_pw2` | 4ch 数据 |

握手：

```verilog
dw2_to_pw2_buffer_fire = dw2_to_pw2_in_valid && dw2_to_pw2_in_ready;
```

这个 fire 成立时，`dw2_to_pw2_buffer` 写入一个 DW2 输出 row/chunk，并更新 row/channel 完整性元数据。

## 11. PW2 从 dw2_to_pw2_buffer 取输入 tile

PW2 与 PW1 类似，但多了 final partial tile 的 `row_valid_mask`。

### 11.1 PW2 input tile request

`u_pw2` 是请求发送方，`u_dw2_to_pw2` 是请求接收方。

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `pw2_input_tile_req_valid` | `u_pw2` | `u_dw2_to_pw2.input_tile_req_valid` | PW2 请求一个输入 tile |
| `pw2_input_tile_req_t_base` | `u_pw2` | `u_dw2_to_pw2.input_tile_req_t_base` | tile 时间基址 |
| `pw2_input_tile_req_row_valid_mask` | `u_pw2` | `u_dw2_to_pw2.input_tile_req_row_valid_mask` | 该 tile 哪些 row 有效 |
| `pw2_input_tile_req_ready` | `u_dw2_to_pw2` | `u_pw2.input_tile_req_ready` | buffer 确认该 tile 可加载 |

握手：

```verilog
pw2_tile_req_fire = pw2_input_tile_req_valid && pw2_input_tile_req_ready;
```

这里也做了一拍 staging：

```text
第 N 拍：
  u_pw2 输出 valid/t_base/row_valid_mask。
  u_dw2_to_pw2 锁存 t_base 和 row_valid_mask，并检查 row_full_valid/tag。

第 N+1 拍：
  u_dw2_to_pw2 用 tile_req_ready_q 输出 ready。
```

fire 成立后，`u_dw2_to_pw2` 使用锁存后的：

```verilog
tile_req_t_base_q
tile_req_row_valid_mask_q
```

进入 `RD_LOAD`，保证输入端口后续变化不会影响已接受请求的地址和 mask。

### 11.2 PW2 activation vector request/response

请求握手：

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `pw2_act_req_valid` | `u_pw2` | `u_dw2_to_pw2.act_req_valid` | PW2 请求某个 input channel 的 rows vector |
| `pw2_act_req_t_base` | `u_pw2` | `u_dw2_to_pw2.act_req_t_base` | 应匹配已加载 tile |
| `pw2_act_req_k` | `u_pw2` | `u_dw2_to_pw2.act_req_k` | 输入通道索引 |
| `pw2_act_req_row_valid_mask` | `u_pw2` | `u_dw2_to_pw2.act_req_row_valid_mask` | 有效行 mask |
| `pw2_act_req_ready` | `u_dw2_to_pw2` | `u_pw2.act_req_ready` | buffer 当前可接受 act 请求 |

数据返回握手：

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `pw2_act_vec_valid` | `u_dw2_to_pw2` | `u_pw2.act_vec_valid` | activation vector 有效 |
| `pw2_act_vec` | `u_dw2_to_pw2` | `u_pw2.act_vec` | rows x 1-channel 数据 |
| `pw2_act_vec_ready` | `u_pw2` | `u_dw2_to_pw2.act_vec_ready` | PW2 可接收 |

fire：

```verilog
pw2_act_req_fire = pw2_act_req_valid && pw2_act_req_ready;
pw2_act_vec_fire = pw2_act_vec_valid && pw2_act_vec_ready;
```

## 12. PW2 到 FC

PW2 到 FC 不是逐 tile ready/valid 数据流。PW2 内部完成 PW2 + GAP 累加后输出：

| 信号 | 输出方 | 输入方 | 含义 |
|---|---|---|---|
| `pw2_gap_frame_done_wire` | `u_pw2` | top | GAP 一帧完成 |
| `gap_count` | `u_pw2` | top / `u_fc` | GAP 累加 row 数 |
| `gap_sum_vec` | `u_pw2` | top / `u_fc` | GAP 累加结果 |

顶层等待 `front_complete_now`，确认前级全部 drain：

```text
PW2 GAP 已完成
所有 start 已发完
所有计算模块 idle
所有 stream FIFO empty
所有级间 buffer wr/rd idle
pending FIFO empty
```

然后顶层拉高一拍：

```verilog
fc_start
```

`u_fc` 输出最终分类结果时使用 ready/valid：

| 信号 | 输出方 | 输入方 | 方向说明 |
|---|---|---|---|
| `fc_result_valid` | `u_fc` | 外部/top output | FC 结果有效 |
| `fc_logits_vec` | `u_fc` | 外部/top output | logits |
| `fc_class_idx` | `u_fc` | 外部/top output | 分类 |
| `fc_result_ready` | 外部 | `u_fc.result_ready` | 外部愿意接收结果 |

最终结果握手：

```verilog
fc_result_fire = fc_result_valid && fc_result_ready;
```

`u_fc` 在 `STATE_OUT` 中保持 `result_valid`，直到 `fc_result_ready` 为 1；fire 后 `fc_done_wire` 拉高，顶层置 `done`。

## 13. pending FIFO 与 stream FIFO 的区别

### 13.1 DW1 pending FIFO

保存内容：

```verilog
dw1_pending_t_base
dw1_pending_ch_base
dw1_pending_final_chunk
```

push 条件：

```verilog
dw1_pending_push =
    busy
    && stem_to_dw1_in_valid
    && stem_to_dw1_in_ready;
```

也就是 stem tile 真正进入 `stem_to_dw1_buffer` 后才 push。

pop 条件：

```verilog
dw1_pending_pop = dw1_start_now;
```

pop 时顶层给 DW1 发启动参数：

```verilog
dw1_start <= 1'b1;
dw1_ch_base <= dw1_pending_ch_base[rd_ptr];
dw1_input_t_base <= dw1_pending_t_base[rd_ptr];
dw1_final_chunk <= dw1_pending_final_chunk[rd_ptr];
```

### 13.2 DW2 pending FIFO

保存内容：

```verilog
dw2_pending_t_base
dw2_pending_ch_base
dw2_pending_final_chunk
```

push 条件：

```verilog
dw2_pending_push = pw1_to_dw2_in_valid && pw1_to_dw2_in_ready;
```

结合 top 组合逻辑，等价于：

```text
PW1 tile 已经从 stream FIFO 出口真正进入 pw1_to_dw2_buffer
并且 DW2 pending FIFO 有空间
```

pop 条件：

```verilog
dw2_pending_pop = dw2_start_now;
```

pop 时顶层给 DW2 发启动参数。

### 13.3 两类 FIFO 的本质区别

| 类型 | 存什么 | 作用 |
|---|---|---|
| `emg_stream_fifo_slice` | 真实数据 payload | 隔离 ready 长组合路径，提供小深度弹性缓冲 |
| `dw1_pending/dw2_pending` | t_base/ch_base/final_chunk 任务票 | 告诉调度器下一层什么时候可以启动 |

关键约束：

```text
pending push 必须晚于或等于真实数据进入对应级间 buffer。
```

否则会出现下一层已经启动，但数据还停在 stream FIFO 里，导致读空或读旧数据。

## 14. 当前主要 ready 路径隔离结果

### 14.1 Stem 写侧

修改前：

```text
stem_to_dw1_buffer.slot/tag/safe
  -> stem_out_ready
  -> u_stem
```

修改后：

```text
u_stem.out_ready
  <- u_stem_to_dw1_ready_slice.in_ready
```

真实 buffer 的 slot/tag/safe 只影响：

```text
u_stem_to_dw1_ready_slice.out_ready
```

只有 slice 被填满时，后级反压才会重新传到 stem。

### 14.2 DW1/PW1/DW2/PW2 输出写侧

DW1、PW1、DW2 的输出也同样先进入 stream FIFO：

```text
u_dw1.out_ready <- u_dw1_to_pw1_ready_slice.in_ready
u_pw1.out_ready <- u_pw1_to_dw2_ready_slice.in_ready
u_dw2.out_ready <- u_dw2_to_pw2_ready_slice.in_ready
```

真实级间 buffer 的 ring overwrite 判断只影响 FIFO 出口 ready。

### 14.3 PW tile request ready

PW1/PW2 的 input tile request ready 不再同周期组合返回：

```text
row_full_valid/tag
  -> tile_req_ready_q
  -> 下一拍 input_tile_req_ready
```

这样切断：

```text
row_full_valid/tag -> input_tile_req_ready -> PW FSM
```

这条长组合路径。

## 15. 总体数据流图

```text
top scheduler
  -> stem_start

u_stem
  -> valid/data -> u_stem_to_dw1_ready_slice
  -> valid/data -> u_stem_to_dw1_buffer
  -> push dw1_pending

dw1_pending
  -> dw1_start

u_dw1
  -> win_req valid/data -> u_stem_to_dw1_buffer
  <- win_vec valid/data <- u_stem_to_dw1_buffer
  -> valid/data -> u_dw1_to_pw1_ready_slice
  -> valid/data -> u_dw1_to_pw1_buffer

u_pw1
  -> input_tile_req valid/data -> u_dw1_to_pw1_buffer
  <- input_tile_req_ready <- u_dw1_to_pw1_buffer
  -> act_req valid/data -> u_dw1_to_pw1_buffer
  <- act_vec valid/data <- u_dw1_to_pw1_buffer
  -> valid/data -> u_pw1_to_dw2_ready_slice
  -> valid/data -> u_pw1_to_dw2_buffer
  -> push dw2_pending

dw2_pending
  -> dw2_start

u_dw2
  -> win_req valid/data -> u_pw1_to_dw2_buffer
  <- win_vec valid/data <- u_pw1_to_dw2_buffer
  -> valid/data -> u_dw2_to_pw2_ready_slice
  -> valid/data -> u_dw2_to_pw2_buffer

u_pw2
  -> input_tile_req valid/data/mask -> u_dw2_to_pw2_buffer
  <- input_tile_req_ready <- u_dw2_to_pw2_buffer
  -> act_req valid/data/mask -> u_dw2_to_pw2_buffer
  <- act_vec valid/data <- u_dw2_to_pw2_buffer
  -> gap_frame_done/gap_sum_vec

top scheduler
  -> fc_start

u_fc
  -> fc_result_valid/logits/class
  <- fc_result_ready
```

## 16. 读这套握手时最容易混淆的信号

### 16.1 `pw1_out_ready` 和 `pw1_to_dw2_in_ready`

`pw1_out_ready`：

```text
输出方：u_pw1_to_dw2_ready_slice.in_ready
输入方：u_pw1.out_ready
含义：FIFO 入口有空间
```

`pw1_to_dw2_in_ready`：

```text
输出方：u_pw1_to_dw2.pw1_out_ready
输入方：top 组合逻辑 / FIFO 出口 ready 生成
含义：真实 pw1_to_dw2_buffer 当前可写
```

它们不是同一个 ready。中间隔着 `u_pw1_to_dw2_ready_slice`。

### 16.2 `stem_to_buffer_ready` 和 `stem_to_dw1_in_ready`

`stem_to_buffer_ready`：

```text
输出方：u_stem_to_dw1_ready_slice.in_ready
输入方：u_stem.out_ready
含义：stem 输出 FIFO 有空间
```

`stem_to_dw1_in_ready`：

```text
输出方：u_stem_to_dw1.stem_out_ready
输入方：u_stem_to_dw1_ready_slice.out_ready
含义：真实 stem_to_dw1_buffer 当前可写
```

### 16.3 `input_tile_req_ready` 和 `act_req_ready`

`input_tile_req_ready`：

```text
确认整个 tile 可以加载到 buffer 内部 tile cache。
当前版本是经过 tile_req_ready_q staging 后返回。
```

`act_req_ready`：

```text
确认 tile cache 已经准备好，并且当前这个 k 的 activation vector 请求可以被接受。
```

二者不是同一层含义。前者是 tile 级请求，后者是 tile cache 内部按输入通道读取的请求。
