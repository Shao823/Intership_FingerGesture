## 2. 当前 RTL 模块状态

### 2.1 stem_conv_array_4x4

文件：

```text
emg_gesture_vivado/src/stem_conv_array_4x4.v
```

当前功能：

```text
1. 经典 4x4 output-stationary systolic array。
2. 一个 start 计算 4 个时间点 x 4 个输出通道。
3. Conv + BN folded scale/bias + ReLU 合并在该计算资源内完成。
4. PE 内部 accumulator 固定输出，activation 横向流动，weight 纵向流动。
5. BN 后处理拆成 POST0/POST1 两拍，减轻单周期组合路径。
6. PE 乘法器通过 mux 在 MAC phase 和 BN scale phase 之间复用，避免 BN 被 RTL 明确写成另一套并行乘法器。
```

接口约定：

```text
weight:
  weight_req_valid / weight_req_ready
  weight_req_oc_base
  weight_req_k
  weight_vec_valid / weight_vec_ready
  weight_vec = {W[oc_base+3][k], ..., W[oc_base+0][k]}

activation:
  act_req_valid / act_req_ready
  act_req_t_base
  act_req_k
  act_vec_valid / act_vec_ready
  act_vec = {A[tile_t_base+3][k], ..., A[tile_t_base+0][k]}

output:
  out_valid / out_ready
  out_tile_t_base
  out_tile_oc_base
  out_tile[(row*OC_LANES + col)*DATA_W +: DATA_W]
```

stem 输出写入 stem/dw1 buffer 的地址约定：

```text
buffer[out_tile_t_base + row][out_tile_oc_base + col]
    <= out_tile[(row*OC_LANES + col)*DATA_W +: DATA_W]
```

这个 metadata 是后续 DW1 正确读取窗口的基础。

### 2.2 dw_conv1_4ch_5tap

文件：

```text
emg_gesture_vivado/src/dw_conv1_4ch_5tap.v
```

当前功能：

```text
1. DW1 depthwise Conv1d, 32 channels, k=5, pad=2。
2. 每次 start 处理一个 4-channel group。
3. 每个输出时间点一次性完成 4 个通道的 5-tap depthwise MAC。
4. 计算资源为 4 channels x 5 taps = 20 个乘法器。
5. DW1 权重很小，当前直接本地驻留在 weight_mem[32][5]。
6. 输入不是直接接大数组，而是对接未来 stem_to_dw1 window loader。
```

DW1 输入窗口接口：

```text
win_req_valid / win_req_ready
win_req_out_t
win_req_ch_base
win_vec_valid / win_vec_ready
win_vec[(tap*P_CH + lane)] = X[out_t + tap - PAD][ch_base + lane]
```

未来 window loader 的责任：

```text
1. 从 stem_to_dw1 buffer 中读取对应时间点和通道。
2. 对开头和结尾 padding 位置返回 0。
3. 只有 requested window 已经可用时才返回 win_vec_valid。
```

DW1 输出接口：

```text
out_valid / out_ready
out_t
out_ch_base
out_vec[lane] = Y[out_t][out_ch_base + lane]
```

DW1 输出写入 dw1/pw1 buffer 的地址约定：

```text
buffer[out_t][out_ch_base + lane] <= out_vec[lane]
```

dw1/pw1 buffer 需要维护每个 out_t 的 channel-group valid bit。
PW1 可以在 4 个连续 out_t 行的全部 channel groups 都 valid 后启动。

## 3. 当前调度约定

### 3.1 stem 到 DW1

第一版保守调度：

```text
for stem_t_base = 0, 4, 8, ...:
    for oc_base = 0, 4, 8, ..., 28:
        stem 计算 4 time rows x 4 out channels
        写入 stem_to_dw1 buffer

    当该 stem 时间块的 32 通道都写入完成后：
        for ch_base = 0, 4, 8, ..., 28:
            启动 DW1 处理该 stem 时间块对应的可计算输出时间范围
```

这样做的好处：

```text
1. stem/dw1 buffer 的 ready 条件简单。
2. DW1 总是能看到一个完整 4-time x 32-channel stem chunk。
3. DW1 输出带 out_t/out_ch_base，可由 dw1/pw1 buffer 统一聚合。
4. PW1 只需要等待 4 个连续时间点的全通道 valid，不需要关心 DW1 的内部输出顺序。
```

### 3.2 DW1 输出时间范围

对于一个新写入的 stem chunk：

```text
input_t_base = 0, input_rows = 4, final_chunk = 0:
    DW1 可输出 out_t = 0..1，共 2 行

middle chunk, input_t_base = B:
    DW1 可输出 out_t = B-2..B+1，共 4 行

final chunk, input_t_base = 344, input_rows = 4, final_chunk = 1:
    DW1 可输出 out_t = 342..347，共 6 行
```

注意：

```text
DW1 内部一个 ch_base job 是 out_t 递增输出。
系统层面建议固定一个 stem 时间块，扫完 ch_base=0,4,...,28 后再进入下一个 stem 时间块。
```

## 5. 相较基线已经做出的实现取舍

### 5.1 stem weight 和 activation 都改为流式接口

早期讨论过本地保存 `35 x 32` stem weight。
当前实现改为：

```text
stem compute tile 不保存完整 stem conv weight。
weight 由外部 ROM/controller 按 k 返回 4-lane vector。
activation 由外部 ping-pong buffer/window loader 按 k 返回 4-row vector。
```

理由：

```text
1. 计算模块边界更清晰。
2. 后续 weight ROM、activation buffer 可以独立设计和验证。
3. stem 模块只负责计算，不绑定具体存储实现。
```

### 5.2 BN/ReLU 保留在 stem PE 内部

当前 stem PE 后处理：

```text
POST0: conv_q8_8 * w_fold
POST1: + bias_fold, saturate, ReLU
```

理由：

```text
1. stem 输出到 dw1 buffer 的数据已经是 conv + BN + ReLU 完整结果。
2. 避免后级 buffer 中混入未激活或未归一化的数据。
3. 使用 mux 让 PE multiplier 参与 BN scale multiply，减少综合出额外乘法资源的风险。
```

### 5.3 DW1 采用 window-loader 模式

当前 DW1 不直接管理 stem_to_dw1 buffer，也不内部维护复杂滑窗 RAM。

理由：

```text
1. DW1 计算单元保持纯粹。
2. padding、valid 判断、跨 chunk 数据可用性由 window loader 统一处理。
3. 后续可以独立优化 buffer/window loader，不影响 DW1 MAC datapath。
```

## 7. 后续可改进点

### 7.1 更激进的跨层早启动

当前保守策略是：

```text
stem 先完成 4 时间点 x 32 通道，再启动对应 DW1 jobs。
```

可改进为：

```text
stem 每完成某些通道组后，DW1 对已满足 5-tap window 的局部数据提前启动。
```

风险：

```text
1. valid 条件明显复杂。
2. stem_to_dw1 buffer 需要更细粒度的 per-row/per-channel valid bitmap。
3. top scheduler 更难验证。
```

第一版暂不采用。

### 7.2 DW1 输出顺序优化

当前 DW1 job 内部是：

```text
固定 ch_base，out_t 递增输出。
```

系统层面是：

```text
固定 stem 时间块，扫 ch_base=0,4,...,28。
```

如果后续 PW1 row packer 需要更快拿齐单个 out_t 的全通道，也可以考虑把 DW1 scheduler 改成：

```text
for out_t in computable_rows:
    for ch_base in 0,4,...,28:
        DW1 计算该 out_t 的 4 通道
```

代价是 DW1 当前模块接口需要从 chunk-job 改成 single-row job 或支持外部指定 out_t。

### 7.3 DW full-tap 与 tap-serial 版本

当前采用：

```text
P_CH=4, P_TAP=5
20 multipliers per DW layer
```

由于 DSP 资源不紧张，当前选择合理。

若后续目标器件变化或希望降低功耗，可考虑：

```text
P_TAP=1 tap-serial
P_TAP=2 partial-tap parallel
DW weights 常系数 shift-add / LUT multiply
```

### 7.4 更完整的系统级 TB

当前 TB 只验证单个计算模块。
后续需要新增 top-level 或 near-top-level TB：

```text
stem -> stem_to_dw1 buffer/window loader -> DW1 -> dw1_to_pw1 buffer
```

重点验证：

```text
1. 348 个时间点全量遍历。
2. 首尾 padding。
3. 每个 out_t 的 32 通道是否全部写齐。
4. PW1 启动条件是否严格等于 4 consecutive rows all-valid。
5. backpressure 是否不会造成数据覆盖或读写竞争。
```

## 8. DW1 review 后补强记录

根据 RTL/TB review，今天对 `dw_conv1_4ch_5tap` 做了补强。

已采纳修改：

```text
1. 输出侧加入 2-entry valid/ready FIFO。
   目的：短时间下游 out_ready=0 时，DW1 可以继续接收 window response，直到 FIFO 填满。

2. weight_we 加运行期保护。
   现在只有 state=IDLE 且 start=0 时才接受权重写入。
   busy/RUN 期间误拉 weight_we 会被忽略，避免单个 job 混用新旧权重。

3. req_count/resp_count 改为 TIME_W+1 位。
   与 job_out_count 同宽，避免隐式位宽扩展 warning。

4. 补充 window request/response 契约注释。
   当前 DW1 仍保持 one outstanding request at most。
   上游 window loader 必须对每个 accepted request 恰好返回一次 response。

5. 补充 final_chunk 契约注释。
   final_chunk 只应在最后一块、右侧 padding 边界已经可计算时拉高。

6. 明确 5-tap lane 当前仍是组合 datapath。
   100 MHz baseline 可接受；若后续目标提高到 200 MHz，再考虑 lane 内部流水化。
```

扩展后的 DW1 TB 覆盖：

```text
1. 首 chunk padding：base=0 -> 2 个输出点。
2. 中间 chunk：稳定 4 个输出点。
3. final chunk：base=344 -> 6 个输出点。
4. 同一 stem 时间块 sweep 全部 8 个 channel groups。
5. output backpressure。
6. long output backpressure，覆盖 output FIFO 填充/恢复。
7. input_rows=0 空任务，应无请求无输出但 done 正常脉冲。
8. 0-cycle window loader，覆盖 req 和 resp 同拍 fire。
9. back-to-back jobs。
10. RUN 期间误拉 start，应被忽略。
11. RUN 期间误写 weight，应被忽略。
12. RUN 期间 reset，abort 后模块可复用。
```

验证结果：

```text

PASS stem_conv_array_4x4 self-check
```

本次未做的修改：

```text
1. 未把 window request 改成多 outstanding。
   原因：当前 DW1 计算仍明显快于 stem，one-outstanding 协议更简单，适合第一版组合。
   注意：若 window loader 是 1-cycle 同步 BRAM，one-outstanding 会让 DW1 约每 2 拍得到 1 个输出。

2. 未给 dw_conv1_lane_5tap 加流水。

```

## 10. PW1 4x8 模块实现记录

新增文件：

```text
emg_gesture_vivado/src/pw_conv1_array_4x8.v
emg_gesture_vivado/src/tb_pw_conv1_array_4x8.v
```

### 10.1 模块职责

`pw_conv1_array_4x8` 实现 LightCNN1D 的 PW1 pointwise convolution：

```text
输入：DW1 输出，32 channels，high-resolution time length = 348
PW1：1x1 Conv, 32 -> 64
后处理：BN folded scale/bias + ReLU
再后处理：MaxPool1d(kernel=2,stride=2)
输出：64 channels，low-resolution time length = 174
```

一个 `start` job 对应：

```text
4 high-resolution input rows x 8 output channels
```

模块内部输出并不是 4 行 PW1 raw output，而是已经完成：

```text
PW1 Conv + BN + ReLU + MaxPool
```

后的：

```text
2 low-resolution output rows x 8 output channels
```

### 10.2 关键接口和 metadata

启动侧：

```text
start
tile_t_base
oc_base
input_tile_req_valid
input_tile_req_t_base
input_tile_req_ready
```

其中 `input_tile_req_*` 表示 PW1 主动请求一个精确的 4-row 输入 tile：

```text
input_tile_req_valid = PW1 正在请求输入 tile
input_tile_req_t_base = 被请求的 high-resolution 起始时间点
input_tile_req_ready = row packer 已经凑齐 input_tile_req_t_base..+3
                       这 4 个 high-resolution rows 的全部 32 通道，
                       并且能在本 job 期间保持可读。
```

运行期 activation request：

```text
act_req_valid / act_req_ready
act_req_t_base = tile_t_base
act_req_k      = input channel index, 0..31
act_vec        = {A[tile_t_base+3][k], ..., A[tile_t_base+0][k]}
```

运行期 weight request：

```text
weight_req_valid / weight_req_ready
weight_req_oc_base = oc_base
weight_req_k       = input channel index, 0..31
weight_vec         = {W[oc_base+7][k], ..., W[oc_base+0][k]}
```

输出侧：

```text
out_valid / out_ready
out_pool_t_base = tile_t_base / 2
out_oc_base     = oc_base
out_tile[(pool_row*8 + col)*DATA_W +: DATA_W]
```

写入 pool_to_dw2 buffer 的地址约定：

```text
buffer[out_pool_t_base + pool_row][out_oc_base + col]
    <= out_tile[(pool_row*8 + col)*DATA_W +: DATA_W]
```

### 10.3 计算结构

核心阵列：

```text
4x8 output-stationary systolic array
ROWS     = 4
OC_LANES = 8
PW1_IC   = 32
PW1_OC   = 64
```

数据流：

```text
activation 横向流动
weight 纵向流动
每个 PE 固定一个 output accumulator
```

周期结构：

```text
32 个 MAC feed cycles
ROWS + OC_LANES - 2 = 10 个 drain cycles
POST0: conv_q * folded_bn_scale
POST1: + folded_bn_bias, saturate, ReLU
POOL:  MaxPool row0/1, row2/3，打一拍寄存
DONE:  out_valid 保持到 out_ready
```

MaxPool 规则：

```text
pool_row 0 = max(relu_row0, relu_row1)
pool_row 1 = max(relu_row2, relu_row3)
```

### 10.4 与 dw1_to_pw1 buffer 的关系

当前 PW1 模块不自己判断 `dw1_to_pw1 buffer` 中哪些 row/channel 已写齐。

这个判断仍属于后续 `dw1_to_pw1 row packer`：

```text
1. 接收 DW1 输出 out_t/out_ch_base/out_vec。
2. 对每个 out_t 维护 8 个 channel-group valid bit。
3. 当 PW1 拉高 input_tile_req_valid 时，读取 input_tile_req_t_base。
4. 只有该精确 tile 的 4 个连续 out_t rows 全部 channel groups 都 valid：
       input_tile_req_ready = 1
       允许 PW1 对应 tile_t_base 启动。
5. PW1 busy 期间，row packer 必须保持这 4 行数据可读，不应提前覆盖。
```

### 10.5 TB 覆盖

`tb_pw_conv1_array_4x8` 当前覆盖：

```text
1. 基本 1-cycle activation/weight response。
2. start 早于 input_tile_req_ready，模块等待完整 4-row tile 后才请求数据。
3. weight/activation request 独立反压。
4. 0-cycle response，覆盖 req 和 response 同拍 fire。
5. short output backpressure，验证 out_valid 保持。
6. long output backpressure，out_ready 延迟 35+ 拍，验证 done 不提前且状态机不死锁。
7. 同一 input time tile sweep 全部 8 个 output-channel groups。
8. 最后一个 aligned high-resolution tile: tile_t_base=344 -> out_pool_t_base=172。
9. row packer 先给出另一个 tile ready 时，PW1 必须等待当前 requested tile。
10. back-to-back jobs。
11. RUN/MAC 期间误拉 start，应被忽略。
12. RUN/MAC 期间误写 BN 参数，应被忽略。
13. 运行中 reset，abort 后重新 load BN 并复用模块。
14. 自检 reference 覆盖 Conv + BN + ReLU + MaxPool 的完整数值路径。
```

### 10.6 后续注意点

当前 PW1 和 stem 一样，request 端可以比 response 端更快。
这要求 activation loader / weight ROM controller 满足：

```text
1. 对每个 accepted request 恰好返回一次 response。
2. response 按 request 顺序返回。
3. 不乱序、不丢失、不重复。
```

如果后续 weight ROM 或 activation buffer 是共享资源并可能长时间反压，建议在 controller 侧加请求 FIFO，或者在 PW1 内部增加更明确的 outstanding 计数和 response metadata。

当前 PW1 PE 的后处理为：

```text
POST0: conv_q * folded_bn_scale
POST1: + folded_bn_bias + saturate + ReLU
POOL:  compare + register
```

如果 Vivado 实现后 100 MHz slack 不足，优先考虑：

```text
1. POST1 和 POOL 再拆拍。
2. 或者给 4x8 systolic PE 的 post path 加 pipeline。
```

### 10.7 PW1 启动条件澄清

`pw_conv1_array_4x8` 内部不直接扫描 `dw1_to_pw1 buffer` 的 valid bitmap。
它主动向 row packer 请求一个精确 tile：

```text
input_tile_req_valid
input_tile_req_t_base
input_tile_req_ready
```

row packer 必须针对 `input_tile_req_t_base` 精确生成 ready：

```text
row_full[t] = &valid_group[t][7:0]

input_tile_req_ready =
    row_full[input_tile_req_t_base]
 && row_full[input_tile_req_t_base + 1]
 && row_full[input_tile_req_t_base + 2]
 && row_full[input_tile_req_t_base + 3]
```

第一次 DW1 因为 padding 只能产生 `out_t=0,1` 两个高分辨率时间点。
此时：

```text
row_full[0] = 1
row_full[1] = 1
row_full[2] = 0
row_full[3] = 0
input_tile_req_ready(input_tile_req_t_base=0) = 0
```

所以 PW1 不能启动。等下一个 stem/DW1 chunk 产生 `out_t=2..5` 后，
`row_full[0..3]` 全部为 1，PW1 才能启动 `tile_t_base=0` 的 4-row job。

如果 row packer 中同时有多个候选 tile ready，不能输出泛泛的 `any_tile_ready`。
必须比较：

```text
ready_tile_t_base == input_tile_req_t_base
```

TB 已覆盖该风险：当 row packer 先声明另一个 tile ready 时，PW1 不允许发
activation/weight 请求；只有当前请求的 tile ready 后才能继续。

PW1 输出 maxpool 后的 metadata：

```text
out_pool_t_base = tile_t_base / 2
out_oc_base     = oc_base
```

写入 `pool_to_dw2 buffer` 的地址约定：

```text
buffer[out_pool_t_base + pool_row][out_oc_base + col]
    <= out_tile[(pool_row*8 + col)*DATA_W +: DATA_W]
```

后续 `pool_to_dw2` buffer 同样需要维护每个 low-resolution time row 的
channel-group valid bit，供 DW2 window loader 判断 k=5 窗口是否可读。
