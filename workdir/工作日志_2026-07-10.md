## 2. DW2->PW2 Buffer 设计结论

### 2.1 存储粒度

DW2 输出给 PW2 的级间数据仍按如下 word 组织：

```text
word = 1 个时间点 x 4 个通道
word width = 4 * 16 = 64 bit
```

PW2 一次计算需要：

```text
4 个 low-res 时间点 x 64 个输入通道
```

因此 buffer 对 PW2 的逻辑输出仍是：

```text
act_vec = 4 rows x 1 channel index
```

### 2.2 Ring rows 和 BRAM depth

DW2 输出节奏分析：

```text
第一次理论不产出
中间流水每次产生 2 个时间点
最后一次产生 4 个时间点
PW2 一次读取 4 个时间点全通道
```

最低需要覆盖的时间点窗口：

```text
2 + 2 + 4 = 8 rows
```

按两倍冗余：

```text
RING_ROWS = 16
```

因为一个 bank 是 4 通道，PW2 输入通道为 64：

```text
BANKS = 64 / 4 = 16
DEPTH = RING_ROWS * BANKS = 16 * 16 = 256
WIDTH = 64 bit
```

所以 DW2->PW2 buffer 对应的 BRAM IP 应为 true dual RAM：

```text
blk_mem_gen_dw_to_pw_conv2
  width = 64 bit
  depth = 256
  address width = 8
  read latency = 1
```

## 4. 五级流水线顶层

新增/继续完善的顶层目录：

```text
emg_gesture_vivado/src/top/stem_dw1_pw1_dw2_pw2_pipeline/
  stem_dw1_pw1_dw2_pw2_pipeline_top.v
  tb_stem_dw1_pw1_dw2_pw2_pipeline_top.v
  gen_testdata.py
  testdata/
```

流水线结构：

```text
stem_conv_array_4x4
  -> stem_to_dw1_buffer
  -> dw_conv1_4ch_5tap
  -> dw1_to_pw1_buffer
  -> pw_conv1_array_4x8
  -> pw1_to_dw2_buffer
  -> dw_conv2_4ch_5tap
  -> dw2_to_pw2_buffer
  -> pw_conv2_array_4x12
  -> GAP
```

关键调度策略：

```text
1. Stem jobs 按 tile_t_base 和 ch_base 扫描。
2. Stem done 后将 DW1 job 推入 dw1_pending FIFO。
3. PW1 按 tile_t_base=0,4,...,344 单调启动。
4. PW1 输出 tile 被写入 PW1->DW2 buffer，同时推入 DW2 pending FIFO。
5. DW2 按 PW1 输出 tile 产生的 pending job 启动。
6. PW2 按 low-res tile_t_base=0,4,...,172 单调启动。
7. PW2 不输出 feature tile，而是直接累加 96 通道 GAP。
```

五级流水线 mock 典型结果：

```text
CASE 0 cycles=37013 gap_count=174 stem=696 dw1_rows=2784 pw1_out=696 dw2_rows=2784 pw2_tiles=44 max_dw1_pending=1 max_dw2_pending=4 waits p1=8251 p2=10230 stalls d1=946 d2=28 errors_delta=0
```

字段含义：

```text
cycles:
  从顶层 start 后到顶层 done 的内部工作周期。

gap_count:
  PW2 累加到 GAP 的 low-res 时间点数量，应为 174。

stem:
  stem 输出 tile 数量，87 个 time tile * 8 个 channel group = 696。

dw1_rows:
  DW1 输出 row 数量，348 个时间点 * 8 个 channel group = 2784。

pw1_out:
  PW1 pooled 输出 tile 数量，87 个 high-res tile * 8 个 output-channel group = 696。

dw2_rows:
  DW2 输出 row 数量，174 个 low-res 时间点 * 16 个 channel group = 2784。

pw2_tiles:
  PW2 启动并完成的 low-res tile 数量，ceil(174 / 4) = 44。

max_dw1_pending:
  Stem done 但 DW1 尚未启动的 pending job 峰值。

max_dw2_pending:
  PW1 输出后等待 DW2 启动的 pending job 峰值。

waits p1/p2:
  PW1/PW2 发出 input_tile_req 但对应 buffer 还不能提供完整 tile 的累计等待周期。

stalls d1/d2:
  DW1/DW2 输出 valid 但级间 buffer 暂时不能 ready 的累计反压周期。
```

关于 stalls：

```text
stalls d1=946
stalls d2=28
```

这是真实 valid/ready 局部 backpressure，不是数据丢失。原因主要是 PW1/PW2 搬运完整 4 时间点全通道 tile 时，buffer 读侧会占用控制状态，写侧在局部时刻不能马上接受 DW 输出。

当前设计中：

```text
1. DW1/DW2 输出端采用 valid/ready。
2. buffer 不 ready 时 DW1/DW2 保持 out_valid 和 out_vec。
3. ready 恢复后再完成握手。
```

因此这些 stall 表示吞吐节奏差异，不表示功能错误。

## 6. FC 接入 Full Top

今日将 FC 接入：

```text
stem_dw1_pw1_dw2_pw2_pipeline_top.v
```

新增顶层端口：

```text
input  wire fc_result_ready
output wire fc_busy
output wire fc_result_valid
output wire [2:0] fc_class_idx
output wire fc_input_count_error
output wire signed [5*16-1:0] fc_logits_vec

output reg [31:0] fc_jobs_started
output reg [31:0] fc_jobs_done
```

控制策略：

```text
1. 前五级流水线按原有逻辑启动和 drain。
2. PW2 完成所有 44 个 tile，并产生 gap_frame_done。
3. 顶层判断 front_complete_now。
4. 顶层打一拍 fc_start。
5. FC 计算期间 busy 保持为 1，前五级不启动新一轮。
6. FC done 后顶层 done 拉高一拍，busy 清零。
7. 下一轮 5*348 输入必须等顶层 done 后重新写入/启动。
```

当前没有单独设计 PW2->FC 级间 BRAM，因为 FC 输入只需要：

```text
gap_count
gap_sum_vec[96]
```

这两个信号在 PW2 完成后已经稳定，直接由顶层交给 FC 即可。

## 10. 综合结果与 100MHz 时序优化方向

今日完成 full-top 首次综合：

```text
Top: stem_dw1_pw1_dw2_pw2_pipeline_top
Device: xc7k70tfbv676-1
综合结果: synth_design completed successfully
综合耗时: elapsed ~= 52 min
```

资源摘要：

```text
Slice LUTs:      111409 / 41000 = 271.73%
Slice Registers: 73970  / 82000 = 90.21%
DSP48E1:         240    / 240   = 100.00%
Bonded IOB:      5885   / 300   = 1961.67%
```

BRAM 在该综合报告中显示为 0，是因为 `blk_mem_gen_*` 仍作为 black box/OOC IP 统计，不代表工程没有使用 BRAM。

### 10.1 资源结论

当前 xc7k70t 器件对完整 full top 明显不够：

```text
1. LUT 已超过当前器件 2.7 倍。
2. DSP 已打满 240/240。
3. IO 端口数量远超当前器件封装，主要来自裸露 debug/status/宽向量端口。
```

资源不足会显著恶化后续实现阶段的布局布线和时序收敛。因此当前小板子的 timing 结果不能作为最终硬件平台的绝对结论，但它能暴露 RTL 级关键路径和实现风格问题。

后续会换更大 FPGA，因此资源方向上的首要处理是：

```text
1. 去掉不必要的顶层输入配置口。
2. 去掉调试输出端口，尤其是宽向量、内部计数器、buffer 状态信号。
3. 后续通过 AXI/status register 暴露少量必要状态，而不是把内部信号直接拉到顶层 IO。
```

### 10.2 100MHz 时序差的主要原因

100MHz 跑不过不是因为模型理论上不能达到 100MHz，而是当前实现方式让单周期路径过长。

综合日志中 DSP report 显示大量 DSP 内部寄存器未使用：

```text
AREG = 0
BREG = 0
MREG = 0
PREG = 0
```

这意味着多个 MAC/BN/FC 相关路径基本是：

```text
输入寄存器
  -> LUT/MUX 选择
  -> DSP 乘法
  -> 48-bit 加法/累加
  -> 饱和/BN/ReLU/状态选择
  -> 输出寄存器
```

过多逻辑被压在一个 10ns 周期内，100MHz 失败是预期结果。

日志还大量出现：

```text
Synth 8-5844:
Detected registers with asynchronous reset at DSP/BRAM block boundary.
Consider using synchronous reset for optimal packing.
```

涉及模块包括：

```text
pw_conv2_array_4x12_compute.v
fc_classifier_96_5_compute.v
stem_conv_array_4x4_compute.v
```

这说明 datapath 附近大量寄存器使用异步复位，Vivado 难以把寄存器吸收到 DSP 的 AREG/BREG/MREG/PREG 或 BRAM 边界寄存器中，导致 DSP 前后缺少有效流水。

同时资源报告中：

```text
MUXF7 = 17432
MUXF8 = 4299
LUT6  = 62089
```

说明设计中存在大量宽 mux 或数组索引选择逻辑，可能来源包括：

```text
1. buffer 的 slot_valid/slot_tag metadata 查询。
2. FC 中 avg_mem/hidden_mem/logit_mem 的 variable-index 访问。
3. 顶层宽 debug/status 输出。
4. 多层 ready/valid/state 组合判断。
5. 宽向量切片和动态通道选择。
```

### 10.3 需要优先优化的 RTL 方向

优先级 1：数据通路改同步复位或无复位

```text
1. 控制状态机、valid 位继续保留 reset。
2. MAC accumulator、pipeline data、DSP 输入输出寄存器尽量改同步复位。
3. 对纯 datapath 寄存器，如果 valid 能保证数据有效性，可以考虑不 reset。
```

目的：

```text
让 Vivado 能把寄存器推入 DSP/BRAM 边界，提高 AREG/BREG/MREG/PREG 使用率。
```

优先级 2：显式给 systolic PE 和 FC MAC 打流水

当前需要重点检查：

```text
stem_conv_systolic_pe
pw_conv1_systolic_pe
pw_conv2_systolic_pe
fc_classifier_96_5_compute
```

目标不是只在 Verilog 中多写几个寄存器，而是让综合报告中 DSP 显示类似：

```text
AREG > 0
BREG > 0
MREG > 0
PREG > 0
```

建议后续 PE 结构按如下方向拆分：

```text
cycle N:
  register act/weight/acc_in/control

cycle N+1:
  DSP multiply

cycle N+2:
  DSP accumulate / post add

cycle N+3:
  BN/ReLU/saturation/output valid
```

实际可按现有计算节奏折中，但不能让 16x16 multiply + 48-bit add + postprocess 全部处在同一组合周期。

优先级 3：减少 buffer metadata 的大组合扫描

当前 buffer 中 `slot_valid/slot_tag` 的组合查询在仿真时已出现大数组 sensitivity warning。综合后大量 MUXF7/MUXF8 也提示这类逻辑可能较重。

后续优化方向：

```text
1. 避免在一个组合 always 中扫描全部 slot。
2. 对当前请求的 bank/slot 做直接索引比较。
3. 将 row_ready / tile_ready 判断寄存一拍。
4. 对 served watermark / overwrite 判断做分阶段寄存。
```

允许 buffer ready 多一拍延迟，比形成长组合路径更可控。

优先级 4：重新生成 BRAM IP output products 的 OOC 时钟

综合日志中出现：

```text
Clock period '20.000' specified during out-of-context synthesis ...
actual clock period '10.000'
```

说明部分 BRAM IP 的 OOC 综合结果按 50MHz 生成，而 full top 按 100MHz 约束。

```text

2. reset/regenerate output products。
3. 确认不再出现 20ns vs 10ns 的 OOC clock mismatch warning。
```

这不是当前最主要的时序失败根因，但应在正式实现前清理。

### 10.4 不建议立即做的事


### 10.5 FC AVG 路径首轮拆拍优化

根据 Vivado 最差 setup path：

```text
Startpoint: gap_latched_mem_reg[55][0]
Endpoint:   avg_mem_reg[0][0]
```

确认 FC classifier 中 `gap_sum / 174 -> saturate -> avg_mem` 曾被压在同一个周期内。今日先对
`fc_classifier_96_5_compute.v` 做首轮 RTL 拆拍：

```text
AVG_LANES = 12
AVG_GROUPS = 8

STATE_AVG_LOAD:
  每次从 gap_latched_mem 取 12 个通道，写入 avg_gap_group。

STATE_AVG_DIV:
  对这 12 个通道执行 /174，写入 avg_div_group。

STATE_AVG_STORE:
  对这 12 个结果做 saturate，写入 avg_mem。
```

这样 96 个通道的 GAP average 不再由单拍完成，而是 8 组处理，每组 3 拍，总计 24 拍。目的不是提高吞吐，而是拆断：

```text
gap_latched_mem dynamic mux -> /174 divider -> saturation -> avg_mem
```

这条长组合路径。

`AVG_LANES` 已从 compute 透传到 `fc_classifier_96_5.v`，并在 full top 中暴露为
`FC_AVG_LANES` 参数，默认值为 12。后续可以在不改 compute 源码的情况下试 12/24 通道一组的不同折中。

当前验证结果：

```text
PASS fc_classifier_96_5_compute self-check
PASS fc_classifier_96_5 self-check
PASS fc_classifier_96_5 self-check with AVG_LANES=24 override
PASS tb_stem_dw1_pw1_dw2_pw2_pipeline_top
```

full-top mock 1 case 结果：

```text
CASE 0 cycles=39764 gap_count=174 class=2 fc=1/1 errors_delta=0
```

注意：该修改减少了 96 路同时 average 的扇出和 mux/postprocess 深度，但 `STATE_AVG_DIV` 内部仍是 12 路并行 48-bit `/174`。如果重新综合后最差路径仍落在 FC AVG 除法阶段，下一步应把 `/174` 改成常数倒数乘法/移位或进一步拆成更深流水，而不是继续只减少并行通道数量。
