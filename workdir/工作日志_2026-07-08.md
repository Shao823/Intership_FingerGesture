## 2. stem_to_dw1_buffer 级间缓存

### 2.1 模块结构

当前采用 controller + with_ram 两层结构：

```text
stem_to_dw1_buffer.v
  只负责地址映射、tag/valid、水位、写读控制和握手。

stem_to_dw1_buffer_with_ram.v
  例化 stem_to_dw1_buffer + blk_mem_gen_stem_to_dw_conv1。
```

这样保持了和 stem weight/activation controller 一致的项目风格：

```text
controller 本体可用 mock RAM 测 latency=1/2。
with_ram 层用于真实 Vivado BMG IP 仿真。
```

### 2.2 存储组织

当前 BRAM payload 逻辑组织：

```text
word = 4 channels at one time point
word width = 4 * 16 = 64 bit

addr = bank * RING_ROWS * ROWS + slot * ROWS + row
bank = ch_base / 4
slot = time ring slot
row  = local time row in tile
```

metadata 不放进 BRAM，而是用寄存器/LUTRAM 维护：

```text
slot_valid[bank][slot]
slot_tag[bank][slot]
served_valid[bank]
served_out_t[bank]
```

这样 BRAM 只存 payload，tag/valid 更新和查询更直接，也避免读写 payload 时额外拆 tag 位。

### 2.3 读写契约

stem 写入：

```text
stem_out_tile_valid
stem_out_tile_t_base
stem_out_tile_oc_base
stem_out_tile[4 rows x 4 channels]
```

DW1 读取：

```text
dw1_win_req_valid
dw1_win_req_out_t
dw1_win_req_ch_base

返回 5 taps x 4 channels。
padding 位置补 0。
非 padding 位置必须已有对应 tag/valid。
```

当前设计依赖的 scheduler 契约：

```text
DW1 对每个 bank 的 win_req_out_t 必须按单调顺序推进，不能跳过后长期不补。
served watermark 用于判断 ring slot 是否可以被 stem 后续覆盖。
```

如果 top scheduler 跳过某个 out_t，buffer 可能因 served watermark 不推进而反压 stem。仿真里已加入 watermark stress case，用于暴露这类问题。

## 4. stem activation controller 优化

今天继续确认 stem activation RAM 的读延迟和 FIFO 设计。

当前关键点：

```text
1. Vivado BMG IP 报告的 READ_LATENCY_A/B 是 IP 输出寄存器/读端口级延迟。
2. controller 内部的 BRAM_READ_LATENCY 参数必须和仿真/真实 IP 配置匹配。
3. 对于已经发出的 RAM read，即使下游突然 ready=0，读响应仍会回来，所以 response FIFO 必须能吸收在途数据。
```

已完成：

```text
1. stem activation controller 支持 mock latency=1/2 测试。
2. with_ram 真实 BMG 测试通过。
3. 顶层 stem_conv_array_4x4 使用真实 activation RAM + weight ROM 测试通过。
```

关于 FIFO 深度：

```text
FIFO_DEPTH = BRAM_READ_LATENCY + 额外余量

目的不是让下游永远不 ready 时继续无限发请求，而是保护：
  已经发出、但读延迟还在路上的 response 不会因为下游突然反压而丢失。
```

## 6. PW1 cycle 分析

### 6.1 单个 4x8 oc_group

按当前 RTL，一个 PW1 `4x8` 小 tile 指：

```text
input:  4 high-res rows x 32 channels
output: 2 pooled rows x 8 output channels
```

理想/无 backpressure 情况下：

```text
cycle 0     start + input_tile_req_ready
cycle 1     进入 MAC，开始发 k=0 请求
cycle 4     第一组 weight/activation 对齐后 feed PE
cycle 4-35  32 个 k 连续 feed
cycle 36-45 systolic flush，10 cycles = 4 + 8 - 2
cycle 46    POST0
cycle 47    POST1
cycle 48    POOL
cycle 49    DONE 状态装载 out_valid/out_tile
cycle 50    out_ready 接走输出
```

因此：

```text
start -> out_fire: 约 50 cycles
进入 MAC 后算:   约 49 cycles
```

和 stem `4x4` 小 tile 的约 49 cycles 基本同速。

### 6.2 完整 4-row x 64-channel PW1 job

当前 PW1 顶层一次 start 会内部扫：

```text
oc_base = 0,8,16,24,32,40,48,56
```

即 8 个 `4x8` oc_group。

估算：

```text
first group: 50 cycles
后续 group: 50 cycles + 1 cycle STATE_CLEAR

total = 50 + 7 * 51 = 407 cycles
```

纯 MAC 下限：

```text
4 rows * 64 out_ch * 32 in_ch = 8192 MAC
4x8 array = 32 MAC/cycle
8192 / 32 = 256 cycles
```

当前有效利用率：

```text
256 / 407 = 62.9%
```

结论：

```text
PW1 不是纯计算慢，而是每个 oc_group 都重复付出了 activation 搬运、flush、pool/output 等开销。
```

## 7. 明日优先工作：PW1 优化方向

### 7.1 问题 1：activation 重复搬运

当前 PW1 完整 4-row job：

```text
8 个 oc_group
每个 oc_group 请求 32 个 act_vec

activation 请求 = 8 * 32 = 256 次
```

但同一个 `4 rows x 32 channels` 输入 tile 对 8 个 oc_group 完全相同，理论上只需要读一次：

```text
activation 请求 = 32 次
```

推荐明天优先实现：

```text
pw1_input_tile_cache:
  depth = 32
  width = 4 rows * 16 bit = 64 bit
  total = 32 * 64 = 2048 bit

第一个 oc_group:
  正常从 upstream row packer 读取 act_vec[k]。
  同时写入 act_cache[k]。

后续 7 个 oc_group:
  不再发外部 act_req。
  直接从 act_cache[k] 取 feed_act_vec。
```

预期收益：

```text
1. activation 外部请求从 256 次降到 32 次。
2. 降低 DW1->PW1 row packer 读端压力。
3. 降低后续集成时因 activation 读取端口/握手造成 bubble 的概率。
4. 和设计文档中的 pw1_input_tile_regs = 4 x 32 x 16 = 2048 bit 一致。
```

注意：

```text
这项优化主要降低搬运压力和 backpressure 风险。
如果 compute 每拍仍只消费一个 k，单靠 cache 不会把 407 cycles 直接降到 256 cycles。
```

### 7.2 问题 2：每个 oc_group 重复付出 flush / pool / output 开销

当前 PW1 `4x8` 每组固定开销：

```text
flush = ROWS + OC_LANES - 2 = 4 + 8 - 2 = 10 cycles
POST0 = 1 cycle
POST1 = 1 cycle
POOL  = 1 cycle
DONE/out handshake = 约 2 cycles
组间 STATE_CLEAR = 1 cycle
```

相比 stem `4x4`：

```text
stem flush = 4 + 4 - 2 = 6 cycles
PW1  flush = 4 + 8 - 2 = 10 cycles
```

推荐明天第二步实现：

```text
next-oc_group weight prefetch FIFO:
  在当前 group 的 flush / post / pool / output 阶段预取下一组权重。
  目标是隐藏下一组开头的 ROM 响应延迟。

需要本地 FIFO:
  width = 8 lanes * 16 bit = 128 bit
  depth 至少覆盖若干个 k，理想可到 32。
```

可先做保守版本：

```text
1. 不改变输出协议。
2. 不改变 PE array。
3. 不改变 weight ROM IP。
4. 只在 compute 内部增加 next weight prefetch FIFO 和状态控制。
5. 仿真先证明功能等价，再统计 start->done cycles 是否下降。
```

进一步优化方向：

```text
如果后续仍有背压风险，再考虑更激进的结构：

1. 增大 PW1 OC_LANES，例如 4x16，减少 oc_group 数。
2. 输出侧增加 tile FIFO，避免 out_ready 短 stall 阻塞下一组计算。
3. 将 pool/output 与下一组 MAC 有限重叠，但这需要更小心地处理 PE post/pool 寄存器生命周期。
```
