## 2. 当前推荐硬件参数

最终今天收敛到的第一版 RTL 推荐参数：

```text
stem: 4x4
DW1:  P_CH_DW=4, P_TAP_DW=5
PW1:  4x8
DW2:  P_CH_DW=4, P_TAP_DW=5
PW2:  4x12
FC:   可独立 P_OUT=16，也可优先复用 PW2
```

对应前端并行乘法资源：

```text
stem 4x4:                 16
DW1 P_CH=4/P_TAP=5:       20
PW1 4x8:                  32
DW2 P_CH=4/P_TAP=5:       20
PW2 4x12:                 48
GAP 前端合计:             136
独立 FC P_OUT=16:         16
all-DSP + 独立 FC 总计:   152
```

判断：

```text
1. 若 all-DSP frontend，136 个 DSP 适合 150+ DSP 级别器件，但占用不低。
2. 对 80~100 DSP 级别小 FPGA 偏紧。
3. 优先保 DSP 的是 PW1/PW2。
4. DW 更适合转 LUT / 常系数乘法 / shift-add。
5. 若目标只有约 90 DSP，可进一步考虑 stem 转 LUT，使 DSP 估算降到 80。
```

## 3. 周期模型和流水判断

使用脚本：

```bash
python3 preparation/lightcnn1d_streaming_pipeline_design/cycle_model.py --stem-poc 4 --pw1-poc 8 --pw2-poc 12 --dw-pch 4 --dw-ptap 5 --recommended-only
```

关键结果：

```text
stem = 560 cycles / 8 high-res rows
DW1  = 64 cycles / 8 high-res rows
PW1  = 512 cycles / 8 high-res rows
DW2  = 64 cycles / 4 low-res rows
PW2  = 512 cycles / 4 low-res rows
II   = 560 cycles / chunk
```

今天明确纠正了一个点：`DW P_TAP=5` full-tap 版本下，DW1/DW2 中间 chunk 不是 320 cycles，而是：

```text
DW1 middle = 8 rows * ceil(32/4) = 64 cycles
DW2 middle = 4 rows * ceil(64/4) = 64 cycles
```

`320 cycles` 只对应 `P_TAP_DW=1` 的 tap-serial 微架构。

同时用脚本核对了降低 PW 并行度的后果：

```text
PW1 4x7 / PW2 4x11:
    middle II = 640 cycles
    bottleneck = PW1
```

因此在保持 `stem 4x4` 且不让 stem backpressure 的前提下：

```text
PW1 P_OC=8  基本是下限
PW2 P_OC=12 基本是下限
```

## 4. 事件驱动 buffer/pipeline 模型

今天增强了：

```text
preparation/lightcnn1d_streaming_pipeline_design/buffer_model.py
```

新增 `End-to-GAP/AveragePool timing` 输出，用于计算在 `stem` 连续计算、`PW1/PW2` 等待 4 行启动、最后 partial flush 的条件下，`AveragePool/GAP` 什么时候收到全部 PW2 数据。

运行命令：

```bash
python3 preparation/lightcnn1d_streaming_pipeline_design/buffer_model.py --stem-poc 4 --pw1-poc 8 --pw2-poc 12 --dw-pch 4 --dw-ptap 5
```

关键结果：

```text
Stem last 4-row micro-tile done:   24360 cycles
DW1 last row produced:             24408 cycles
PW1/Pool all low-resolution rows:  24894 cycles
DW2 last row produced:             24958 cycles
PW2 final group starts:            25438 cycles
PW2/GAP receives all rows:         25950 cycles
Drain after final stem micro-tile:  1590 cycles
```

结论：

```text
到 AveragePool/GAP 输入数据全部拿到为止:
25,950 cycles ~= 259.50 us @100 MHz
```

同时明确了 PW2 最后一组只有 2 个 low-res rows：

```text
rows = [172, 173]
```

但在当前 `PW2 4x12` valid-mask 架构下，最后 2 行仍然占用一个完整 PW2 tile：

```text
ceil(2/4) * ceil(96/12) * 64 = 512 cycles
```

因为 4 个 time lanes 中只有 2 个有效，并不会减少 `Cin x Cout` 的循环次数。

## 5. 级间缓存和存储统计

今天增强了：

```text
preparation/lightcnn1d_streaming_pipeline_design/storage_model.py
```

新增拆分 subtotal：

```text
Required inter-stage windows/FIFOs/tile regs
Optional input/debug/backpressure buffers
GAP/FC runtime state vectors
Weight ROM only
Folded BN and FC bias ROM
```

当前 16-bit Q8.8 参数下，关键存储结果：

```text
必需级间缓存:
30,720 bit = 3,840 B

加 optional input/debug FIFO:
37,664 bit = 4,708 B

级间/state subtotal，含 optional FIFO + GAP/FC 状态:
45,424 bit = 5,678 B

计算单元本地寄存器:
11,107 bit ~= 1,388 B

纯 weight ROM 全片上:
311,808 bit = 38,976 B

weight + folded BN + FC bias 参数 ROM:
319,568 bit = 39,946 B

runtime with parameter ROM:
376,099 bit ~= 47,012 B
```

如果完整输入 `input[5][348]` 也全部放片上，额外增加：

```text
5 x 348 x 16 = 27,840 bit = 3,480 B
```

结论：

```text
1. 当前设计存储压力不大，真正级间缓存只有 3.84 KB。
2. 参数 ROM 约 39 KB，是主要片上存储项。
3. 实际综合时还要考虑 BRAM/LUTRAM/register 的 banking、端口数和字宽对齐。
4. PW1/PW2 的 weight ROM bank/读口设计比纯 bit 数更关键。
```

## 6. 资源优化路线

今天补充了当前架构对小 FPGA 的资源适配判断。

推荐优化顺序：

```text
第一版:
    all-DSP frontend + FC 复用或独立均可

第二版:
    DW 转 LUT / 常系数乘法 / shift-add
    DSP 从 136 降到 96
    II 仍保持 560 cycles/chunk

第三版:
    DW tap 串行
    DW 乘法从 40 降到 8
    DW 周期变为 320 cycles，但仍不超过 560-cycle stem chunk

第四版:
    stem 转 LUT 或低比特量化
    DSP 可进一步降到约 80
```

不建议优先降低 `PW1/PW2` 的并行度，因为它们已经接近维持 `560-cycle/chunk` 的下限。

## 8. 当前后续建议

下一步建议开始 RTL 模块拆分，不再继续扩展旧 baseline。

优先顺序：

```text
1. 定义全局 fixed-point 参数、layer shape 参数、valid/ready 接口规范。
2. 实现 stem_conv_array_4x4。
3. 实现 DW slide PU，先用 P_CH=4/P_TAP=5 full-tap。
4. 实现 PW1 4x8 + folded BN + ReLU + MaxPool。
5. 实现 PW2 4x12 + folded BN + ReLU + GAP accumulator。
6. 实现 row FIFO / row packer / valid_mask final flush。
7. 最后再接 FC0/FC1，并优先考虑复用 PW2 资源。
```

今天没有运行旧 RTL baseline，也没有继续维护旧 baseline RTL。后续 RTL 应以 `lightcnn1d_streaming_pipeline_design` 中的文档和脚本为准。
