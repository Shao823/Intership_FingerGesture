## 2. Stem Weight ROM Controller

### 2.1 ROM IP 配置

当前 `blk_mem_gen_stem_weight` 配置：

```text
Memory type: Single_Port_ROM
Width: 64 bit
Depth: 280
Address width: 9
READ_LATENCY_A: 1
Init file: weight_data/stem/weight_packed.coe
```

权重组织方式：

```text
word width = 4 lanes * 16 bit = 64 bit
depth      = 35 * (32 / 4) = 280

addr = (oc_base / 4) * 35 + k

word[lane*16 +: 16] = W_stem[oc_base + lane][k]
lane 0 in LSB
lane 3 in MSB
```

### 2.2 Controller 逻辑

`stem_weight_controller.v` 的职责：

```text
1. 接收 compute 的 weight_req_oc_base / weight_req_k。
2. 计算 ROM 地址。
3. 管理 blk_mem_gen 的 read latency 和 ena 推进。
4. 用 response FIFO 吸收下游 backpressure。
5. 按请求顺序返回 weight_vec。
```

当前已参数化：

```text
parameter integer BRAM_READ_LATENCY = 1
```

内部使用：

```text
rom_valid_d[BRAM_READ_LATENCY:0]
FIFO_DEPTH = BRAM_READ_LATENCY + 3
```

关键点：

```text
真实 BMG READ_LATENCY_A=1 时，douta 在下一拍时钟边沿后更新。
controller 不能在同一个边沿采刚更新的数据。
因此 controller 会保持 rom_en 推进读流水，并在对应 valid 延迟后采样 rom_dout。
```

已补充的测试指标：

```text
1. first_gap
   检查无 backpressure 时首个 response 的实际延迟。

2. unique_addrs
   检查一个 sequence 内实际访问了多少个唯一 ROM 地址。

3. invalid_req_count
   仿真态检查非法 oc_base/k 请求是否被抓到。
```

## 3. Stem Activation RAM Controller

### 3.1 RAM IP 配置

当前 `blk_mem_gen_stem_activation` 配置：

```text
Memory type: Simple_Dual_Port_RAM
Port A: write
Port B: read
Width: 64 bit
Depth: 435
Address width: 9
READ_LATENCY_B: 1
Load_Init_File: false
```

activation 是运行时写入的中间/输入数据，因此没有 `.coe` 初始化文件是合理的。

RAM 数据组织方式：

```text
INPUT_LEN  = 348
STEM_IC    = 5
ROWS       = 4
TIME_WORDS = ceil(348 / 4) = 87
DEPTH      = 5 * 87 = 435

addr = in_ch * TIME_WORDS + (input_t / 4)
lane = input_t % 4

word[lane*16 +: 16] = input[input_t][in_ch]
```

### 3.2 Controller 搬运逻辑

compute 每个 `k` 请求一个 4-row activation vector：

```text
act_req_t_base = tile_t_base
act_req_k      = k_flat

in_ch = k_flat / 7
tap   = k_flat % 7
in_t(row) = tile_t_base + row + tap - padding
padding = 3
```

controller 对 4 个 row 分别判断：

```text
if in_t < 0 or in_t >= INPUT_LEN:
  该 row 是 padding，输出 0，不读 RAM。
else:
  计算 RAM addr 和 lane。
```

因为 RAM 宽度为 4 个时间点，所以一次 4-row 读取可能有两种情况：

```text
1. 四个 row 落在同一个 RAM word：
   只读 1 个地址。

2. 四个 row 跨过 4-wide 边界：
   读 2 个地址，再按 lane 拼成 act_vec。
```

例子：

```text
tile_t_base = 37
k_flat = 10

in_ch = 10 / 7 = 1
tap   = 10 % 7 = 3

row0 in_t = 37
row1 in_t = 38
row2 in_t = 39
row3 in_t = 40

37/4 = 9, lane 1
38/4 = 9, lane 2
39/4 = 9, lane 3
40/4 = 10, lane 0

需要读:
  addr = 1*87 + 9  = 96
  addr = 1*87 + 10 = 97
```

返回：

```text
act_vec[15:0]   = A[37][ch1]
act_vec[31:16]  = A[38][ch1]
act_vec[47:32]  = A[39][ch1]
act_vec[63:48]  = A[40][ch1]
```

padding 例子：

```text
tile_t_base = 0
k_flat = 0
tap = 0

row0 in_t = -3 -> 0
row1 in_t = -2 -> 0
row2 in_t = -1 -> 0
row3 in_t =  0 -> A[0][ch0]
```

补零不是 RAM 里真的存 padding zero，而是在 `assemble_vec()` 拼输出时用 `valid_mask` 控制：

```text
valid_mask[row] = 1 -> 从 RAM word 对应 lane 取数据
valid_mask[row] = 0 -> sample 保持 0
```

## 4. Stem 顶层接口调整

### 4.1 Compute 拆分

原始计算阵列保留为：

```text
stem_conv_array_4x4_compute.v
module stem_conv_array_4x4_compute
```

它仍然只关心计算语义：

```text
weight_req / weight_vec
act_req / act_vec
start / done / out_valid
```

### 4.2 当前顶层 wrapper

新的 `stem_conv_array_4x4.v` 是集成 wrapper。

对外不再暴露：

```text
weight_req_*
weight_vec_*
act_req_*
act_vec_*
```

对外暴露 activation RAM 写口：

```text
act_wr_en
act_wr_addr
act_wr_data
act_wr_ready
```

含义：

```text
外部先把 input activation 写入 stem activation RAM。
启动 start 后，compute 的 act_req 会在顶层内部交给 activation controller。
activation controller 从 RAM 读取、处理 padding/跨 word，并返回 act_vec。
```

当前顶层仍保留 folded BN 写口：

```text
w_fold_we / w_fold_oc / w_fold_wdata
bias_fold_we / bias_fold_oc / bias_fold_wdata
```

## 9. Vivado 报错记录

今天遇到 Vivado 报错：

```text
[Common 17-180] Spawn failed: No such file or directory
```

当前判断：

```text
如果仿真仍然正常编译、运行并输出 PASS，则该错误大概率不是 RTL 功能错误。
更可能是 Vivado 尝试启动某个外部程序或脚本时路径不存在。
```

高概率来源：

```text
1. Windows Vivado 与 WSL /mnt/d 路径混用。
2. Vivado 配置的外部 editor/browser/terminal 路径不存在。
3. 某个仿真脚本或 post-process 命令找不到可执行文件。
```

后续如果需要定位，需要查看该错误前后 20 行 Vivado log，确认 Vivado 试图 spawn 的具体对象。

### 11.2 Activation controller 性能优化

当前 activation controller 是正确性优先版本：

```text
每个 act_req 独立处理。
一个请求最多读两个 RAM word。
请求处理完并返回 act_vec 后才接收下一个请求。
```

好处：

```text
状态机简单。
padding / 跨 word / latency / backpressure 行为清晰。
仿真容易证明正确。
```

后续如需提高吞吐，可以考虑：

```text
1. 对连续 k/tile 的相邻窗口做 small cache。
2. 允许下一个请求提前发起 RAM 读。
3. 增加 response FIFO，支持有限 outstanding request。
```
