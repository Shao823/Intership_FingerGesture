## 三、Vitis HLS IP 重新生成

1. 在 Vitis HLS 2022.2 中重新生成 `mnist_nn_predict` HLS IP。
2. 观察导出 RTL 时出现的长 IP 名称警告，例如：
   - `mnist_nn_predict_fadd...`
   - `mnist_nn_predict_fdiv...`
   - `mnist_nn_predict_fexp...`
   - `mnist_nn_predict_fmul...`
3. 判断这些长名称警告不是致命错误，但说明原始 `float + expf/div` 实现会生成大量浮点 IP。
4. 解释 HLS C/C++ 中哪些部分会综合成时序逻辑，哪些是组合逻辑：
   - 寄存器、状态机、BRAM 接口、流水线阶段属于时序逻辑。
   - 单周期表达式、比较器、加法器、MUX、查表选择逻辑通常属于组合逻辑。

## 四、DSP 超标问题定位

1. Vivado implementation 报告显示 DSP48E1 超标：
   - 需求约 `329`
   - Zynq-7020 可用 `220`
2. 判断主要原因是 HLS 源码中使用了大量 `float` 乘加、`expf` 和除法。
3. 原始 sigmoid：
   ```cpp
   1 / (1 + expf(-x))
   ```
   会生成 `fexp`、`fdiv`、`fadd`、`fmul` 等浮点 IP，资源开销较大。

## 五、HLS 源码资源优化

主要修改文件：

```text
mnist-nnet-hls-zynq7020-fpga/hls/mnist/src/hls_mnist.cpp
```

完成的优化包括：

1. 移除 `float` 内部计算路径，将网络内部计算改成定点数。
2. 引入定点类型：
   ```cpp
   typedef ap_fixed<18, 8, AP_RND, AP_SAT> data_t;
   typedef ap_fixed<36, 16, AP_RND, AP_SAT> acc_t;
   ```
3. 保留外部输入输出接口为 `float`，以减少 Vivado BD 和软件侧修改。
4. 将内部权重、bias、中间输出、累加器改为定点数。
5. 用查表 sigmoid 替代 `expf/div` 实现：
   - 范围：`[-8, 8]`
   - 步长：`0.5`
   - 表长：`33`
   - 超出范围时饱和到接近 `0/1`
6. 第一阶段曾将乘法全部绑定到 LUT/fabric，以降低 DSP 使用量。
7. 后续发现 LUT 压力较大，于是改为部分乘法使用 DSP：
   - 第一层继续走 LUT/fabric。
   - 第二层和第三层走 DSP。
8. 新增两个乘法函数：
   ```cpp
   lut_mul()
   dsp_mul()
   ```
   分别控制乘法使用 LUT 或 DSP。

## 六、HLS 综合结果对比

### 全部乘法尽量走 LUT/fabric 后

HLS 结果大致为：

```text
DSP: 1 / 220
LUT: 51761 / 53200, 约 97%
Estimated Fmax: 68.78 MHz
```

结论：

1. DSP 问题解决。
2. LUT 使用率过高，Vivado implementation 风险较大。

### 第二层和第三层改用 DSP 后

最新 HLS 结果：

```text
Clock target: 20.00 ns
Estimated delay: 14.064 ns
Estimated Fmax: 71.10 MHz
DSP: 97 / 220 = 44%
LUT: 19474 / 53200 = 36%
FF: 7879 / 106400 = 7%
BRAM: 49 / 280 = 17%
Latency: 50447 cycles
```

结论：

1. DSP 使用低于 Zynq-7020 上限。
2. LUT 从约 97% 降到约 36%，资源余量明显改善。
3. HLS timing 对 50 MHz 时钟满足要求。

## 七、时钟约束确认

2. 查看 Vivado 截图和工程文件，确认 Vivado 中 PL 时钟：
   ```text
   FCLK_CLK0 = 50 MHz = 20 ns
   ```
3. Vivado BD 中确认：
   ```text
   mnist_nn_predict_0/ap_clk
   ```
   连接到：
   ```text

   ```
4. Vivado 生成的 OOC 约束中也确认：
   ```tcl
   create_clock -name processing_system7_0_FCLK_CLK0 -period 20 [get_pins processing_system7_0/FCLK_CLK0]
   ```
5. 结论：当前 `20 ns` 的 HLS IP 与 Vivado 工程时钟匹配。

## 八、Vitis HLS 报错处理

处理过的主要错误：

1. `ap_fixed` 与 `float` 字面量混合导致重载歧义：
   ```text
   use of overloaded operator '<=' is ambiguous
   use of overloaded operator '>=' is ambiguous
   use of overloaded operator '*' is ambiguous
   ```
2. 解决方式：
   - 将 `-4.0f`、`4.0f`、`0.125f`、`0.5f` 等 float 常量改成显式 `acc_t` 或 `data_t`。
   - 将累加器初始化从 `0.0f` 改成 `(acc_t)0`。

## 十、后续注意事项

1. Vivado 最终结果以 implementation 后的报告为准，HLS 资源和时序只是估计值。
2. 如果 Vivado timing fail，应优先查看 critical path，而不是盲目继续改 HLS。
3. 如果 DSP 仍有余量但 LUT 或 timing 紧张，可以继续将部分 LUT 乘法切回 DSP。
4. 如果准确率下降明显，需要重新评估定点位宽和 sigmoid 查表精度。
5. Vitis/Vivado 自动生成文件较多，后续应重点保留可复现工程所需文件，避免无关中间文件影响工程维护。
