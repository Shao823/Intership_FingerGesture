## 五、RTL 架构设计文档

新建并持续修改：

```text
preparation/LightCNN1D_RTL加速器当前设计思路.md
```

形成的主要设计判断：

1. `stem / PW1 / PW2 / FC` 应统一映射到 output-stationary GEMM/Conv PE 阵列。
2. `DW1 / DW2` 不适合直接用 GEMM 阵列，应使用专用 k=5 depthwise 滑窗 PU。
3. `PW1 + BN/ReLU + MaxPool` 和 `PW2 + BN/ReLU + GAP` 的融合不是拉长组合关键路径，而是注册化后处理流水，用来减少中间 buffer 读写。
4. 真实超级流水线需要多套计算资源；单套 GEMM 阵列只能做层内 tile 流水和分时复用。

## 六、流水线粒度与 systolic 参数分析

新建：

```text
preparation/LightCNN1D_systolic参数与流水线周期分析.md
```

今天重点澄清了一个关键概念：

```text
PE micro-tile 和 pipeline macro-tile 不是同一个东西。
```

例如 `4x8` GEMM 阵列表示：

```text
P_T  = 4 个时间点
P_OC = 8 个输出通道
```

一次 tile 完成后得到的是：

```text
4 time x 8 output channels = 32 values
```

它只是 PE 内部 micro-tile，不能直接送入下一层作为跨层流水单位。因为 DW1/DW2 有 k=5 滑窗依赖，MaxPool 会把时间长度减半，如果只传 4 个时间点，到后续 DW2 时数据长度不够。

因此跨层流水应使用更大的 macro-tile，并携带 halo。以最终 PW2 输出 `B` 个时间点为例：

```text
input 需要 2B + 18 点
stem  需要 2B + 12 点
PW1   需要 2B + 8 点
pool  需要 B + 4 点
PW2   输出 B 点
```

当前推荐：

```text
PE micro-tile: 4x8
pipeline macro-tile: B=16
DW PU: P_CH_DW=8
后处理: P_POST=16
```

## 七、当前结论

今天形成的核心结论：

1. `4x8` 不是一次生成 8 个时间点，而是一次覆盖 4 个时间点和 8 个输出通道。
2. `4x8` 输出太小，不能直接作为 DW/PW/Pool/DW2 的流水输入。
3. 应先用 `4x8` 连续计算，凑够带 halo 的 macro-tile，再交给下一层。
4. DW 是快速 producer，PW 是慢速 consumer，不能为了迎合 DW 输出节奏把 PW 阵列缩成 `2x2`。
5. 第一版优化建议先保留整层 buffer，完成 `4x8 GEMM + P_CH_DW=8 DW PU + 注册化后处理流水` 的正确性验证，再做 macro-tile 流水。
