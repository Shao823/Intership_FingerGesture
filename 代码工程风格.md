一、整体结论先讲

  项目代码整体规范程度较高：握手协议、复位策略、位宽推导模板、调试统计块有清晰的工程化约定。但compute 
  家族明显存在三代设计叠放（stem→pw1/pw2→fc 是不同代际的写法），buffer 家族有两套并存的读侧 
  FSM，外加若干零碎不一致点（命名、注释长度、属性标注）。
  
  ---
  二、做得好、风格一致的部分（建议保留）

  ┌──────────────┬───────────────────────────────────────────────────────────────────┐
  │     项目     │                               表现                                │
  ────────────────────────────────────────
  项目: 握手命名 
  表现: xxx_req_valid/ready/fire + resp_fire + out_valid/ready 跨模块统一
  ────────────────────────────────────────
  项目: 位宽推导模板
  表现: (X <= 1) ? 1 : $clog2(X + 1) 这种 localparam 写法在所有模块一致
  ────────────────────────────────────────
  项目: weight_controller 本体
  表现: stem/pw1/pw2/fc
    四套几乎是模板复制：fifo_data/fifo_count/fifo_rd_ptr/fifo_wr_ptr、rom_valid_d/rom_addr_hold/rom_pipe_en、reserved_count_after_pop、count_
  valid/any_pre_response_valid/next_fifo_ptr/calc_weight_addr
     四个 function 完全对齐
  ────────────────────────────────────────
  项目: *_with_rom / *_with_ram wrapper
  表现: 都是 60–115 行的 IP 集成 stub，风格完全一致
  ────────────────────────────────────────
  项目: \ifndef SYNTHESIS` 调试统计块
  表现: weight_controller 4 个 + buffer 都有，统一放在文件末尾，统计 invalid/stall 请求

  ────────────────────────────────────────
  项目: packed-data 字节序约定
  表现: 注释里反复写明"lane 0 在 LSB"，全工程一致
  
  ---
  三、关键不一致点（按重要性排序）
  
  ★★★ 1. compute 家族三代设计叠放
  
  维度: FSM 状态
  stem (早期): IDLE/MAC/POST0/POST1/POST2/DONE (6)
  pw1/pw2 (中期): IDLE/WAIT/RUN (3)
  fc (异类): 17 状态，编号跳号 (1,6,7,…,22 跳过 2-5/13/19)
  ────────────────────────────────────────
  维度: PE 后处理位置
  stem (早期): 外层 FSM 全局驱动 pe_post0/1/2_en
  pw1/pw2 (中期): PE 内部独立流水 (mac_valid/first/last+stage0/1)
  fc (异类): 无 PE，function 直接调
  ────────────────────────────────────────
  维度: 输出方式
  stem (早期): DONE 态一次性锁存整 tile
  pw1/pw2 (中期): 流式 post_valid + group-by-group
  fc (异类): STATE_OUT 等单次握手
  ────────────────────────────────────────
  维度: act_cache 跨 group 复用
  stem (早期): 无
  pw1/pw2 (中期): 有
  fc (异类): N/A
  ────────────────────────────────────────
  维度: out_ready 命名
  stem (早期): out_ready
  pw1/pw2 (中期): out_ready
  fc (异类): result_ready
  ────────────────────────────────────────
  维度: saturate_int8/requant_product_q15/relu_int8
  stem (早期): PE 内一份
  pw1/pw2 (中期): PE 内一份（与 stem 重复）
  fc (异类): compute 内又一份（共 4 份相同实现散落）
  
  建议：把 saturate_int8/requant_product_q15/relu_int8 抽到 \include文件（如emg_quant_funcs.vh`），所有 PE/compute
  共用；如果后续要加层，统一按 pw1/pw2 的"PE 内流水 + 流式输出"模板，stem 那一代可以视为待回收技术债。
  
  ★★★ 2. buffer 家族两套读侧 FSM 并存
  
  ┌────────────────────┬─────────────────────────────────────────────┬──────┐
  │       buffer       │                   读 FSM                    │ 行数 │
  ├────────────────────┼─────────────────────────────────────────────┼──────┤
  │ stem_to_dw1_buffer │ RD_IDLE/RD_RUN/RD_RESP (3 状态)             │ 556  │
  ├────────────────────┼─────────────────────────────────────────────┼──────┤
  │ pw1_to_dw2_buffer  │ RD_IDLE/RD_RUN/RD_RESP (3 状态)             │ 576  │
  ├────────────────────┼─────────────────────────────────────────────┼──────┤
  │ dw1_to_pw1_buffer  │ RD_IDLE/RD_LOAD/RD_COMMIT/RD_READY (4 状态) │ 737  │
  ├────────────────────┼─────────────────────────────────────────────┼──────┤
  │ dw2_to_pw2_buffer  │ RD_IDLE/RD_LOAD/RD_COMMIT/RD_READY (4 状态) │ 893  │
  └────────────────────┴─────────────────────────────────────────────┴──────┘

  写 FSM 都一致（WR_IDLE/WR_WRITE），但读侧出现两套不同 FSM，明显是不同时期写的两代。如果两边都过仿真，可考虑把 dw1_to_pw1/dw2_to_pw2 重构到
  3 状态模板（或反过来）。

  ★★ 3. bn_write_en / 折叠写端口风格不统一

  - stem compute：抽了 wire bn_write_en = (state == STATE_IDLE) && !start;
  - pw1 compute：内联 if ((state == STATE_IDLE) && !start)，没抽 wire
  - pw2 compute：又抽了 bn_write_en
  - dw_conv1/2：没有这个机制，用单独的 weight_we/weight_ch/weight_tap/weight_wdata 直接写 weight_mem

  dw_conv 与其他 compute 的"运行时改权重"接口完全是两套设计，命名也不同。

  ★★ 4. 状态命名约定不统一

  - IDLE/RUN 风格：pw1/pw2/dw_conv1/dw_conv2 ✓
  - IDLE/MAC/POST*：stem
  - IDLE/WAIT/RUN：pw1/pw2（多了 WAIT，但和 dw_conv 的 IDLE/RUN 不一致）
  - IDLE/AVG_LOAD/AVG_STORE/FC0_BIAS_REQ/...：fc（17 个）
  - WR_IDLE/WR_WRITE + RD_IDLE/RD_RUN/RD_RESP：buffer
  
  建议至少约定：localparam 状态名统一前缀 STATE_（stem/pw1/pw2/dw 都遵守了，fc 也遵守，buffer 改成 WR_/RD_ 前缀破坏了约定）。
  
  ★★ 5. emg_stream_fifo_slice.v 是孤岛

  - 无文件头注释（全工程唯一无注释模块）
  - 参数 DEPTH=2 但内部硬编码 data0/data1 两个寄存器
  - localparam FULL_COUNT = 2 + initial $error if DEPTH != 2 —— 参数化是假的
  - 与 weight_controller 里的 FIFO 实现风格完全不同（那边是 fifo_data 数组 + rd/wr_ptr）
  
  建议：要么真做参数化（用 reg [DATA_W-1:0] data [0:DEPTH-1]），要么直接去掉 DEPTH 参数写死，并加文件头注释说明"故意深度=2 的弹性 slice"。

  ★★ 6. 文件头注释长度悬殊

  ┌───────────────────┬──────────┐
  │       模块        │ 注释行数 │
  ├───────────────────┼──────────┤
  │ dw_conv1          │ ~70      │
  ├───────────────────┼──────────┤
  │ stem compute      │ ~60      │
  ├───────────────────┼──────────┤
  │ dw2_to_pw2 buffer │ ~35      │
  ├───────────────────┼──────────┤
  │ pw2 compute       │ ~50      │
  ├───────────────────────┼──────────┤
  │ pw1 compute           │ ~25      │
  ├───────────────────────┼──────────┤
  │ fc compute            │ ~20      │
  ├───────────────────────┼──────────┤
  │ weight_controller     │ 12-16    │
  ├───────────────────────┼──────────┤
  │ emg_stream_fifo_slice │ 0        │
  └───────────────────────┴──────────┘
  
  ★ 7. clear 信号的有无
  
  - buffer 4 套：统一有 clear 输入
  - pw2 compute：有 gap_clear
  - stem/pw1 compute、dw_conv1/2：只有 rst_n
  
  没有统一约定"何时该提供 clear、何时只靠 rst_n"。
  
  ★ 8. ram_style / use_dsp 属性标注零散
  
  - (* ram_style = "distributed" *)：stem_to_dw1 buffer 的 slot_valid/slot_tag
  - (* ram_style = "registers" *)：pw1/pw2 compute 的 act_cache
  - (* use_dsp = "yes" *)：stem/pw1/pw2 PE 的 mul_p
  - fc compute 的 a*b in function：无 DSP 标注
  - weight_controller 的 fifo_data：无属性
  
  没有统一的"哪种大小的存储该加哪种属性"规则。
  
  ★ 9. PW1 compute 注释里残留 deprecated 端口

  pw_conv2_array_4x12_compute.v:39-46 写了一段 "Deprecated debug tile stream, kept as interface reference only" —— 死代码注释，应该删掉或写到
   CHANGELOG。

  ★ 10. wrapper 文件 \define PROJECT_ROOT` 不统一

  - stem/pw1/pw2 wrapper：开头都有 \ifndef PROJECT_ROOT `define PROJECT_ROOT "D:/Intership"`
  - fc wrapper：没有（用 compute 内部已 define 的）
  - 但所有路径都硬绑定了 "D:/Intership" —— 不应该把绝对路径写死，会让别人在你这里 clone 后无法综合。
