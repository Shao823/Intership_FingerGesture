## 4. AXI IP 总体架构确定

今天最终采用的结构：

```text
                    +-----------------------------+
PS / CPU ---------->| S00_AXI                     |
                    | control/status registers    |
                    |                             |
DMA M_AXIS_MM2S --->| S00_AXIS                    |
                    | DMA frame controller        |
                    |                             |
                    | accelerator                 |
                    | internal activation banks   |
                    +-----------------------------+
                                  |
                                  +--> irq
```

职责划分：

```text
S00_AXI:
  控制寄存器
  状态寄存器
  软件启动
  自动启动
  DONE sticky
  class result
  error status
  irq enable/output

S00_AXIS:
  接收 DMA 数据
  TREADY 背压
  TKEEP/TLAST 检查
  帧计数
  写 accelerator activation RAM

accelerator:
  神经网络实际计算
  内部 activation ping-pong bank 切换
  busy/done/class 输出
```

最初曾考虑把 `accelerator` 直接实例化在 `S00_AXI` 中。今天最终没有采用这种方式，而是在 `emg_accelerator_v1_0.v` 顶层实例化。

原因：

```text
1. S00_AXI 应尽量只承担 AXI-Lite 协议和寄存器职责。
2. accelerator 同时连接 AXI-Lite 控制面和 AXI-Stream 数据面。
3. 如果实例化在 S00_AXI 内，S00_AXIS 的 activation 写端口需要穿过 AXI-Lite 模块，层次关系反而更混乱。
4. 顶层集中进行 start 仲裁、frame consume、busy/done/result 连接更清楚。
5. 后续增加 M_AXIS、性能计数器或 CDC 时更容易扩展。
```

## 5. DMA Ping-Pong Controller 实现

新增文件：

```text
ip_repo/emg_accelerator_1_0/src/emg_dma_pingpong_controller.v
```

### 5.1 固定输入帧格式

网络输入：

```text
5 channels × 348 samples × INT8
= 1740 bytes
```

AXI-Stream 数据宽度：

```text
32 bit = 4 INT8 samples/beat
```

因此每帧：

```text
1740 / 4 = 435 beats
```

controller 输出：

```text
act_wr_en
act_wr_addr = 0..434
act_wr_data = S_AXIS_TDATA
```

数据布局保持与 activation RAM 相同：

```text
channel-major
每个通道内部按时间顺序排列

TDATA[7:0]   = sample t+0
TDATA[15:8]  = sample t+1
TDATA[23:16] = sample t+2
TDATA[31:24] = sample t+3
```

由于 `348` 可以被 `4` 整除，因此每个通道正好占：

```text
348 / 4 = 87 words
```

完整地址顺序：

```text
channel 0: addr   0..86
channel 1: addr  87..173
channel 2: addr 174..260
channel 3: addr 261..347
channel 4: addr 348..434
```

### 5.2 AXI-Stream 握手和背压

正常接收条件：

```verilog
axis_fire = S_AXIS_TVALID && S_AXIS_TREADY;
```

`TREADY` 只在以下条件满足时拉高：

```text
1. controller 已退出复位。
2. 没有一帧完整数据正在等待 accelerator 消费。
3. accelerator 当前 activation 写 bank 可以接受写入。
```

当 accelerator 正在计算上一帧时，它已经在 `start` 时交换了 read/load bank，因此 DMA 可以继续把下一帧写入另一个 bank。

当下一帧已经完整接收但 accelerator 还没有消费时：

```text
frame_ready = 1
S_AXIS_TREADY = 0
```

这样最多保留：

```text
1 个正在计算的 frame
1 个完整等待启动的 frame
```

### 5.3 TKEEP 和 TLAST 规则

当前输入张量固定为完整 32-bit beat，因此要求：

```text
每一拍 TKEEP = 4'b1111
TLAST 只允许出现在 beat 434
```

错误类型：

```text
EARLY_TLAST:
  TLAST 在第 435 拍之前出现。

MISSING_TLAST:
  第 435 拍没有 TLAST。

BAD_KEEP:
  某拍 TKEEP 不是 4'b1111。

DROPPED_FRAME:
  当前帧因协议错误被丢弃。
```

出现中间帧错误后，controller 会进入：

```text
drop_until_tlast
```

状态，继续接收并丢弃数据，直到遇到下一个 `TLAST`，然后重新从地址 0 接收新帧，避免错误帧造成后续帧地址永久错位。

## 7. S00_AXI 八寄存器设计

今天重写：

```text
ip_repo/emg_accelerator_1_0/hdl/emg_accelerator_v1_0_S00_AXI.v
```

寄存器地址宽度保持 5 bit，可以覆盖 8 个 32-bit 寄存器：

| Offset | Register | 主要用途 |
|---:|---|---|
| `0x00` | CONTROL | START、AUTO_START、CLEAR_DONE、CLEAR_ERRORS、IRQ_ENABLE |
| `0x04` | STATUS | BUSY、DONE、FRAME_READY、RX_ACTIVE、错误和 irq 状态 |
| `0x08` | RESULT | 最近一次分类结果 class index |
| `0x0C` | FRAME_INFO | 固定帧长度 435 words / 1740 bytes |
| `0x10` | RX_STATUS | 当前 beat count 和成功 frame count |
| `0x14` | ERROR_STATUS | TLAST、TKEEP、丢帧和 start reject 错误 |
| `0x18` | CYCLE_COUNT | 最近一次 inference 周期数 |
| `0x1C` | VERSION | IP 版本号 |

### 7.1 CONTROL

```text
bit 0 START         W1P
bit 1 AUTO_START    R/W
bit 2 CLEAR_DONE    W1P
bit 3 CLEAR_ERRORS  W1P
bit 4 IRQ_ENABLE    R/W
```

其中：

```text
START:
  软件请求启动一次计算。

AUTO_START:
  收到完整 frame 且 accelerator idle 时自动启动。

CLEAR_DONE:
  清除 sticky DONE。

CLEAR_ERRORS:
  清除 AXIS receive error 和 rejected start error。

IRQ_ENABLE:
  允许 DONE sticky 驱动 irq。
```

### 7.2 STATUS 和 sticky 状态

`done` 和 `class_valid` 在 accelerator 中是一拍脉冲，软件不一定能及时读到，因此 AXI-Lite 内增加：

```text
done_sticky
result_class_idx
```

行为：

```text
core_done -> done_sticky = 1
core_class_valid -> 锁存 class_idx
CLEAR_DONE 或新 start_accept -> 清除 done_sticky
```

中断：

```verilog
irq = irq_enable && done_sticky;
```

因此 `irq` 是 level-high sticky interrupt，不是只有一个周期的 pulse。

### 7.3 启动拒绝

手动 START 可能被拒绝：

```text
START_NO_FRAME:
  accelerator idle，但 controller 中没有完整 frame。

START_BUSY:
  accelerator 正在运行。
```

今天调整了优先级：

```text
如果 accelerator busy，优先只报告 START_BUSY，
避免同时出现 START_BUSY 和 START_NO_FRAME。
```

## 8. AXI IP 顶层封装

重写文件：

```text
ip_repo/emg_accelerator_1_0/hdl/emg_accelerator_v1_0.v
```

顶层实例化：

```text

```

核心启动条件：

```verilog
core_start = !core_busy
          && frame_ready
          && (manual_start || auto_start);
```

当 `core_start` 成立时，同时发生：

```text
1. accelerator 开始计算。
2. accelerator 把当前 activation_load_bank 变为 read bank。
3. accelerator 切换新的 load bank。
4. controller 清除 frame_ready。
5. AXI-Stream 可以开始接收下一帧。
```

顶层新增输出：

```text
irq
```

同时增加 inference cycle counter，用于记录最近一次计算持续周期。

### 8.1 复位修正

今天测试时发现一个重要问题：

```text
如果 AXIS controller 在复位期间仍把 TREADY 拉高，
上游可能认为第一拍已经完成握手，
但 controller 内部寄存器仍处于 reset，导致第一拍实际丢失。
```

修正后：

```verilog
S_AXIS_TREADY = rst_n && internal_ready;
```

同时顶层对数据面复位采用：

```text
异步置位复位
同步释放复位
```

使 accelerator 和 frame controller 在 `s00_axis_aclk` 域中一致退出复位。

## 9. 时钟域方案和当前限制

当前第一版 AXI IP 仍采用两个模板时钟端口：

```text
s00_axi_aclk
s00_axis_aclk
```

但控制信号、frame status 和 accelerator 状态目前没有实现完整 CDC，因此要求：

```text
s00_axi_aclk == s00_axis_aclk
```

并且：

```text
s00_axi_aresetn 和 s00_axis_aresetn 应同时 assert/release。
```

推荐 Block Design 第一版连接：

```text
PS FCLK_CLK0
  -> AXI-Lite interconnect
  -> AXI DMA MM2S
  -> emg_accelerator S00_AXI clock
  -> emg_accelerator S00_AXIS clock
```

如果 DMA 输出 stream 位于不同 clock domain，应在 DMA 与本 IP 之间加入：

```text
AXI4-Stream Clock Converter
```

而不是直接给本 IP 的 AXI-Lite 和 AXIS 端口使用两个异步时钟。

后续如果确实需要本 IP 原生支持双时钟，需要增加：

```text
start/clear pulse CDC
auto_start level CDC
done/result/status snapshot CDC
错误和计数器跨域同步
```

## 12. 关于综合 endpoint 数量变化

今天还讨论了“RTL 没有修改，但每次综合后的 endpoint 数量不同”的现象。

当前判断：

```text
1. Vivado 综合、opt 和物理优化会重新做常量传播、逻辑合并、寄存器复制和层次优化。
2. IP output products、综合网表缓存、incremental checkpoint、compile order 或参数刷新状态不同，也可能改变最终综合网表。
3. synth 后报告中的 hold endpoint 数量不等于 route 后真实 failing endpoint 数量。
4. 只比较 endpoint 数量不能直接判断 RTL 功能发生变化。
5. 需要同时固定 Vivado 版本、part、constraint、IP 状态、run strategy、seed 和 incremental 状态，结果才更容易复现。
6. 正式结论应看 routed timing summary 中的 WNS/TNS/WHS/THS 和 failing endpoints。
```

对于当前 AXI IP，后续重新 package 和生成 BRAM output products 后，综合层次和 endpoint 数量发生一次变化是可以预期的，需要重新建立 implementation timing 基线。

## 19. Vivado XSim Library 空值问题

第一次在 Edit Packaged IP 临时工程中运行系统仿真时，XSim 报错：

```text
ERROR: [XSIM 43-3268]
Logical library name should not contain white space, /, \\, = or .

ERROR: [XSIM 43-3217]
Incorrect project file syntax
```

生成的 `.prj` 中出现：

```text
verilog   \
".../emg_accelerator_v1_0.v"
```

正确格式应为：

```text
verilog xil_defaultlib \
".../emg_accelerator_v1_0.v"
```

根因是 Edit Packaged IP 临时 `.xpr` 中 Verilog 文件被设置成：

```text
Library = ""
```

XSim 因此把第一个源文件路径误认为逻辑库名。

当前安全处理方式是在每次打开 Edit Packaged IP 后，对当前临时工程执行：

```tcl
set rtl_files [get_files -of_objects [get_filesets sources_1] \
    -filter {FILE_TYPE == Verilog || FILE_TYPE == SystemVerilog}]

set sim_files [get_files -of_objects [get_filesets sim_1] \
    -filter {FILE_TYPE == Verilog || FILE_TYPE == SystemVerilog}]

set_property LIBRARY xil_defaultlib $rtl_files
set_property LIBRARY xil_defaultlib $sim_files

update_compile_order -fileset sources_1
update_compile_order -fileset sim_1
```

Library 的作用是仿真和编译阶段的逻辑命名空间，不会改变进入 FPGA 的卷积、权重、寄存器或 AXI 硬件逻辑。

影响总结：

```text
XSim 仿真：空 Library 会直接导致编译失败。
纯 Verilog 综合：通常会使用默认库，风险相对较低。
混合语言或存在同名模块：Library 错误可能影响综合和 elaboration。
Implementation/Bitstream：综合成功生成网表后，基本不再受源码 Library 影响。
```

## 24. Edit Packaged IP 与 Merge Changes 的正确理解

```text
修改端口或 AXI interface
修改参数
修改权重路径
修改 Testbench 文件列表
修改 Used In
```

以下操作不需要 Merge：

```text
仅重新打开 Edit Packaged IP
查看源文件
设置当前临时工程 Library
设置 Simulation Top
运行 Behavioral Simulation
查看波形
```

需要注意：

```text
sync_package_sources.tcl 会执行 merge_project_changes 和 save_core，
因此只应在确实修改了 package 内容时运行。

如果只是为了仿真设置 Library，使用独立的 set_property 命令即可，
不要运行完整同步脚本。
```
