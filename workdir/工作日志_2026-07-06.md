## 2. 当前计算核的数据读取契约

### 2.1 共同原则

现在所有计算核都没有直接绑定具体 ROM/BRAM 物理布局。

计算核只发出逻辑请求：

```text
weight_req_*:
  请求某一层、某个 output-channel group、某个 input-channel/tap index 的权重向量。

act_req_* 或 win_req_*:
  请求某一层、某个 time tile、某个 input-channel index 或 window 的激活向量。
```

外部 ROM controller / activation loader / window loader 的责任是：

```text
1. 把计算核发出的逻辑请求映射到实际存储地址。
2. 按计算核要求的 lane 顺序拼成 vector。
3. 对每个 accepted request 恰好返回一次 response。
4. response 必须按 request 顺序返回，不乱序、不丢失、不重复。
```

因此，代码注释中的：

```text
weight_vec = W[oc_base + lane][k]
act_vec    = A[tile_t_base + row][k]
```

应该理解成“逻辑视图”，不是要求物理存储必须长成二维数组。

## 3. 各模块的实际请求模式

### 3.1 stem_conv_array_4x4

请求模式：

```text
weight_req_oc_base = 外部传入的 oc_base
weight_req_k       = 0..34
weight_vec[lane]   = W_stem[oc_base + lane][k_flat]

act_req_t_base = tile_t_base
act_req_k      = 0..34
act_vec[row]   = logical im2col activation for row and k_flat
```

其中 `k_flat` 建议定义为：

```text
k_flat = input_channel * 7 + tap
input_channel = k_flat / 7
tap           = k_flat % 7
```

真实 raw input buffer 是：

```text
input[time][channel]
```

所以 stem activation loader 应该生成：

```text
act_vec[row] =
    input[tile_t_base + row + tap - 3][input_channel]

越界 time 返回 0 padding。
```

因此 stem 的 `act_vec` 不是直接从 `[time][k_flat]` 物理表读取，而是一个 im2col/window-loader 逻辑视图。

推荐 stem weight ROM packed word：

```text
stem_weight_word[oc_group][k_flat]
  lanes 0..3 = W_stem[oc_group*4 + lane][k_flat]

address = oc_group * 35 + k_flat
oc_group = weight_req_oc_base / 4
```

### 3.2 pw_conv1_array_4x8

PW1 已改成 full-output-channel job。

一次 start 的内部请求顺序：

```text
for oc_base in 0,8,16,24,32,40,48,56:
    for k in 0..31:
        weight_req_oc_base = oc_base
        weight_req_k       = k
        act_req_t_base     = tile_t_base
        act_req_k          = k
```

向量语义：

```text
weight_vec[lane] = W_pw1[oc_base + lane][k]
act_vec[row]     = DW1_out[tile_t_base + row][k]
```

这里 `act_vec` 对每个 output-channel group 会重复请求同一个 4-row input tile。

这是可行的，因为周期模型本来就是：

```text
PW1 4 rows = 8 output-channel groups * 32 input channels = 256 MAC feed cycles
```

但这要求 `dw1_to_pw1` row packer 在整个 PW1 job 期间保持这 4 行稳定，不要提前覆盖。

推荐实现：

```text
dw1_to_pw1 row packer:
  1. 等 row_full[tile_t_base..tile_t_base+3] 全为 1。
  2. input_tile_req_ready 拉高。
  3. 把 4 x 32 tile 锁进 pw1_input_tile_regs 或保持 FIFO read-lock。
  4. 对每个 act_req_k 返回:
       act_vec[row] = tile_regs[row][act_req_k]
  5. 直到 PW1 done 或 out_last_group 后再释放这 4 行。
```

推荐 PW1 weight ROM packed word：

```text
pw1_weight_word[oc_group][k]
  lanes 0..7 = W_pw1[oc_group*8 + lane][k]

address = oc_group * 32 + k
oc_group = weight_req_oc_base / 8
```

### 3.3 dw_conv1_4ch_5tap

DW1 不直接请求单点 activation，而是请求 5-tap window：

```text
win_req_out_t
win_req_ch_base

win_vec[tap][lane] =
    stem_out[win_req_out_t + tap - 2][win_req_ch_base + lane]
```

越界 time 返回 0 padding。

需要的级间状态：

```text
stem_to_dw1 buffer[time][channel]
valid_group[time][channel_group]
```

其中：

```text
channel_group = ch_base / 4
```

window loader 必须判断窗口内非 padding 的行和通道组是否已经 valid。

### 3.4 dw_conv2_4ch_5tap

DW2 和 DW1 的 datapath 一样，但输入是 low-resolution pool output：

```text
win_vec[tap][lane] =
    pool_out[win_req_out_t + tap - 2][win_req_ch_base + lane]
```

越界 time 返回 0 padding。

`pool_to_dw2` buffer 需要维护：

```text
buffer[low_time][channel]
valid_group[low_time][channel_group]
```

PW1 一次输出 8 通道，所以 writer 对每个输出 tile 应设置两个 4-channel group valid bit：

```text
valid_group[out_pool_t_base + row][out_oc_base/4]     <= 1
valid_group[out_pool_t_base + row][out_oc_base/4 + 1] <= 1
```

### 3.5 pw_conv2_array_4x12

PW2 是 full-output-channel job。

一次 start 的内部请求顺序：

```text
for oc_base in 0,12,24,36,48,60,72,84:
    for k in 0..63:
        weight_req_oc_base = oc_base
        weight_req_k       = k
        act_req_t_base     = tile_t_base
        act_req_k          = k
        act_req_row_valid_mask = row_valid_mask
```

向量语义：

```text
weight_vec[lane] = W_pw2[oc_base + lane][k]
act_vec[row]     = DW2_out[tile_t_base + row][k]
```

final partial tile：

```text
tile_t_base = 172
row_valid_mask = 4'b0011
```

此时 row 0/1 有效，row 2/3 无效。activation loader 可对无效 row 返回 0；PW2 当前会在模块内部只把 `row_valid_mask` 置 1 的 row 累加到 `gap_sum_vec`，并在最后一个 output-channel group 完成后让 `gap_count += popcount(row_valid_mask)`。

推荐 PW2 weight ROM packed word：

```text
pw2_weight_word[oc_group][k]
  lanes 0..11 = W_pw2[oc_group*12 + lane][k]

address = oc_group * 64 + k
oc_group = weight_req_oc_base / 12
```

## 4. ROM 和 activation 存储组织建议

### 4.1 权重 ROM

建议不要按软件 PyTorch 的原始 tensor layout 直接暴露给计算核。

更稳妥的是导出时预先重排成硬件 packed word：

```text
stem:
  8 groups x 35 words/group x 4 lanes

pw1:
  8 groups x 32 words/group x 8 lanes

pw2:
  8 groups x 64 words/group x 12 lanes

dw1:
  本地 weight_mem[32][5]

dw2:
  本地 weight_mem[64][5]
```

packed lane 规则统一为：

```text
vec[lane*DATA_W +: DATA_W] = logical_value[base + lane]
```

虽然注释里常写成：

```text
{W[base+N], ..., W[base+0]}
```

但实际 Verilog 低位 lane 0、高位 lane N。ROM controller 必须按这个 bit layout 打包。

如果未来 ROM 端口不够宽，可以在 ROM controller 内部多拍读取后再一次返回 `weight_vec_valid`。计算核支持 request/response 解耦，但必须保持 response 顺序。

### 4.2 PW activation tile

PW1/PW2 都是“同一个 input tile 对多个 output-channel groups 重复使用”。

为了避免每个 output-channel group 都从大 row FIFO 反复读，建议：

```text
PW1:
  input_tile_req_ready 后锁存 4 x 32 到 pw1_input_tile_regs。

PW2:
  input_tile_req_ready 后锁存 4 x 64 到 pw2_input_tile_regs。
```

然后计算期间：

```text
act_vec[row] = tile_regs[row][act_req_k]
```

这样大 buffer 只负责把完整 tile 交给局部寄存器，小型 tile regs 负责高频读。

### 4.3 FC classifier

当前 FC tail 已实现为 `fc_classifier_96_5`。

输入来自 PW2：

```text
gap_sum_vec[96]  // 每个元素 48-bit
gap_count        // 正常应为 174
```

模块内部先执行：

```text
gap_avg[ch] = sat_q8_8(gap_sum_vec[ch] / 174)
```

FC 权重本地驻留，bias 作为 augmented matrix 的最后一列：

```text
FC0 weight: fc0_weight_mem[out][k], out=0..95, k=0..96
  k=0..95: data weight
  k=96:    bias

FC1 weight: fc1_weight_mem[class][k], class=0..4, k=0..96
  k=0..95: hidden weight
  k=96:    bias
```

计算调度：

```text
AVG: 96 cycles
FC0: 6 groups * (1 init + 96 mac + 1 store) cycles
FC1: 1 init + 96 mac + 1 store + argmax/output

约 786 cycles，不含 result_ready 反压。
100 MHz 下约 7.9 us。
```

### 4.4 DW window loader

DW window loader 推荐独立模块，不要塞进 DW compute core。

原因：

```text
1. padding 判断属于地址生成和数据可用性问题，不属于 MAC datapath。
2. DW1/DW2 都能复用同一类 window-loader 结构。
3. 单独 TB 可以验证边界、valid bitmap、窗口拼接，不会和 MAC reference 形成共同模式错误。
```

## 5. 当前风险和后续必须补的验证

### 5.1 当前 testbench 只验证了理想 vector provider

现在 stem/PW1/PW2 的 TB 都是 mock provider 根据请求即时生成：

```text
make_weight_vec(...)
make_act_vec(...)
```

这验证了计算核的请求顺序、握手、数值路径和 backpressure，但还没有验证真实 ROM 地址映射和真实级间 buffer 读取。

后续必须新增：

```text
1. stem_weight_rom / pw1_weight_rom / pw2_weight_rom TB
2. input_to_stem im2col activation loader TB
3. dw1_to_pw1 row packer + tile_regs TB
4. pool_to_dw2 window buffer/loader TB
5. dw2_to_pw2 row packer + tile_regs TB
```

### 5.2 full-channel PW job 对 row packer 的要求更强

PW1/PW2 现在一次 start 内部都会连续扫描多个 output-channel groups。

因此 `input_tile_req_ready=1` 的含义必须是：

```text
这个 input tile 不只是当前拍可读，
而是整个 PW job 期间都能稳定读取。
```

否则中间 `oc_base` group 还没算完，上游 buffer 覆盖了这 4 行，会造成后续输出通道错误。

### 5.3 response 顺序不能乱

所有 systolic 计算核目前都假设：

```text
accepted request 顺序 == response 顺序
```

如果未来多个计算核共享一个 ROM controller，controller 可以仲裁和插入等待，但不能对同一个计算核乱序返回。若要支持乱序，需要给 response 增加 metadata 或在核内增加 request FIFO，目前没有这么设计。

### 5.4 stem 的 act_req_k 是 flattened conv index

stem 的 `act_req_k` 不是单纯 input channel。它对应：

```text
5 input channels x 7 taps = 35
```

后续 input loader 必须把 raw `[time][5]` 输入转换成 im2col 逻辑向量。这个点最容易在系统集成时出错。

## 6. 当前结论

当前 RTL 的读取方式是可落地的，但前提是后续存储系统遵守这些逻辑 vector 契约。

最推荐的系统边界是：

```text
compute core:
  只负责 MAC/BN/ReLU/Pool 和 request/response 协议。

ROM controller:
  负责按 layer-specific packed layout 返回 weight_vec。

activation/window loader:
  负责从级间 buffer、valid bitmap、tile regs 中拼 act_vec/win_vec。

top scheduler:
  负责 start 顺序、final_chunk、row_valid_mask、buffer 生命周期。
```

不要让计算核直接依赖物理 BRAM 排列。这样后续即使用 packed ROM、banked BRAM、同步 RAM 或多拍拼包，都不需要改计算核，只需要改 controller/loader。

### 7.4 明天优先设计 stem weight ROM controller

stem weight controller 只负责把计算核的逻辑 weight request 转成
BMG ROM 读请求：

```text
输入:
  weight_req_valid
  weight_req_oc_base
  weight_req_k
  weight_vec_ready

输出:
  weight_req_ready
  weight_vec_valid
  weight_vec[63:0]

BMG:
  blk_mem_gen_stem_weight
  clka
  ena
  addra[8:0]
  douta[63:0]
```

地址公式：

```text
oc_group = weight_req_oc_base / 4
addr     = oc_group * 35 + weight_req_k
```

有效范围：

```text
weight_req_oc_base = 0,4,8,...,28
weight_req_k       = 0..34
addr               = 0..279
```

BMG 当前 read latency = 1，所以 controller 要在 accepted request 后延迟
1 拍产生 `weight_vec_valid`。建议 controller 写成可参数化：

```text
parameter BRAM_READ_LATENCY = 1
```

便于以后 GUI 改成 2 拍时只改参数。

### 7.5 明天同时需要定 stem activation buffer/controller 边界

stem activation 和 weight 一样，返回给 compute core 的接口都是 vector：

```text
act_vec = 4 rows x 16 bit = 64 bit
```

但 activation buffer 不应按每个 tile 的 35 个 im2col 向量重复存。
原因是输入窗口高度重叠。activation buffer 只保存原始输入帧：

```text
X[channel][time]
channel = 0..4
time    = 0..347
```

推荐第一版不做 ping-pong，只做单帧输入缓存：

```text
外部输入写完整 5 x 348 后，再启动 stem。
stem 计算期间只读，不覆盖。
下一帧等当前推理完成后再写入。
```

推荐 activation RAM 物理组织：

```text
blk_mem_gen_stem_activation:
  Simple Dual Port RAM
  Port A: external input write
  Port B: stem activation controller read
  width = 64
  depth = 435
  no init file
```

`435` 的来源：

```text
每个 64-bit word 存同一 channel 的 4 个连续 time 点。
348 / 4 = 87 blocks/channel
5 channels * 87 blocks = 435 words
```

建议布局：

```text
addr = channel * 87 + time_block
time_block = time / 4

word[15:0]   = X[channel][block*4 + 0]
word[31:16]  = X[channel][block*4 + 1]
word[47:32]  = X[channel][block*4 + 2]
word[63:48]  = X[channel][block*4 + 3]
```

stem activation controller/window loader 需要把 compute core 的逻辑请求
转换成原始输入读：

```text
input_channel = act_req_k / 7
tap           = act_req_k % 7
time(row)     = act_req_t_base + row + tap - 3
row           = 0..3
```

职责：

```text
1. 对 time < 0 或 time >= 348 的 row 返回 0 padding。
2. 对正常 time 从 activation RAM 读取。
3. 因为 time(row) 可能不是 4 对齐，必要时读取相邻两个 64-bit block 并拼接。
4. 按计算核约定输出:
   act_vec[row*16 +: 16] = X[input_channel][time(row)]
5. accepted request 和 response 必须一一对应、顺序返回。
```

这也是 stem weight controller 和 stem activation controller 的本质区别：

```text
weight controller:
  参数已按计算顺序预打包，地址映射简单。

activation controller:
  输入只按原始帧保存，需要动态 window/im2col、padding、跨 block 拼接。
```
