## 2. 当前量化方式确认

今天重点重新确认了当前 RTL 对应的量化流程，避免继续把旧 Q8.8 逻辑和新 INT8/Q15 逻辑混在一起理解。

当前量化主流程：

```text
1. 输入 activation: signed INT8
2. 权重 weight: signed INT8
3. MAC:
     acc = sum(input_int8 * weight_int8)
4. 加 bias:
     acc_bias = acc + bias_int32
5. Q15 requant:
     product = acc_bias * multiplier_q15
6. 按通道 right_shift 做 round-half-away-from-zero:
     q = round(product / 2^right_shift)
7. 饱和到 INT8:
     q_sat = clamp(q, -127, 127)
8. 对带 ReLU 的层：
     out = max(q_sat, 0)
```

重要点：

```text
1. 这里不是简单“除以 127”。
2. checkpoint 中的浮点 scale 已经离线折算成 multiplier_q15 和 right_shift。
3. RTL 中运行时只做整数乘法、加法、移位、舍入和饱和。
4. saturate_int8 输出 8-bit signed，但函数内部用 17-bit signed 输入，是为了容纳 round/shift 后的中间整数范围。
5. INT8 饱和范围当前使用 [-127, 127]，不是 [-128, 127]。
```

以 stem 某个输出通道、某个时间点为例：

```text
输入窗口:
  x0, x1, x2, ... 为 signed int8 activation

权重:
  w0, w1, w2, ... 为 signed int8 weight

MAC:
  acc = x0*w0 + x1*w1 + x2*w2 + ...

加 bias:
  acc_bias = acc + bias_int32[oc]

Q15 乘法:
  product = acc_bias * multiplier_q15[oc]

舍入右移:
  abs_product = abs(product)
  rounded_abs = (abs_product + (1 << (right_shift[oc] - 1))) >> right_shift[oc]
  rounded = product < 0 ? -rounded_abs : rounded_abs

饱和 + ReLU:
  q_sat = clamp(rounded, -127, 127)
  stem_out = q_sat < 0 ? 0 : q_sat
```

## 3. AXI-Lite / AXI-Stream / Controller 方案讨论

今天讨论了 stem activation ping-pong buffer 接口过于底层的问题。当前内部接口暴露地址、写使能、写数据等信号，如果直接作为最终外部接口，确实不适合从 DMA 或上游连续高速搬运数据。

关键判断：

```text
1. AXI-Lite 适合控制寄存器、状态读取、启动/完成/中断等低带宽控制面。
2. AXI-Stream 适合 activation 这种连续样本输入，尤其是后续希望从 DMA 直接搬运。
3. 不能只用 AXI-Stream 完成所有事情，因为 start、done、class、busy、error、buffer 状态等仍然需要控制/状态面。
4. 更合理的结构是：
     AXI-Lite: 控制/状态寄存器
     AXI-Stream: activation 数据流入口
     controller/loader: 把 stream 数据解析成内部 ping-pong buffer 写请求
```

推荐结构：

```text
PS/DMA
  -> AXI-Stream
  -> activation_stream_loader/controller
  -> ping-pong activation buffer write port
  -> accelerator core

PS
  -> AXI-Lite
  -> control/status registers
  -> start/done/class/irq/config
```

因此后续 IP 封装不建议把 ping-pong buffer 的底层地址接口直接暴露给 block design 外部，而应该由 loader/controller 封装成更高层的数据流接口。

## 5. Hold / Setup 问题分析

今天围绕多个 screenshot 中的 hold 问题做了分析，重点区分综合阶段报告和 implementation 后真实时序。

关键结论：

```text
1. hold 是最小延迟问题，常见于 FF->FF、FF->BRAM 这类极短路径。
2. 拆流水线会改善 setup，但也会增加寄存器边界，让原本被组合逻辑自然延迟遮住的 hold 短路径暴露出来。
3. synth 后 hold endpoints 数量很容易被高估，因为此时还没有真实布局布线延迟。
4. route_design 阶段 router 会通过布线、delay/skew optimization、post hold fix 等步骤修正大量 hold。
5. 因此工程上应优先看 routed timing summary，而不是只根据 synth 后 hold endpoints 盲目改 RTL。
```

Q8.8 时期已经存在的短路径类型：

```text
1. act_resp_count 自反馈。
2. load_capture_count 自反馈。
3. bram_a_idx_d / bram_b_idx_d 这类 BRAM read latency shift。
4. DW fifo_vec / fifo_t 移位。
5. DW 输出寄存器到 DW-to-PW buffer BRAM DI。
```

INT8/Q15 后新增或放大的短路径类型：

```text
1. Q15 后处理 stage 之间的 product/shift/valid metadata 对齐寄存器。
2. valid/t/ch_base/vec_c 等 metadata 伴随数据路径多打一拍。
3. round/saturate 与 FIFO/BRAM 写入边界重新划分后产生的新短路径。
```

今天的一个重要判断：

```text
删除 wire 临时变量通常不会删除硬件。
只有删除或合并不必要的寄存器级，才可能减少 hold endpoint。
```

```text

3. DW1/DW2 不再保留独立的 s4 寄存器 stage。
```
