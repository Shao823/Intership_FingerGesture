## 2. AXI-Lite AW/W 独立握手修复

### 2.1 昨日问题

昨天的 AXI-Lite slave 写通道要求：

```verilog
S_AXI_AWVALID && S_AXI_WVALID
```

在同一周期成立后才接收写事务。

这不能严格支持 AXI4-Lite 允许的以下情况：

```text
1. AW 地址先到，W 数据后到。
2. W 数据先到，AW 地址后到。
3. AW 和 W 同拍到达。
4. BREADY 延迟，写响应暂时不能被主机接收。
```

PS AXI master 通常可能让 AW 和 W 接近同时到达，但 slave 不应依赖这一行为。

### 2.2 当前实现

当前增加独立的 hold register：

```text
aw_hold_valid
aw_hold_addr

w_hold_valid
w_hold_data
w_hold_strb
```

READY 条件分别为：

```verilog
S_AXI_AWREADY = !aw_hold_valid && !axi_bvalid;
S_AXI_WREADY  = !w_hold_valid  && !axi_bvalid;
```

独立握手：

```verilog
aw_handshake = S_AXI_AWREADY && S_AXI_AWVALID;
w_handshake  = S_AXI_WREADY  && S_AXI_WVALID;
```

只有地址和数据均已锁存，且当前没有未完成写响应时，才提交寄存器写：

```verilog
slv_reg_wren = aw_hold_valid && w_hold_valid && !axi_bvalid;
```

完成写事务后：

```text
1. 清除 aw_hold_valid。
2. 清除 w_hold_valid。
3. 产生 BVALID。
4. 等待 BREADY 后结束响应。
```

### 2.3 当前模型边界

当前仍是单 outstanding transaction 模型：

```text
最多锁存一个AW。
最多锁存一个W。
最多保留一个B响应。
```

这已经满足当前寄存器控制需求，不需要增加多事务 FIFO。

### 2.4 定向验证

AXI-Lite testbench 已增加或更新以下场景：

```text
1. AW先到、W后到。
2. W先到、AW后到。
3. AW/W同拍到达。
4. BREADY延迟。
5. CONFIG读写。
6. COMMAND写不能改变CONFIG。
7. CLEAR_DONE和CLEAR_ERRORS脉冲语义。
8. RESULT、STATUS、ERROR_STATUS和VERSION读取。
```

AXI shell 定向测试今天重新运行通过：

```text
PASS tb_emg_accelerator_axi_shell
```

## 3. COMMAND 与 CONFIG 分离

### 3.1 昨日问题

原 CONTROL 寄存器同时包含：

```text
瞬时命令：START、CLEAR_DONE、CLEAR_ERRORS
持久配置：AUTO_START、IRQ_ENABLE
```

因此软件只写 START 时，可能同时覆盖 AUTO_START 和 IRQ_ENABLE。

这会强制软件每次执行 read-modify-write，也容易出现控制常量写错。

### 3.2 当前寄存器语义

当前将寄存器拆分为：

```text
0x00 COMMAND
0x20 CONFIG
```

COMMAND：

| Bit | Name | Access | 说明 |
| ---: | --- | --- | --- |
| 0 | START | W1P | 完整帧已准备且core空闲时启动推理 |
| 1 | CLEAR_DONE | W1P | 清除sticky DONE并撤销完成IRQ |
| 2 | CLEAR_ERRORS | W1P | 清除接收错误和启动拒绝错误 |

CONFIG：

| Bit | Name | Access | 说明 |
| ---: | --- | --- | --- |
| 0 | AUTO_START | R/W | 完整帧准备好且core空闲时自动启动 |
| 1 | IRQ_ENABLE | R/W | DONE sticky置位时允许irq输出 |

当前软件常量建议：

```c
#define EMG_CMD_START          0x00000001U
#define EMG_CMD_CLEAR_DONE     0x00000002U
#define EMG_CMD_CLEAR_ERRORS   0x00000004U

#define EMG_CFG_AUTO_START     0x00000001U
#define EMG_CFG_IRQ_ENABLE     0x00000002U
```

命令可以按位组合，例如：

```c
EMG_CMD_CLEAR_DONE | EMG_CMD_CLEAR_ERRORS
```

但不能再使用昨天旧版 CONTROL 的 `0x13`、`0x19` 等常量。

### 3.3 地址宽度和版本

因为增加了 `0x20 CONFIG`，当前 AXI-Lite 地址宽度为：

```text
C_S00_AXI_ADDR_WIDTH = 6
```

当前 IP version：

```text
0x0001_0001
major = 1
minor = 1
```

软件启动时应读取 VERSION，并拒绝使用旧版寄存器布局的驱动。

## 4. 当前完整寄存器表

主工程 accelerator AXI-Lite 基地址：

```text
0x8001_0000
```

| Offset | Name | 主要用途 |
| ---: | --- | --- |
| `0x00` | COMMAND | START、CLEAR_DONE、CLEAR_ERRORS |
| `0x04` | STATUS | busy、done、frame ready、rx active、error等 |
| `0x08` | RESULT | 最近一次分类结果 |
| `0x0C` | FRAME_INFO | 435 words、1740 bytes |
| `0x10` | RX_STATUS | beat count和frame count |
| `0x14` | ERROR_STATUS | TLAST、TKEEP、drop和start错误 |
| `0x18` | CYCLE_COUNT | 最近一次推理周期 |
| `0x1C` | VERSION | IP版本 |
| `0x20` | CONFIG | AUTO_START、IRQ_ENABLE |

软件不应硬编码 FRAME_WORDS 和 FRAME_BYTES 后完全忽略 FRAME_INFO。建议初始化时检查：

```text
FRAME_INFO[15:0]  == 435
FRAME_INFO[31:16] == 1740
```

如果不匹配，应停止 DMA，避免软件数据格式和硬件版本不一致。

## 6. DDR 到 DMA 到 Accelerator 的真实数据过程

### 6.1 控制路径

DDR 不会主动通知 DMA。

第一版 bare-metal 软件负责：

```text
1. 将输入数据写入DDR。
2. 执行cache flush。
3. 向AXI DMA写MM2S源地址。
4. 向AXI DMA写传输长度1740 bytes。
5. DMA从DDR读取并生成AXI-Stream。
```

主工程控制路径：

```text
PS M_AXI_HPM0_LPD
  -> AXI Interconnect
     -> AXI DMA S_AXI_LITE
     -> Accelerator S00_AXI
```

DMA AXI-Lite 基地址：

```text
0x8000_0000
```

### 6.2 数据路径

```text
DDR
  -> PS S_AXI_HP0_FPD
  -> SmartConnect
  -> AXI DMA M_AXI_MM2S
  -> AXI DMA M_AXIS_MM2S
  -> Accelerator S00_AXIS
  -> activation ping-pong BRAM
```

每帧格式：

```text
5 channels × 348 samples × INT8
= 1740 bytes
= 435个32-bit AXIS beat
```

每个 beat：

```text
TDATA[7:0]   = sample[t+0]
TDATA[15:8]  = sample[t+1]
TDATA[23:16] = sample[t+2]
TDATA[31:24] = sample[t+3]
TKEEP        = 4'b1111
```

最后一个 beat：

```text
beat index = 434
TLAST      = 1
```

### 6.3 435不是神经网络意义上的435行

当前输入为 channel-major：

```text
channel 0: beat 0   ~ 86
channel 1: beat 87  ~ 173
channel 2: beat 174 ~ 260
channel 3: beat 261 ~ 347
channel 4: beat 348 ~ 434
```

每个 channel：

```text
348 samples / 4 samples per beat = 87 beats
```

因此软件必须按以下内存布局准备 DDR 数据：

```text
ch0[t0..t347]
ch1[t0..t347]
ch2[t0..t347]
ch3[t0..t347]
ch4[t0..t347]
```

不能直接使用 time-major：

```text
t0[ch0..ch4]
t1[ch0..ch4]
...
```

否则 DMA 和 AXIS 传输本身可能完全正常，但神经网络结果会错误。

## 7. 当前 Ping-Pong Activation RAM 的真实行为

### 7.1 双 bank 确实存在

`stem_activation_controller_with_ram` 中实例化两个 activation BRAM：

```text
activation bank 0
activation bank 1
```

运行时维护：

```text
activation_load_bank
activation_read_bank
```

DMA 始终向 load bank 写入，Stem 始终从 read bank 读取。

### 7.2 第一帧

复位后第一帧通常写入 Bank0：

```text
DDR case0
  -> DMA
  -> 435个AXIS beat
  -> Bank0
```

数据不是先完整进入某个 DMA buffer，再复制到 Bank0。

每次 AXIS 握手时直接写：

```text
act_wr_en   = 1
act_wr_addr = beat_count
act_wr_data = TDATA
```

### 7.3 同一帧不会边接收边计算

只有收到完整435 beats，并且最后一个 beat 的 TLAST 正确时，才产生：

```text
frame_ready = 1
```

推理启动条件：

```text
core_busy == 0
frame_ready == 1
START == 1 或 AUTO_START == 1
```

所以当前不是同一帧 streaming compute：

```text
错误理解：收到前几个beat后Stem立即计算同一帧。

当前实现：整帧写入一个activation bank后，Stem才开始读取该帧。
```

### 7.4 启动时切换 bank

第一帧启动时：

```verilog
activation_read_bank <= activation_load_bank;
activation_load_bank <= !activation_load_bank;
```

结果：

```text
Bank0：第0帧read bank，供accelerator计算
Bank1：新的load bank，供DMA写第1帧
```

### 7.5 帧间搬运与计算可以重叠

正确时序：

```text
Bank0计算case0的同时，DMA向Bank1写case1。
Bank1计算case1的同时，DMA向Bank0写case2。
```

即：

```text
同一帧：先完整接收，再计算。
相邻帧：前一帧计算与后一帧DMA接收重叠。
```

### 7.6 写冲突保护

当 DMA 写 bank 与 accelerator 当前读 bank 相同时：

```text
act_wr_ready = 0
S_AXIS_TREADY = 0
```

DMA 会受到 AXI-Stream backpressure，不会覆盖正在计算的输入。

当读写 bank 不同时：

```text
act_wr_ready = 1
```

允许 DMA 和 accelerator 并行访问两个独立 BRAM bank。

### 7.7 当前只能保存两帧状态

最多同时存在：

```text
1帧正在计算
+
1帧正在接收或已经接收完成
```

如果：

```text
Bank0正在计算case0
Bank1已经完整保存case1
```

则：

```text
frame_ready = 1
S_AXIS_TREADY = 0
```

此时不能继续发送 case2。

必须等 case0 完成、case1 启动、Bank0 被释放为新的 load bank 后，case2 才能进入。

## 8. 软件是否真正发挥 Ping-Pong 的关键

### 8.1 硬件不会自动启动下一次 DMA

AUTO_START 只控制 accelerator：

```text
完整帧准备好且core空闲时自动开始计算。
```

AUTO_START 不会自动配置：

```text
下一帧DDR地址
DMA MM2S源地址
DMA传输长度
DMA启动
```

当前 Simple DMA 用法下，软件仍需逐帧配置 DMA。

### 8.2 不推荐的软件流程

如果软件这样运行：

```text
DMA case0
等待DMA完成
启动case0
等待case0计算完成
DMA case1
等待DMA完成
启动case1
```

则实际时序为：

```text
DMA0 -> Compute0 -> DMA1 -> Compute1
```

虽然硬件中存在两个 BRAM bank，但 DMA 和计算没有重叠，ping-pong 不产生吞吐收益。

### 8.3 推荐的软件流程

建议第一版连续10帧测试使用：

```text
CONFIG.AUTO_START = 1
CONFIG.IRQ_ENABLE = 1
```

运行顺序：

```text
1. 启动DMA发送case0。
2. case0接收完整后，accelerator自动启动case0。
3. 收到DMA0完成中断后，立即启动DMA发送case1。
4. case0计算时，case1写入另一个bank。
5. case1收满后等待case0计算结束。
6. case0完成，case1自动启动。
7. 收到accelerator0完成中断后读取case0结果并清DONE。
8. Bank0释放后，软件启动DMA发送case2。
9. 重复直到case9。
```

更直观的 bank 时序：

```text
                   Bank0               Bank1
----------------------------------------------------
阶段1              接收case0            空
阶段2              计算case0            接收case1
阶段3              接收case2            计算case1
阶段4              计算case2            接收case3
阶段5              接收case4            计算case3
...
```

### 8.4 何时启动下一次DMA

不能仅凭“上一笔 DMA 已完成”就无限连续提交后续帧，因为 accelerator 只提供两个 bank。

软件应维护：

```text
next_dma_case
next_result_case
frames_submitted
frames_completed
```

安全启动下一次 DMA 的条件可以是：

```text
1. 当前没有DMA传输进行中。
2. Accelerator STATUS.ACT_WRITE_READY == 1。
3. Accelerator STATUS.FRAME_READY == 0。
4. 仍有未提交输入帧。
```

其中 `FRAME_READY==0` 表示当前 load bank 还没有一帧完整等待；`ACT_WRITE_READY==1` 表示写 bank 没有与正在读取的 bank 冲突。

对于当前固定时序，也可以在每次 accelerator 启动新一帧后，立即启动下一帧 DMA。

### 8.5 DMA比计算快时的正常现象

当前单帧完整计算：

```text
40527 cycles @ 100 MHz
约405.27 us
```

DMA只需要传输：

```text
435个32-bit beat
```

因此第二帧通常会在第一帧计算结束前很早就写满另一个 bank。

之后出现：

```text
frame_ready = 1
S_AXIS_TREADY = 0
```

是正常的等待状态，不是 DMA 或 accelerator 死锁。

## 10. DDR 数据与 Cache 注意事项

### 10.1 HP0不是cache coherent路径

CPU写入DDR后，数据可能仍停留在CPU cache中。

DMA读取之前必须：

```c
Xil_DCacheFlushRange((UINTPTR)buffer, length);
```

否则可能出现：

```text
CPU看到新输入
DMA却从DDR读取旧输入
```

这种错误往往表现为：

```text
DMA完成
accelerator完成
没有ERROR_STATUS
但是分类结果错误
```

### 10.2 Flush范围

建议：

```text
1. buffer地址按cache line对齐。
2. flush长度覆盖完整1740 bytes。
3. 为避免尾部cache line遗漏，可将长度向上对齐到cache line。
4. 不要在DMA读取期间修改同一输入buffer。
```

### 10.3 Memory barrier

在以下操作之间建议使用平台提供的 barrier：

```text
写DDR输入
-> cache flush
-> memory barrier
-> 写DMA源地址和长度
```

避免CPU或总线写顺序导致DMA过早启动。

## 11. DMA 软件侧注意事项

### 11.1 固定长度

每次 MM2S 必须严格设置：

```text
1740 bytes
```

长度错误可能导致：

```text
小于1740：EARLY_TLAST
大于1740：MISSING_TLAST、DROPPED_FRAME或后续stream被drain
```

DMA通常根据 BTT 产生最后一拍 TLAST，因此 BTT 是帧协议的一部分，不只是性能参数。

### 11.2 地址合法性

每帧源地址必须：

```text
位于DMA可访问DDR区域
满足DMA地址宽度要求
在传输期间不被CPU覆盖
```

如果10帧连续存储：

```c
frame_addr[i] = base + i * 1740;
```

为了cache line和未来突发访问方便，也可以给每帧设置更大的对齐stride，但 DMA BTT 仍必须为1740。

### 11.3 DMA完成不等于推理完成

两个中断语义不同：

```text
DMA IRQ：1740 bytes已经送入AXI-Stream并完成该次MM2S事务。
Accelerator IRQ：神经网络已经完成分类，DONE sticky有效。
```

软件不能在 DMA IRQ 中直接认为 RESULT 已经更新。

推荐：

```text
DMA IRQ：标记dma_idle，并尝试调度下一帧。
Accelerator IRQ：读取RESULT/CYCLE_COUNT，记录结果并CLEAR_DONE。
```

### 11.4 DMA错误恢复

如果DMA状态寄存器出现内部错误、slave error或decode error：

```text
1. 停止提交新帧。
2. 记录DMA状态寄存器。
3. 复位DMA。
4. 等待DMA reset完成和idle。
5. 清accelerator ERROR_STATUS。
6. 必要时重新初始化整个数据面。
```

不要只重复写 BTT 而忽略 DMA error 状态。

## 12. Accelerator 软件侧注意事项

### 12.1 START前检查

手动 START 模式下建议先确认：

```text
STATUS.BUSY        == 0
STATUS.FRAME_READY == 1
```

否则会产生：

```text
START_NO_FRAME
START_BUSY
```

这些错误为sticky，必须通过 `COMMAND.CLEAR_ERRORS` 清除。

### 12.2 AUTO_START模式

连续多帧测试推荐 AUTO_START，因为它能减少软件对“最后一个 AXIS beat 到达”和“写START”的精确时序依赖。

但 AUTO_START 不表示：

```text
自动配置DMA
自动修改DDR地址
自动清DONE
自动保存每一帧RESULT
```

软件仍需及时读取并保存结果。

### 12.3 RESULT覆盖

RESULT 保存最近一次完成结果。

如果软件长时间不处理 accelerator IRQ，而下一帧又完成，旧 RESULT 可能被新 RESULT 覆盖。

因此 ISR 或主循环必须保证：

```text
每次DONE都读取RESULT并保存到软件数组。
```

当前单帧约405 us，软件有足够时间，但不能完全忽略中断。

### 12.4 DONE和IRQ

当前：

```text
irq = irq_enable && done_sticky
```

读取 RESULT 不会自动清 DONE。

软件处理完成中断后必须写：

```c
EMG_CMD_CLEAR_DONE
```

否则 irq 会持续保持有效。

### 12.5 错误状态

如果 STATUS.ERROR 为1，应读取 ERROR_STATUS：

| Bit | 含义 |
| ---: | --- |
| 0 | EARLY_TLAST |
| 1 | MISSING_TLAST |
| 2 | BAD_KEEP |
| 3 | DROPPED_FRAME |
| 4 | START_NO_FRAME |
| 5 | START_BUSY |

记录错误后写：

```c
EMG_CMD_CLEAR_ERRORS
```

如果是 AXIS 帧格式错误，必须同时确认当前 DMA 帧已经结束或 DMA 已复位，避免下一帧从错误边界继续。

## 13. 推荐的10帧软件状态机

建议不要在多个 ISR 中直接进行复杂阻塞操作，而是使用事件标志加主循环状态机。

软件状态：

```c
uint32_t next_dma_case;
uint32_t completed_cases;
bool dma_busy;
bool dma_done_event;
bool accelerator_done_event;
uint8_t results[10];
uint32_t cycles[10];
```

初始化：

```text
next_dma_case  = 0
completed_cases = 0
dma_busy = false
```

推荐伪代码：

```c
accelerator_write(COMMAND,
    EMG_CMD_CLEAR_DONE | EMG_CMD_CLEAR_ERRORS);
accelerator_write(CONFIG,
    EMG_CFG_AUTO_START | EMG_CFG_IRQ_ENABLE);

submit_dma_frame(0);
next_dma_case = 1;

while (completed_cases < 10) {
    if (accelerator_done_event) {
        accelerator_done_event = false;

        results[completed_cases] = accelerator_read(RESULT) & 0x7;
        cycles[completed_cases] = accelerator_read(CYCLE_COUNT);
        completed_cases++;

        accelerator_write(COMMAND, EMG_CMD_CLEAR_DONE);
    }

    if (dma_done_event) {
        dma_done_event = false;
        dma_busy = false;
    }

    status = accelerator_read(STATUS);

    if (!dma_busy
        && next_dma_case < 10
        && !(status & STATUS_FRAME_READY)
        && (status & STATUS_ACT_WRITE_READY)) {
        submit_dma_frame(next_dma_case);
        next_dma_case++;
    }

    if (status & STATUS_ERROR) {
        handle_accelerator_error();
    }
}
```

实际代码中需要：

```text
1. 对共享事件变量使用volatile或合适的同步原语。
2. 在ISR中先清GIC/DMA中断源。
3. 避免ISR和主循环同时提交DMA。
4. 为每次DMA提交记录对应case index。
5. 为每次accelerator完成维护严格的结果顺序。
```

当前 accelerator 和 DMA 都按顺序运行，因此结果顺序与输入帧顺序一致，不需要 frame id，但软件计数不能丢中断。

## 18. 上板前推荐最小测试顺序

为降低一次引入多帧、中断和ping-pong调度的定位难度，建议分阶段：

### 阶段1：单帧轮询

```text
1. IRQ_ENABLE=0，AUTO_START=0。
2. DMA发送case0。
3. 轮询FRAME_READY。
4. 写COMMAND.START。
5. 轮询DONE。
6. 检查RESULT=3。
7. 检查CYCLE_COUNT约40527。
8. 检查ERROR_STATUS=0。
```

### 阶段2：单帧中断

```text
1. IRQ_ENABLE=1。
2. 分别验证DMA IRQ和accelerator IRQ。
3. ISR中只记录事件，主循环读取寄存器。
4. 确认CLEAR_DONE后accelerator IRQ撤销。
```

### 阶段3：两帧ping-pong

```text
1. AUTO_START=1。
2. DMA发送case0。
3. case0开始计算后立即发送case1。
4. ILA确认case0 busy期间DMA IRQ再次出现。
5. 确认case1在case0完成后自动启动。
6. 检查两个分类结果顺序正确。
```

### 阶段4：十帧连续运行

```text
1. 使用软件状态机逐帧提交DMA。
2. 保存10个RESULT和CYCLE_COUNT。
3. 对比10组golden class。
4. 检查frame_count=10。
5. 检查所有ERROR_STATUS为0。
```

不要直接从零开始运行十帧复杂中断流程，否则出现问题时难以区分：

```text
cache
DMA配置
AXIS帧边界
accelerator启动
ping-pong调度
结果保存
中断清除
```

## 21. 今日结论

今天完成了 AXI-Lite 控制接口的关键健壮性修复，并消除了命令位覆盖持久配置位的问题。当前 packaged IP、主工程 BD snapshot、寄存器文档和定向仿真已经保持一致，昨天提出的 AW/W 耦合和 CONTROL 覆盖风险均已落实解决。

当前 activation ping-pong 不是概念设计，而是由两个独立 BRAM bank、load/read bank切换和写冲突反压共同实现的真实硬件结构。它实现的是“第N帧计算期间接收第N+1帧”，而不是“同一帧尚未接收完整时就开始计算”。

硬件只能提供搬运与计算重叠的能力，是否真正发挥该能力取决于软件。软件必须在当前帧计算期间及时提交下一帧DMA，同时正确管理cache、DMA完成、accelerator完成、FRAME_READY、ACT_WRITE_READY、DONE sticky、错误状态和结果顺序。

下一阶段应先生成bitstream并完成单帧板级闭环，再逐步推进中断、双帧ping-pong和十帧连续运行，避免一次叠加过多软件状态导致定位困难。
