## 2. Stem weight ROM controller

### 2.1 ROM 数据组织

当前 stem weight IP：

```text
IP name: blk_mem_gen_stem_weight
Memory type: Single_Port_ROM
Width: 64 bit
Depth: 280
Address width: 9
READ_LATENCY_A: 1
ENA: enabled
Output register of memory primitives: true
Output register of memory core: false
Resource estimate: 1 BRAM36
```

权重 packed 格式：

```text
word width = 4 lanes * 16 bit = 64 bit
depth      = 35 * (32 / 4) = 280

addr = (oc_base / 4) * 35 + k

word[lane*16 +: 16] = W_stem[oc_base + lane][k]
lane 0 in LSB
lane 3 in MSB
```

已经确认：

```text
weight_data/stem/weight_packed.coe
weight_data/stem/weight_packed.mem
```

展开后都是 280 个 word，内容一致。

### 2.2 真实 BMG IP 时序问题

一开始用 mock ROM 测试时，controller 只按“1 拍后 rom_dout 可采样”理解，mock 仿真可以通过。

后来接入真实 `blk_mem_gen_stem_weight` 仿真模型后发现数据整体错位：

```text
第一笔读出 0
后续 response 比 expected 慢/错一位
背压或最后一笔时更明显
```

原因：

```text
BMG 的 READ_LATENCY_A=1 表示 douta 在下一拍时钟边沿后更新。
controller 不能在同一个边沿采到刚更新的数据。
同时 ENA 不只是“发起读请求”的脉冲，它还影响输出寄存器/读流水推进。
```

最终修正：

```text
rom_valid_d0 <= req_fire
rom_valid_d1 <= rom_valid_d0

fifo_push = rom_valid_d1

rom_en = req_fire || rom_valid_d0
```

也就是说：

```text
真实请求 accepted 后，rom_en 额外保持 1 拍，用来推进 BMG 输出寄存器。
数据在 rom_valid_d1 时采入 response FIFO。
```

controller 内部使用 4-entry response FIFO，保证：

```text
1. 正常无背压时仍能连续每拍接受一个 weight request。
2. 下游 weight_vec_ready 拉低时，不丢失 in-flight ROM response。
3. request_ready 会预留 rom_valid_d0 / rom_valid_d1 两拍在途数据的 FIFO 空间。
```

### 4.2 Top TB

`tb_stem_conv_array_4x4.v` 是顶层集成测试。

它使用：

```text
activation input/cases:
  emg_gesture_vivado/src/stem_conv_array_4x4/testdata/raw_input.mem
  emg_gesture_vivado/src/stem_conv_array_4x4/testdata/case_oc_base.mem
  emg_gesture_vivado/src/stem_conv_array_4x4/testdata/case_t_base.mem

weight and folded BN:
  weight_data/stem/weight_packed.mem
  weight_data/stem/bn_scale.mem
  weight_data/stem/bn_bias.mem

golden output:
  emg_gesture_vivado/src/stem_conv_array_4x4/testdata/expected_top_checkpoint.mem
```

原因：

```text
top TB 的目标是验证真实 checkpoint 权重经 blk_mem_gen_stem_weight IP 进入 compute。
所以 top TB 不应该使用 testdata/weight.mem。
expected_top_checkpoint.mem 是由 gen_testdata.py 根据 raw_input.mem 和
weight_data/stem checkpoint 权重/BN 参数生成的固定标准答案。
```

为了避免后续维护误解，今天已经在：

```text

tb_stem_conv_array_4x4_compute.v
gen_testdata.py
```

中加注释说明两套测试数据的用途。

## 8. 当前仍未处理/后续计划

### 8.1 Stem activation 侧

下一步需要设计：

```text
stem_activation_controller_with_rom
```

或 activation buffer/window controller。

其职责：

```text
1. 接收 compute 的 act_req_t_base / act_req_k。
2. 将 k_flat 拆成:
     input_channel = k_flat / 7
     tap           = k_flat % 7
3. 读取 raw input[time][channel]。
4. 对越界 time 做 0 padding。
5. 拼成:
     act_vec[row] = input[tile_t_base + row + tap - 3][input_channel]
6. 按 request 顺序返回 act_vec。
```
