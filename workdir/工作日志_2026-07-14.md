## 2. AXI-Lite IP 封装讨论

今天先讨论了 `accelerator` 顶层如何包装成 AXI 总线通信的 IP 核。

结论：

```text
1. Vivado GUI 生成 AXI-Lite IP 后，不需要自己从零手写完整 AXI-Lite slave 协议。
2. GUI 生成的 S00_AXI.v 已经包含 AW/W/B/AR/R 通道握手和 slv_reg 读写框架。
3. 后续真正要做的是给 slv_reg0..7 定义业务含义，并把这些寄存器转换成 accelerator 的 start、activation write、status/class 读取等信号。
4. 如果 accelerator 只通过寄存器控制和少量状态读回，那么 AXI-Lite 足够。
5. 如果后续 activation 输入数据量更大、更像连续流，推荐增加一个独立 AXI-Stream activation loader IP。
```

今天明确的 AXI-Lite 寄存器分配：

```text
slv_reg0  CTRL
  bit0 写 1 -> 产生 accel_start 一拍
  bit1 写 1 -> 清 done/class/error sticky

slv_reg1  STATUS
  读时动态返回 busy/done/class_valid/act_wr_ready/error 等状态

slv_reg2  CLASS
  读出 accelerator 最终 class_idx

slv_reg3  ACT_ADDR
  CPU/上游写 activation RAM 地址

slv_reg4  ACT_DATA_LO
  64-bit activation word 低 32 位，保留给较宽数据接口或兼容旧接口

slv_reg5  ACT_DATA_HI
  写这个寄存器时提交一次 activation write

slv_reg6  IRQ_ENABLE
  暂时用于控制/暴露 irq_pending 语义

slv_reg7  VERSION
  版本号/只读标识
```

需要特别注意：

```text
CPU 写 ACT_ADDR/ACT_DATA 的意思是 PS 或外部上游把输入样本写进 accelerator 内部 activation RAM。
这些 activation 输入不是 accelerator 从自己内部“获得”的，而是系统侧加载给 accelerator 的。
```

## 3. S00_AXI 与 accelerator 实例化位置

今天讨论过 `accelerator` 应该实例化在最外层 IP wrapper，还是实例化在 `S00_AXI.v` 中。

最终理解：

```text
1. 如果 S00_AXI 负责 slv_reg 与 accelerator 控制/状态信号之间的映射，那么 accelerator 放在 S00_AXI 中最直接。
2. 如果最外层 wrapper 负责连线，S00_AXI 只吐出业务信号，也可以把 accelerator 放在最外层。
3. 当前工程倾向于让 S00_AXI 直接管理寄存器语义，因此把 accelerator 实例化放进 S00_AXI 更容易理解。
4. 最外层 emg_accelerator_axi_v1_0.v 主要保留 AXI 端口和少量必要的外部 output，例如 irq 或 debug/status。
```

也明确了一个 Verilog 设计规则：

```text
同一个寄存器最好只在一个 always 时序块中赋值。
GUI 默认生成的 slv_reg0..7 写逻辑只是裸寄存器模板，没有业务意义。
业务状态如果需要动态覆盖读值，推荐用 read mux 返回 status/class，而不是在多个 always 中反复写同一个 slv_reg。
```

## 4. BRAM IP 打包和路径问题

今天在 Vivado IP packager 中重点分析了 BRAM IP 和预加载权重文件的打包问题。

遇到/解释过的问题：

```text
[IP_Flow 19-530]
  Component file '../../../weight_data/...' does not exist.

[IP_Flow 19-5905]
  All packaged files should be located below the IP definition file (xml)
```

含义：

```text
1. IP packager 希望被打包进 IP 的文件位于 component.xml 所在 IP 目录下面。
2. 如果 COE/MEM 通过 ../../../weight_data 这种路径引用 IP 目录外部文件，Vivado 会警告。
3. 这种路径在当前工程目录里可能能找到，但打包给别的工程/机器后容易失效。
```

今日结论：

```text
1. 主工程自身使用的 BRAM IP 与 weight_data 可以先保持主工程路径正确。
2. 真正用于 IP repo 发布的版本，最好把 weight_data 拷贝到 ip_repo/emg_accelerator_axi_1_0/src/weight_data 下。
3. IP repo 内部的 XCI/COE 路径应指向 IP 目录内部文件，避免 packaged file warning。
4. 这部分用户计划明天继续手动整理。
```

## 6. 新 INT8 量化模型迁移

今天开始以 `lightcnn1d_int8_s001_internal` 作为真实量化来源，替换旧的 16-bit lane 假设。

真实量化策略：

```text
1. activation 使用 signed INT8，zero-point = 0。
2. weight 使用 signed INT8，主要是 per-output-channel scale。
3. bias 使用 INT32。
4. 中间 MAC accumulator 保留宽位宽，目前 RTL 中主要是 ACC_W=48。
5. requantization 使用 16-bit Q15 multiplier + 每通道 right_shift。
6. 输出饱和到 INT8，当前饱和范围是 [-127, 127]。
7. ReLU 层在 requant 后再将负数截到 0。
```

RTL 中的核心 requant 公式：

```text
acc_bias = acc + bias_int32
product  = acc_bias * multiplier_q15
q        = round_half_away_from_zero(product, right_shift)
out      = saturate_int8(q)
```

舍入方式：

```text
1. 对 product 取绝对值。
2. 加上 1 << (right_shift - 1)。
3. 右移 right_shift。
4. 恢复原符号。
```

注意：

```text
right_shift.mem 是十六进制文本。
例如 0017 表示十进制 23，不是十进制 17。
```

当前各层 right_shift 大致范围：

```text
stem: 22-24
dw1 : 22-23
pw1 : 22-23
dw2 : 22-23
pw2 : 22-23
gap : 19
fc0 : 24-26
fc1 : 24
```

## 9. RTL 位宽修改

今天将主要 RTL 从旧 16-bit lane 改成真实 8-bit lane。

核心原则：

```text
DATA_W = 8
MULT_W = 16
bias_int32 = 32 bit
ACC_W/GAP_ACC_W 保留宽位宽
```

重点修改范围：

```text
stem_conv_array_4x4_compute.v
pw_conv1_array_4x8_compute.v
pw_conv2_array_4x12_compute.v
dw_conv1_4ch_5tap.v
dw_conv2_4ch_5tap.v
fc_classifier_96_5_compute.v

stem/pw/fc wrapper
weight_controller
weight_controller_with_rom
activation/interstage buffer
top/accelerator
相关 testbench
```

关键修正：

```text
1. DATA_W 只表示 activation/weight/logit lane 宽度。
2. multiplier_q15 不能跟着 DATA_W 收窄，单独引入 MULT_W=16。
3. w_fold_wdata 应为 MULT_W 宽度。
4. bias_fold_wdata 保持 32 bit。
5. packed ROM/mock ROM/testbench 中旧 64/128/192/256 bit 宽度按 8-bit lane 重算。
```
