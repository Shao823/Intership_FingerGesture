# 计算单元接口与存储设计指南 2026-07-06

本文面向后续 top scheduler、级间寄存器、activation/window loader、weight ROM controller 的设计。

当前已检查的计算单元：

```text
stem_conv_array_4x4
dw_conv1_4ch_5tap
pw_conv1_array_4x8
dw_conv2_4ch_5tap
pw_conv2_array_4x12
fc_classifier_96_5
```

回归结果：

```text
PASS stem_conv_array_4x4 self-check
PASS dw_conv1_4ch_5tap self-check
PASS pw_conv1_array_4x8 self-check
PASS dw_conv2_4ch_5tap self-check
PASS pw_conv2_array_4x12 GAP self-check
PASS fc_classifier_96_5 self-check
```

审查结论：

```text
1. 当前计算单元的计算逻辑、输出粒度、握手语义与 testbench 一致。
2. PW1/PW2 都是 full-output-channel job，不是只算一个 oc group。
3. PW2 active 输出是 GAP 累加结果，不再输出 feature tile stream。
4. FC 内部执行 /174，不要在 PW2 后再额外做一次 average。
5. 后续集成的关键风险不在 MAC datapath，而在 buffer valid bit、窗口拼接、tile 锁存、ROM lane 打包和 response 顺序。
```

## 1. 全局约定

### 1.1 数值格式

当前 RTL 默认：

```text
activation: signed Q8.8, 16-bit
weight:     signed Q8.8, 16-bit
BN scale:   signed Q8.8, 16-bit
BN bias:    signed Q8.8, 16-bit
MAC acc:    signed 48-bit
PW2 GAP sum: signed 48-bit per channel
```

乘法累加规则：

```text
Q8.8 * Q8.8 -> Q16.16
acc 保存 Q16.16
输出前 acc >>> 8 回到 Q8.8，再做饱和
```

FC bias 作为矩阵扩展列写入，但硬件执行时不是把输入补 1 再乘，而是：

```text
acc = bias << FRAC_BITS
acc += activation * weight
```

这和 Q8.8 bias 对齐到 Q16.16 累加域等价。

### 1.2 vector lane bit 顺序

所有打包 vector 统一：

```text
vec[lane*DATA_W +: DATA_W] = logical_value[base + lane]
```

即 lane 0 在低位，不要被注释里 `{W[base+N], ..., W[base+0]}` 的书写习惯误导。

### 1.3 request/response 顺序

stem/PW 的 weight 和 activation 访问都是 request/response：

```text
request accepted: req_valid && req_ready
response accepted: vec_valid && vec_ready
```

controller/loader 必须保证：

```text
1. 每个 accepted request 恰好返回一次 response。
2. 同一个计算核内部 response 顺序必须等于 request 顺序。
3. 不允许乱序、不允许丢 response、不允许额外 response。
4. 若物理 ROM/BRAM 需要多拍拼 vector，可以延迟 response，但不能改变顺序。
```

DW window loader 当前每个核最多一个 outstanding request。它可以多拍准备窗口，但必须在窗口完整且非 padding 数据 valid 后才返回。

### 1.4 start/done/busy

通用规则：

```text
start 只在 IDLE 被接受。
busy=1 时额外 start 会被忽略。
done 通常表示本次 job 的结果已经被下游接受，或内部累加 job 已完成。
```

FC 例外更明确：

```text
result_valid=1 表示 logits/class_idx 有效且保持稳定。
done=1 表示 result_valid 已经被 result_ready 接收。
```

### 1.5 参数写入窗口

```text
stem BN 写入:       仅 IDLE 且 !start 时生效。
DW1/DW2 weight 写入: 仅 IDLE 且 !start 时生效。
PW1/PW2 BN 写入:    仅 IDLE 且 !start 时生效。
FC0/FC1 weight 写入:仅 IDLE 且 !start 时生效。
```

busy 期间误写不会改变当前 job 的参数。

## 2. 计算单元接口合同

### 2.1 stem_conv_array_4x4

计算：

```text
Conv1d 5 -> 32, kernel=7, padding=3
folded BN + ReLU
```

一次 job：

```text
4 high-res time rows x 4 output channels
```

外层扫描：

```text
tile_t_base = 0,4,8,...,344
oc_base     = 0,4,8,...,28
```

activation 请求：

```text
act_req_t_base = tile_t_base
act_req_k      = 0..34

k_flat = act_req_k
input_channel = k_flat / 7
tap           = k_flat % 7

in_t = tile_t_base + row + tap - 3
act_vec[row] = raw_input[in_t][input_channel]
越界 in_t 返回 0 padding
```

weight 请求：

```text
weight_req_oc_base = oc_base
weight_req_k       = 0..34
weight_vec[lane]   = W_stem[oc_base + lane][k_flat]
```

输出：

```text
out_valid / out_ready
out_tile_t_base = tile_t_base
out_tile_oc_base = oc_base
out_tile[4 rows x 4 channels]
done 在 out_ready 接收 out_tile 后脉冲
```

写入级间寄存器：

```text
stem_to_dw1_buf[348][32]
stem_to_dw1_valid_group[348][8]  // 每 bit 对应 4 channels

group = out_tile_oc_base / 4
for row=0..3:
  t = out_tile_t_base + row
  stem_to_dw1_buf[t][out_tile_oc_base + lane] <= out_tile[row][lane]
  stem_to_dw1_valid_group[t][group] <= 1
```

### 2.2 dw_conv1_4ch_5tap

计算：

```text
Depthwise Conv1d, 32 channels, kernel=5, padding=2
无 BN/ReLU
```

一次 job：

```text
一个 4-channel group 的所有新可计算 out_t
```

输入控制：

```text
ch_base      = 0,4,8,...,28
input_t_base = 当前新到的 stem tile base
input_rows   = 4
final_chunk  = 1 only when input_t_base == 344
```

边界输出数量：

```text
base=0,   rows=4, final=0 -> out_t=0..1,     2 点
middle,   rows=4, final=0 -> out_t=base-2..base+1, 4 点
base=344, rows=4, final=1 -> out_t=342..347, 6 点
```

window 请求：

```text
win_req_out_t
win_req_ch_base

win_vec[tap][lane] =
  stem_to_dw1_buf[win_req_out_t + tap - 2][win_req_ch_base + lane]

越界 time 返回 0 padding。
非 padding time 必须等对应 valid_group 为 1 后才可返回。
```

weight：

```text
weight_mem[32][5] 本地驻留
weight_we, weight_ch, weight_tap, weight_wdata 初始化
```

输出：

```text
out_valid / out_ready
out_t
out_ch_base
out_vec[4 channels]
```

写入级间寄存器：

```text
dw1_to_pw1_buf[348][32]
dw1_to_pw1_valid_group[348][8]

group = out_ch_base / 4
dw1_to_pw1_buf[out_t][out_ch_base + lane] <= out_vec[lane]
dw1_to_pw1_valid_group[out_t][group] <= 1
row_full[t] = &dw1_to_pw1_valid_group[t][7:0]
```

### 2.3 pw_conv1_array_4x8

计算：

```text
Pointwise Conv1d 32 -> 64
folded BN + ReLU + MaxPool1d(kernel=2,stride=2)
```

一次 job：

```text
输入:  4 high-res rows x 32 channels
输出:  2 low-res pooled rows x 64 channels
```

重要点：

```text
oc_base input 只为兼容保留，当前 full-channel job 会忽略它。
模块内部固定扫描 oc_base=0,8,16,24,32,40,48,56。
```

input tile ready：

```text
input_tile_req_t_base = tile_t_base
input_tile_req_ready=1 表示:
  dw1_to_pw1 row_full[tile_t_base + 0..3] 全部为 1
  且这 4x32 tile 在整个 PW1 job 期间稳定
```

推荐 row packer：

```text
pw1_input_tile_regs[4][32]

当 input_tile_req_valid && input_tile_req_ready:
  锁存 dw1_to_pw1_buf[tile_t_base + row][k] 到 tile_regs[row][k]

act_req_k = k 时:
  act_vec[row] = tile_regs[row][k]
```

weight 请求：

```text
for oc_base in 0,8,...,56:
  for k in 0..31:
    weight_req_oc_base = oc_base
    weight_req_k       = k
    weight_vec[lane]   = W_pw1[oc_base + lane][k]
```

activation 请求：

```text
act_req_t_base = tile_t_base
act_req_k      = k
act_vec[row]   = tile_regs[row][k]
```

输出：

```text
out_valid / out_ready
out_pool_t_base = tile_t_base / 2
out_oc_base     = 0,8,...,56
out_tile[2 rows x 8 channels]
out_last_group  = 1 only when out_oc_base == 56
done 在最后一组 out_tile 被接收后脉冲
```

写入级间寄存器：

```text
pool_to_dw2_buf[174][64]
pool_to_dw2_valid_group[174][16]  // 每 bit 对应 4 channels

for pool_row=0..1:
  t = out_pool_t_base + pool_row
  pool_to_dw2_buf[t][out_oc_base + col] <= out_tile[pool_row][col]

因为 PW1 一次输出 8 channels:
  pool_to_dw2_valid_group[t][out_oc_base/4]     <= 1
  pool_to_dw2_valid_group[t][out_oc_base/4 + 1] <= 1
```

### 2.4 dw_conv2_4ch_5tap

计算：

```text
Depthwise Conv1d, 64 channels, kernel=5, padding=2
无 BN/ReLU
```

一次 job：

```text
一个 PW1 8-channel tile 内的两个 4-channel group：
  group 0: ch_base
  group 1: ch_base + 4
输出顺序为 group 0 的所有新可计算 low-res out_t，
再输出 group 1 的所有新可计算 low-res out_t。
```

输入控制：

```text
ch_base      = 0,8,16,...,56
input_t_base = 当前新到的 pooled low-res base
input_rows   = 2
final_chunk  = 1 only when input_t_base == 172
```

边界输出数量：

```text
base=0,   rows=2, final=0 -> 0 点
base=2,   rows=2, final=0 -> out_t=0..1, 2 点
middle,   rows=2, final=0 -> 2 点
base=172, rows=2, final=1 -> out_t=170..173, 4 点
```

window 请求：

```text
win_req_out_t
win_req_ch_base

win_vec[tap][lane] =
  pool_to_dw2_buf[win_req_out_t + tap - 2][win_req_ch_base + lane]

越界 time 返回 0 padding。
非 padding time 必须等对应 pool_to_dw2_valid_group 为 1 后才可返回。
```

输出写入：

```text
dw2_to_pw2_buf[174][64]
dw2_to_pw2_valid_group[174][16]

group = out_ch_base / 4
dw2_to_pw2_buf[out_t][out_ch_base + lane] <= out_vec[lane]
dw2_to_pw2_valid_group[out_t][group] <= 1
row_full[t] = &dw2_to_pw2_valid_group[t][15:0]
```

### 2.5 pw_conv2_array_4x12

计算：

```text
Pointwise Conv1d 64 -> 96
folded BN + ReLU
内部直接做 GAP sum 累加
```

一次 job：

```text
输入: 4 low-res rows x 64 channels
内部扫描 oc_base=0,12,24,36,48,60,72,84
输出: 不输出 feature tile，直接累加 gap_sum_mem[96]
```

input tile ready：

```text
input_tile_req_t_base
input_tile_req_row_valid_mask

input_tile_req_ready=1 表示:
  对 row_valid_mask 为 1 的 row，dw2_to_pw2 row_full[t_base+row] 全部为 1
  且这 4x64 tile 在整个 PW2 job 期间稳定
```

推荐 row packer：

```text
pw2_input_tile_regs[4][64]

普通 tile:
  tile_t_base = 0,4,8,...,168
  row_valid_mask = 4'b1111

final partial tile:
  tile_t_base = 172
  row_valid_mask = 4'b0011
```

activation 请求：

```text
act_req_t_base = tile_t_base
act_req_k      = k
act_req_row_valid_mask = row_valid_mask

if row_valid_mask[row]:
  act_vec[row] = tile_regs[row][k]
else:
  act_vec[row] = 0
```

weight 请求：

```text
for oc_base in 0,12,...,84:
  for k in 0..63:
    weight_req_oc_base = oc_base
    weight_req_k       = k
    weight_vec[lane]   = W_pw2[oc_base + lane][k]
```

GAP 行为：

```text
gap_clear 清零 gap_sum_mem[96] 和 gap_count。

每个 oc group 完成后:
  for col=0..11:
    for row=0..3:
      if row_valid_mask[row]:
        gap_sum_mem[oc_base + col] += post_relu[row][col]

只有最后 oc group 才:
  gap_count += popcount(row_valid_mask)
```

frame 完成：

```text
完整 174 行后 gap_count == 174
最后 partial tile 使 gap_frame_done 脉冲 1 拍
```

PW2 到 FC：

```text
gap_sum_vec 是内部 gap_sum_mem 的 wire 视图。
如果 top 在 FC 完成前不会 gap_clear，可以直接给 FC。
更稳妥做法是在 gap_frame_done 时锁存:
  gap_to_fc_sum_regs[96] <= gap_sum_vec
  gap_to_fc_count_reg    <= gap_count
然后 FC 使用这组寄存器。
```

### 2.6 fc_classifier_96_5

计算：

```text
GAP average: gap_avg[ch] = gap_sum_vec[ch] / 174
FC0: 96 -> 96
ReLU
FC1: 96 -> 5
argmax
```

一次 job：

```text
start 时锁存 96 个 gap_sum_vec。
gap_count 应为 174，否则 input_count_error=1。
计算仍固定除以 174。
```

资源：

```text
FC0_OUT_LANES = 16
FC0 分 6 个输出组，每组 16 个输出并行 MAC。
FC1 5 个 class 并行 MAC。
```

权重：

```text
fc0_weight_mem[96][97]
  k=0..95: FC0 data weight
  k=96:    FC0 bias

fc1_weight_mem[5][97]
  k=0..95: FC1 data weight
  k=96:    FC1 bias
```

输出：

```text
result_valid / result_ready
logits_vec[5]
class_idx
done
input_count_error
```

argmax 平局：

```text
只在 logit[class] > current_best 时更新。
最大值相等时选择本次 logits 中 class index 最小者。
不会保留上一帧 class_idx。
```

## 3. 级间寄存器建议

### 3.1 总览

在“不跨帧重叠，FC 完成后再开始下一帧”的策略下，单缓冲加 valid 清零足够。

如果未来要在 FC 运行期间启动下一帧，至少需要对被覆盖的 buffer 做 ping-pong 或 frame tag。

推荐级间存储：

```text
raw_input_buf[348][5]             // stem input loader 使用
stem_to_dw1_buf[348][32]
dw1_to_pw1_buf[348][32]
pool_to_dw2_buf[174][64]
dw2_to_pw2_buf[174][64]
gap_to_fc_sum_regs[96]            // 可选但推荐
gap_to_fc_count_reg
```

valid bit：

```text
stem_to_dw1_valid_group[348][8]   // 32 channels / 4
dw1_to_pw1_valid_group[348][8]
pool_to_dw2_valid_group[174][16]  // 64 channels / 4
dw2_to_pw2_valid_group[174][16]
```

建议容量：

```text
stem_to_dw1: 348*32*16 = 178176 bits
dw1_to_pw1: 348*32*16 = 178176 bits
pool_to_dw2: 174*64*16 = 178176 bits
dw2_to_pw2: 174*64*16 = 178176 bits
gap_to_fc: 96*48 = 4608 bits
```

这些量级都不大。后续更重要的是读端口组织和 tile/window loader，而不是容量。

### 3.2 stem_to_dw1

writer：

```text
stem out_valid/out_ready accepted
写 4 rows x 4 channels
设置对应 valid_group bit
```

reader：

```text
DW1 window loader
按 win_req_out_t/ch_base 读取 5 taps x 4 channels
padding 位置补 0
非 padding 位置必须 valid_group[t][ch_base/4]==1
```

### 3.3 dw1_to_pw1

writer：

```text
DW1 out_valid/out_ready accepted
写 1 row x 4 channels
设置 valid_group[t][ch_base/4]
```

reader：

```text
PW1 row packer
当 row_full[t_base+0..3] 全部为 1:
  锁存 4x32 到 pw1_input_tile_regs
  input_tile_req_ready 拉高
PW1 job 期间 tile_regs 保持稳定
```

### 3.4 pool_to_dw2

writer：

```text
PW1 out_valid/out_ready accepted
写 2 rows x 8 channels
设置两个 4-channel valid_group bit
```

reader：

```text
DW2 window loader
按 win_req_out_t/ch_base 读取 5 taps x 4 channels
padding 位置补 0
非 padding 位置必须 valid_group[t][ch_base/4]==1
```

### 3.5 dw2_to_pw2

writer：

```text
DW2 out_valid/out_ready accepted
写 1 row x 4 channels
设置 valid_group[t][ch_base/4]
```

reader：

```text
PW2 row packer
普通 tile t_base=0,4,...,168:
  row_valid_mask=4'b1111
  需要 row_full[t_base+0..3] 全部为 1

final tile t_base=172:
  row_valid_mask=4'b0011
  只要求 row_full[172] 和 row_full[173]
```

### 3.6 gap_to_fc

最简单方案：

```text
PW2 gap_frame_done 后，直接 start FC。
FC 运行完之后再 gap_clear，开始下一帧。
```

推荐方案：

```text
在 gap_frame_done 那一拍或下一拍锁存:
  gap_to_fc_sum_regs <= gap_sum_vec
  gap_to_fc_count_reg <= gap_count

FC 使用 gap_to_fc_*。
这样 PW2 的 gap_clear 不会影响 FC 输入。
```

## 4. Weight ROM 和参数存储建议

### 4.1 原则

每个模块的权重分开存。不要强行做一个统一大 ROM 给所有层直接访问。

推荐：

```text
stem_weight_rom
dw1_weight_mem/init_rom
pw1_weight_rom
dw2_weight_mem/init_rom
pw2_weight_rom
fc0_weight_rom_or_regs
fc1_weight_rom_or_regs

stem_bn_scale/bias
pw1_bn_scale/bias
pw2_bn_scale/bias
```

每层使用自己的 packed layout。ROM controller 把物理 word 转成计算核期望的 vector。

### 4.2 stem weight ROM

逻辑：

```text
W_stem[32][35]
35 = 5 input channels * 7 taps
```

packed word：

```text
stem_weight_word[8][35]
word width = 4 * DATA_W = 64 bits

word[lane] = W_stem[oc_group*4 + lane][k_flat]
oc_group = weight_req_oc_base / 4
address  = oc_group * 35 + weight_req_k
```

容量：

```text
8 * 35 * 64 = 17920 bits
```

### 4.3 DW1 weight

逻辑：

```text
W_dw1[32][5]
```

当前 RTL 本地驻留：

```text
weight_ch = 0..31
weight_tap = 0..4
```

容量：

```text
32 * 5 * 16 = 2560 bits
```

可以用初始化 FSM 从 ROM 写入，也可以综合为寄存器/小 RAM。

### 4.4 PW1 weight ROM

逻辑：

```text
W_pw1[64][32]
```

packed word：

```text
pw1_weight_word[8][32]
word width = 8 * DATA_W = 128 bits

word[lane] = W_pw1[oc_group*8 + lane][k]
oc_group = weight_req_oc_base / 8
address  = oc_group * 32 + weight_req_k
```

容量：

```text
8 * 32 * 128 = 32768 bits
```

### 4.5 DW2 weight

逻辑：

```text
W_dw2[64][5]
```

当前 RTL 本地驻留：

```text
weight_ch = 0..63
weight_tap = 0..4
```

容量：

```text
64 * 5 * 16 = 5120 bits
```

### 4.6 PW2 weight ROM

逻辑：

```text
W_pw2[96][64]
```

packed word：

```text
pw2_weight_word[8][64]
word width = 12 * DATA_W = 192 bits

word[lane] = W_pw2[oc_group*12 + lane][k]
oc_group = weight_req_oc_base / 12
address  = oc_group * 64 + weight_req_k
```

容量：

```text
8 * 64 * 192 = 98304 bits
```

如果 BRAM 原生宽度不够，ROM controller 可以多拍读取并拼成 192-bit vector 后再返回 `weight_vec_valid`。计算核支持等待，但 response 必须保持顺序。

### 4.7 FC0 weight

逻辑：

```text
W_fc0_aug[96][97]
k=0..95 data weight
k=96 bias
```

当前 RTL 是本地二维数组，计算时每拍需要读取 16 个输出 neuron 对同一个 k 的权重：

```text
for lane=0..15:
  W_fc0[out_base + lane][k]
```

推荐物理组织为 16-lane packed/banked：

```text
fc0_weight_word[6][97]
word width = 16 * DATA_W = 256 bits

word[lane] = W_fc0_aug[out_group*16 + lane][k]
out_group = out_base / 16
address = out_group * 97 + k
```

容量：

```text
96 * 97 * 16 = 148992 bits
```

如果保持当前 RTL 的本地数组，综合时应让 16 个 lane 能并行读。若映射到 BRAM，建议按 lane bank 拆成 16 个 bank 或在初始化时展开成 packed regs。

### 4.8 FC1 weight

逻辑：

```text
W_fc1_aug[5][97]
k=0..95 data weight
k=96 bias
```

当前 RTL 每拍并行读取 5 个 class 对同一个 k 的权重：

```text
for class=0..4:
  W_fc1[class][k]
```

推荐物理组织：

```text
fc1_weight_word[97]
word width = 5 * DATA_W = 80 bits

word[class] = W_fc1_aug[class][k]
address = k
```

容量：

```text
5 * 97 * 16 = 7760 bits
```

### 4.9 BN 参数

folded BN 仅用于 stem、PW1、PW2。

```text
stem:
  w_fold_mem[32]
  bias_fold_mem[32]

PW1:
  w_fold_mem[64]
  bias_fold_mem[64]

PW2:
  w_fold_mem[96]
  bias_fold_mem[96]
```

```text

stem_bn_bias.mem
pw1_bn_scale.mem
pw1_bn_bias.mem
pw2_bn_scale.mem
pw2_bn_bias.mem
```

## 5. 推荐 top 调度顺序

### 5.1 frame 初始化

每帧开始前：

```text
1. 清空所有 valid_group。
2. gap_clear PW2。
3. 加载/确认本帧 raw_input_buf[348][5] 已准备好。
4. 确保权重和 BN 参数已经写完。
```

### 5.2 前向流水触发

推荐调度思路：

```text
stem:
  for tile_t_base = 0,4,...,344:
    for oc_base = 0,4,...,28:
      start stem
      写 stem_to_dw1
      对同一 ch_base 尝试触发 DW1

DW1:
  input_t_base = stem tile_t_base
  input_rows = 4
  final_chunk = (input_t_base == 344)
  ch_base = stem out_tile_oc_base

PW1:
  当 dw1_to_pw1 row_full[t_base+0..3] 全部为 1:
    start PW1 tile_t_base=t_base
  t_base = 0,4,...,344

DW2:
  每个 PW1 输出 out_oc_base 覆盖 8 channels。
  启动一个 DW2 job: ch_base=out_oc_base。
  DW2 内部复用 4-channel datapath 依次计算 out_oc_base 和 out_oc_base+4。
  input_t_base = out_pool_t_base
  input_rows = 2
  final_chunk = (input_t_base == 172)

PW2:
  普通 tile t_base=0,4,...,168, row_valid_mask=1111
  final tile t_base=172, row_valid_mask=0011
  等 dw2_to_pw2 对应 row_full 满足后 start

FC:
  等 PW2 gap_frame_done。
  锁存 gap_sum/count。
  start FC。
  result_valid/result_ready 完成后才开始下一帧。
```

### 5.3 不要做的事

```text
1. 不要在 PW1/PW2 input_tile_req_ready 后释放或覆盖 tile 数据。
   这两个模块会在同一个 input tile 上重复扫描所有 output-channel groups。

2. 不要给 PW2 接旧的 out_tile 接口。
   当前 active 接口是 gap_sum_vec/gap_count/gap_frame_done。

3. 不要在 FC 前再次除以 174。
   FC 内部已经从 gap_sum_vec 计算 average。

4. 不要让 response 乱序返回。
   当前计算核没有 response id。

5. 不要在 busy 期间写权重/BN。
   当前模块会忽略 busy 写，但 top 不应依赖这种误操作作为正常控制流。
```

## 6. 后续模块开发优先级

建议按这个顺序补齐集成模块和 TB：

```text
1. raw_input_buf + stem activation im2col loader
2. stem_weight_rom_controller
3. stem_to_dw1 buffer + DW1 window loader
4. dw1_to_pw1 buffer + PW1 row packer/tile_regs
5. pw1_weight_rom_controller
6. pool_to_dw2 buffer + DW2 window loader
7. dw2_to_pw2 buffer + PW2 row packer/tile_regs
8. pw2_weight_rom_controller
9. gap_to_fc latch
10. FC weight initialization/ROM loader
11. top scheduler
```

每个 loader/controller 的独立 TB 至少覆盖：

```text
1. 地址映射。
2. lane 打包顺序。
3. request/response backpressure。
4. response 顺序。
5. padding。
6. final partial tile / row_valid_mask。
7. valid_group 未满足时不得提前 ready/valid。
8. tile 锁存后，计算 job 期间数据保持稳定。
```

## 7. 当前保留风险

这些不是当前 RTL 仿真失败点，但集成时必须处理：

```text
1. 当前 testbench 的 ROM/activation provider 是逻辑模型，还没有验证真实 BRAM 地址和 bank 冲突。
2. FC 当前用 `/174`，综合若资源不理想，需要改成 reciprocal multiply/shift 并重新做 bit-exact reference。
3. 如果未来跨帧重叠，需要给大 buffer 做 ping-pong 或 frame_id，当前指南默认一帧 FC 完成后再开始下一帧。
4. PW2 gap_sum_vec 在 gap_clear 后会清零。若 FC 与下一帧重叠，必须加 gap_to_fc latch。
```
