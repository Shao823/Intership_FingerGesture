## 2. PS 端软件架构重构

### 2.1 从单帧轮询改为 10 帧中断流水

此前 PS 程序只执行：

```text
单帧数据准备
    ↓
DMA SimpleTransfer
    ↓
轮询 DMA 完成
    ↓
轮询 accelerator FRAME_READY
    ↓
手动 START
    ↓
轮询 accelerator DONE
    ↓
读取 RESULT
```

今天改为：

```text
初始化 DMA 和 GIC
    ↓
配置 accelerator AUTO_START + IRQ_ENABLE
    ↓
提交 frame0 DMA
    ↓
DMA IOC 中断
    ↓
提交下一帧 DMA
    ↓
accelerator 自动启动完整帧
    ↓
accelerator DONE 中断
    ↓
读取 RESULT / ERROR_STATUS / CYCLE_COUNT
    ↓
CLEAR_DONE
    ↓
继续处理后续帧
```

## 3. Accelerator AUTO_START 和 CONFIG 配置

当前测试中写入：

```text
CONFIG = 0x00000003
```

32 位含义：

```text
bit 0 = 1 → AUTO_START
bit 1 = 1 → IRQ_ENABLE
bit 31:2 = 0
```

即：

```text
0x00000003 = AUTO_START | IRQ_ENABLE
```

初始化阶段先写：

```text
COMMAND = 0x00000006
```

该值包含：

```text
bit 1 = CLEAR_DONE
bit 2 = CLEAR_ERRORS
```

随后依靠 AUTO_START，不再对每一帧手动写 COMMAND.START。

### 3.1 Accelerator DONE IRQ 处理

accelerator 中断处理顺序为：

```text
读取 STATUS
读取 RESULT
读取 RX_STATUS
读取 ERROR_STATUS
读取 CYCLE_COUNT
保存当前 observation
写 COMMAND = 0x00000002
清除 sticky DONE 和 accelerator IRQ
```

当前 accelerator IRQ 是 level-high sticky 中断：

```text
irq = irq_enable && done_sticky
```

如果不写 CLEAR_DONE，IRQ 会保持有效。

## 6. DMA 中断时刻的 accelerator 状态分析

DMA 输出中的：

```text
DMA[n] accel=...
```

表示：

> 第 n 次 DMA IOC 中断发生时，PS 顺便读取到的 accelerator STATUS 快照。

它不是 DMA 自己的状态。

### 6.1 第 0 次 DMA 快照

```text
DMA[0] ... accel=0x00000011
```

```text
BUSY = 1
ACT_WRITE_READY = 1
FRAME_READY = 0
RX_ACTIVE = 0
```

含义：

```text
frame0 已经开始计算；
此时下一帧还没有形成完整等待帧。
```

这是启动边界状态。

### 6.2 后续 DMA 快照

```text
DMA[1]～DMA[9] ... accel=0x0000001D
```

```text
BUSY = 1
FRAME_READY = 1
RX_ACTIVE = 1
ACT_WRITE_READY = 1
```

含义：

```text
当前 accelerator 正在计算上一帧；
下一帧已经完整接收或正在接收；
下一帧位于另一个 activation bank；
系统已经进入稳定 Ping-Pong 流水。
```

### 6.3 为什么 `FRAME_READY` 和 `ACT_WRITE_READY` 可以同时为 1

`FRAME_READY` 表示：

```text
已经有一整帧数据写完，等待 accelerator 消费。
```

`ACT_WRITE_READY` 实际表示：

```text
write_bank 与当前 read_bank 没有冲突。
```

它不代表当前接收 bank 还有空闲空间。

真正决定 AXI-Stream 是否可以继续接收的是：

```text
S_AXIS_TREADY = rst_n && (!frame_ready && act_wr_ready)
```

因此可能出现：

```text
FRAME_READY = 1
ACT_WRITE_READY = 1
S_AXIS_TREADY = 0
```

这表示等待 bank 没有读写冲突，但已经写满，因此暂时禁止覆盖，等待 AUTO_START 完成 bank 切换。

## 7. ILA 和断点调试结论

### 7.1 ILA 一次 Arm 的含义

ILA 每次 Arm 通常只能保存一次触发窗口。

使用：

```text
class_valid == 1
```

可以捕获一次分类结果事件。

如果要逐帧查看 10 次，需要每次重新 Arm；如果只验证整体 10 帧流水，建议 Arm 一次并让程序连续运行。

### 7.2 为什么在 ISR 断点逐帧调试会超时

曾经在 accelerator ISR 设置断点并逐帧点击 Resume，出现：

```text
DMA completed = 10
Accelerator completed = 4
Error flags = 0x00000014
ERROR: 10-frame pipeline timeout
```

原因不是硬件计算错误，而是：

```text
A53 在 ISR 断点处暂停；
PL 侧 DMA 和 accelerator 仍然继续运行；
DMA 可以继续完成；
accelerator DONE sticky IRQ 没有及时清除；
RESULT 可能被后续帧覆盖；
软件 observation index 和实际结果产生错位。
```

因此：

```text
连续 10 帧吞吐测试：不要在 ISR 长时间停住。
逐帧波形分析：应使用单帧调试模式。
整体 PASS 验证：只在 ValidatePipelineResults() 处设置断点。
```

### 7.3 当前推荐调试方法

连续 10 帧测试：

```text
1. 禁用 DMA/accelerator ISR 断点。
2. ILA 设置 class_valid == 1。
3. Arm ILA。
4. Vitis 点击 Resume。
5. 等待 UART 输出最终 PASS。
6. 检查 g_pipeline 统计量。
```

逐帧细节调试：

```text
1. 改用单帧 debug 模式。
2. 每次只发送一帧。
3. Arm ILA。
4. 进入 accelerator ISR。
5. 读取 RESULT、STATUS、ERROR_STATUS、CYCLE_COUNT。
6. 清 DONE 后再发送下一帧。
```

## 9. 16 通道设备复用性评估

### 9.1 可以复用的部分

以下系统级模块可以复用：

```text
PS DDR 到 DMA 的数据通路
AXI DMA Simple MM2S
DMA IOC/ERROR 中断
accelerator AXI-Lite 寄存器协议
AUTO_START/IRQ_ENABLE 机制
accelerator DONE IRQ
两个 activation BRAM bank 的 Ping-Pong 框架
AXI-Stream 握手和背压
PS GIC 中断框架
UART、ILA 和测试统计框架
```

### 9.2 必须修改的部分

当前 RTL 仍固定使用：

```text
STEM_IC = 5
STEM_K = 35
INPUT_LEN = 348
ACT_RAM_DEPTH = 435
FRAME_WORDS = 435
FRAME_BYTES = 1740
```

如果 16 通道仍使用 348 个时间采样点：

```text
STEM_IC = 16
STEM_K = 16 × 7 = 112
FRAME_BYTES = 16 × 348 = 5568
FRAME_WORDS = 5568 / 4 = 1392
FRAME_INFO = 0x15C00570
ACT_RAM_DEPTH = 16 × 87 = 1392
```

需要重新适配：

```text
1. 第一层 stem 输入通道和权重。
2. STEM_K 和 stem 乘加调度。
3. activation BRAM 两个 bank 的深度。
4. act_wr_addr 地址宽度。
5. AXI-Stream FRAME_WORDS/TLAST 检查。
6. packaged IP 参数和 component.xml。
7. 16 通道模型、checkpoint、量化数据和 BRAM 初始化。
8. PS 端 frame bytes、frame words 和输入数组。
```

### 9.3 16 通道迁移建议

```text
1. 先在 Python golden model 中验证 [16,348] 输入。
2. 重新训练/导出/量化 16 通道模型。
3. 重新生成 stem 权重和量化参数。
4. 修改 RTL 参数和 BRAM IP 深度。
5. 单帧 RTL 仿真，确认 1392 beats 和 TLAST。
6. 单帧 DMA 上板。
7. 单帧 accelerator IRQ 上板。
8. 双帧 Ping-Pong 上板。
9. 最后恢复 10 帧 AUTO_START 测试。
```

### 9.4 复用结论

```text
系统接口架构：可以复用
DMA/中断/Ping-Pong 框架：可以复用
AXI-Lite 寄存器协议：建议保持不变
当前 accelerator 计算核心：不能直接原样复用
第一层 stem、权重、BRAM 深度：必须修改
训练模型和量化数据：必须重新生成
```

## 10. 今日问题与解决结果

### 问题 1：10 帧测试初次显示 FAIL

初次输出：

```text
DMA completed = 10
Accelerator completed = 10
Autostart overlap count = 8
Error flags = 0x00000020
RESULT = FAIL
```

原因：软件最初要求 overlap 次数必须大于等于 9，判定条件过严。实际已经观察到真实 queued-frame 和 DMA/计算重叠，硬件功能正常。

后续将软件判定调整为：

```text
autostart_overlap_count > 0
queued_frame_count > 0
```

之后重新测试，得到 9 次 overlap，最终 PASS。

### 问题 2：ISR 逐帧断点调试出现 pipeline timeout

原因：A53 停止不会暂停 PL；DMA 和 accelerator 继续运行，结果寄存器可能被后续帧覆盖，软件中断处理和帧序号发生错位。

结论：

```text
连续吞吐测试不要在 ISR 长时间断点。
逐帧 ILA 观察应采用单帧 debug 模式。
```

### 问题 3：PMU-FW 提示

UART 出现：


当前判断：

```text
该提示没有影响本次裸机测试。
```
